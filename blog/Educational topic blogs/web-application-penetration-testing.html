Web Application Penetration Testing — Overview

Web Application Penetration Testing (WAPT) is a structured security assessment that simulates real-world attacks against web applications to identify exploitable vulnerabilities before malicious actors do. It is the most commonly conducted penetration test type — web applications are the primary internet-facing attack surface for virtually every organisation, handling authentication, sensitive data, business logic, and increasingly direct integration with internal systems and cloud infrastructure.

WAPT follows the OWASP Web Security Testing Guide (WSTG), the Penetration Testing Execution Standard (PTES), and the OWASP Testing Framework — ensuring systematic coverage of all vulnerability categories rather than opportunistic scanning that misses business logic flaws, access control issues, and second-order vulnerabilities that automated tools consistently overlook.

WAPT vs Automated Vulnerability Scanning

Phase I — Reconnaissance & Information Gathering

Reconnaissance maps the target web application's complete attack surface before any active probing begins. Passive reconnaissance extracts intelligence from public sources — certificate transparency logs, search engine caches, DNS records, and OSINT — without generating any traffic toward the target. Active reconnaissance directly interacts with the target to enumerate content, fingerprint technologies, and discover hidden endpoints. Both phases are essential: passive recon discovers what the target thinks is private, active recon discovers what they forgot was public.

# PASSIVE — Extract intelligence without touching target amass enum -passive -d target.com # Subdomain enumeration curl -s "https://crt.sh/?q=%.target.com&output=json" | \ python3 -c "import json,sys;[print(c['name_value']) for c in json.load(sys.stdin)]" | sort -u theHarvester -d target.com -b all -l 300 # Emails, hosts, employees shodan domain target.com # Internet-exposed services # ACTIVE — Enumerate content and technology stack whatweb -a 3 -v https://target.com # Tech fingerprint (framework, CMS, server) gobuster dir -u https://target.com \ -w /usr/share/seclists/Discovery/Web-Content/raft-large-directories.txt \ -x php,asp,aspx,jsp,html,js -t 50 --status-codes 200,301,302,403 ffuf -u https://target.com/api/FUZZ \ -w /usr/share/seclists/Discovery/Web-Content/api-endpoints.txt -mc 200,201,401,403 # JS file analysis — discover hidden API endpoints and keys linkfinder -i https://target.com -d -o endpoints.html secretfinder -i https://target.com/static/app.js # Find hardcoded secrets in JS # Wayback Machine — discover retired but still-live endpoints waybackurls target.com | grep -E "\.(php|asp|aspx|jsp|json|xml)$" | sort -u

Phase II — Threat Modelling & Attack Surface Mapping

Threat modelling converts the raw intelligence gathered during reconnaissance into a prioritised testing plan — identifying which components of the application are most likely to harbour vulnerabilities and which vulnerabilities, if exploited, would produce the greatest business impact. STRIDE provides the analytical framework; the output is a ranked list of attack vectors to investigate, ordered by impact potential and probability. This step ensures testing effort concentrates on high-value targets rather than being diffused uniformly across the entire application surface.

Web Application Threat Model — Key Attack Surfaces

Phase III — Scanning & Vulnerability Enumeration

The scanning phase actively probes the application for vulnerabilities across all discovered endpoints. Automated scanning provides coverage breadth — quickly identifying known vulnerability patterns across the entire application surface. Manual testing provides depth — finding business logic flaws, complex injection scenarios, and multi-step vulnerability chains that automated tools cannot reason about. Both are required: automated scanning alone produces high false positive rates and misses critical application-specific vulnerabilities; manual testing alone misses coverage of the full endpoint inventory.

# Automated scanning — broad coverage first nikto -h https://target.com -ssl -C all -nointeractive nuclei -u https://target.com -t cves/ -t exposures/ -t misconfiguration/ -severity critical,high wpscan --url https://target.com --enumerate vp,va,vt,u --api-token YOUR_TOKEN # WordPress targets only # Burp Suite active scan — comprehensive web vulnerability scanner # 1. Spider the target to discover all endpoints # 2. Run active scan across entire site tree # 3. Manually verify all scanner-flagged findings before reporting # Manual — technology-specific checks nmap --script http-methods,http-headers,ssl-cert,http-title -p 80,443 target.com # Check HTTP security headers (missing headers are findings) curl -I -s https://target.com | grep -iE \ "Strict-Transport-Security|Content-Security-Policy|X-Frame-Options|X-Content-Type-Options|Referrer-Policy|Permissions-Policy" # JavaScript analysis — discover hidden endpoints and sensitive data python3 linkfinder.py -i https://target.com -d -o links.html # Look for: /api/v1/admin, /internal/, /debug, environment variables in JS bundles # SSL/TLS configuration assessment testssl.sh --severity MEDIUM https://target.com # Check for weak ciphers, BEAST, POODLE, etc.

Phase IV — Exploitation

The exploitation phase confirms that identified vulnerabilities are genuinely exploitable, demonstrates their impact, and chains vulnerabilities where possible to maximise the finding's risk rating. A single SQL injection becomes critical when chained with privilege escalation; a reflected XSS becomes high when combined with CSRF to achieve account takeover. All exploitation is conducted strictly within the authorised scope and with the minimum impact necessary to prove the finding's validity.

⟩ SQL Injection — Detection, Exploitation & Data Extraction
SQL injection remains one of the most impactful web vulnerabilities — enabling authentication bypass, complete database extraction, and in some configurations remote code execution. Manual SQL injection testing is essential alongside automated tools: SQLmap catches obvious injection points but misses parameter pollution, second-order injection in stored procedures, and ORM-level injection that requires understanding the application's specific query construction logic.
# Manual detection — probe for error-based and boolean-based SQLi # In Burp Repeater, test each parameter with basic injection probes: ' # Single quote — syntax error reveals SQLi '' # Escaped quote — no error suggests parameterised ' OR '1'='1 # Boolean TRUE — logic bypass test ' OR '1'='2 # Boolean FALSE — different response confirms SQLi 1; SELECT SLEEP(5)-- # Time-based blind SQLi — 5-second response delay # Automated exploitation with SQLmap — after manual confirmation sqlmap -u "https://target.com/api/products?id=1" \ --level 3 --risk 2 --dbs --batch sqlmap -u "https://target.com/api/products?id=1" \ -D targetdb -T users --dump --batch # POST body injection (pass Burp request file to SQLmap) sqlmap -r request.txt --level 3 --risk 2 --batch --dbs # Second-order SQLi — inject into profile fields that trigger queries later # Example: username = "admin'--" stored in DB # later query: SELECT * FROM users WHERE username='{stored_username}' # Triggers on profile load/search, not on registration # WAF bypass tampers — when standard payloads are blocked sqlmap -u URL --tamper space2comment,between,hex2char --batch --dbs
⟩ Authentication & Session Testing
Authentication vulnerabilities are disproportionately impactful — a broken login mechanism compromises every account in the application simultaneously. Testing must cover the full authentication lifecycle: login (brute force, credential stuffing, username enumeration), session management (token entropy, fixation, CSRF), MFA implementation (bypass via response manipulation, backup code exhaustion, race condition), and account recovery flows (predictable tokens, user enumeration via timing differences).
# Username enumeration — different responses reveal valid usernames ffuf -u https://target.com/api/login -X POST \ -H "Content-Type: application/json" \ -d '{"username":"FUZZ","password":"wrongpass"}' \ -w usernames.txt -mc 200 -ms 0 -fs 50 # Filter by response size difference # JWT analysis — decode, check algorithm, test alg:none attack python3 -c " import base64, json token = 'eyJhbGc...' header, payload, sig = token.split('.') print(json.loads(base64.b64decode(header + '==')))" # Check: alg:none vulnerability, RS256→HS256 confusion, weak secret (crack with hashcat) hashcat -a 0 -m 16500 jwt_token.txt /usr/share/wordlists/rockyou.txt # Password reset token entropy testing # Request 5+ reset tokens for the same account, analyse for patterns # Short tokens, time-based tokens, sequential tokens are predictable # Session token analysis — check randomness and expiration python3 -c " tokens = ['SESSION_TOKEN_1', 'SESSION_TOKEN_2'] # Capture multiple tokens # Analyse length, character set, entropy, and sequential patterns for t in tokens: print(len(t), set(t)) " # MFA bypass — response manipulation # Submit wrong OTP, intercept response in Burp # Change: {"success":false,"message":"Invalid code"} → {"success":true} # If bypass works: authentication logic trusts client-side response

OWASP Top 10 — Complete Reference

The OWASP Top 10 is the most widely referenced web application security standard — providing a consensus view of the most critical risk categories that every WAPT must address. The current OWASP Top 10 (2021) reflects the evolution of web application architectures toward API-first design, cloud deployment, and supply chain integration, with new entries specifically addressing inadequate access control, cryptographic failures, and software integrity concerns.

A01
Broken Access Control
The most prevalent critical risk category — users can access resources, data, or functionality beyond their intended permissions. Includes IDOR (accessing other users' records by manipulating IDs), privilege escalation, CORS misconfiguration, and forced browsing to unauthenticated API endpoints. Test every data endpoint with a different user's session to detect horizontal access control failures.
CRITICAL
A02
Cryptographic Failures
Sensitive data exposed due to absent or weak cryptography — cleartext transmission over HTTP, weak hashing algorithms (MD5, SHA1) for passwords, predictable encryption keys, missing encryption at rest for sensitive databases. Often the root cause of data breach compliance violations. Test by intercepting traffic, extracting database credential hashes, and assessing key management practices.
CRITICAL
A03
Injection
SQL injection, command injection, LDAP injection, SSTI (Server-Side Template Injection), and XSS — all resulting from insufficient input validation and unsafe data handling. SQLi and XSS remain among the highest-impact web vulnerabilities when present. Test all input fields, URL parameters, headers, and API bodies. SSTI is frequently overlooked: inject {{7*7}} and check for 49 in the response.
CRITICAL
A04
Insecure Design
Security flaws inherent in the application's design rather than its implementation — absent rate limiting enabling unlimited brute force, business logic that allows negative-quantity purchases, password reset flows that don't verify identity, and workflows with missing authorisation gates. Cannot be fixed by patching — requires architectural redesign. Identified through threat modelling and business logic review.
HIGH
A05
Security Misconfiguration
Default credentials, unnecessarily enabled features, overly permissive CORS, verbose error messages exposing stack traces and internal paths, missing security headers, and exposed cloud storage with public read. The most consistently found category in real assessments because it requires no exploitation skill — just systematic checking of known misconfiguration patterns. Use automated checks plus manual header review.
HIGH
A06
Vulnerable Components
Third-party libraries, frameworks, and dependencies with known CVEs — npm packages, Maven dependencies, Python packages, and server-side frameworks using outdated versions with published exploits. Worsened by the JavaScript ecosystem's deep dependency trees. Use Dependabot, Snyk, or OWASP Dependency-Check to identify vulnerable dependencies. Check Shodan for outdated server software versions.
HIGH
A07
Auth & Session Failures
Broken authentication implementations — weak password policies, absent brute force protection, insecure session tokens, JWT vulnerabilities, MFA that can be bypassed via response manipulation, and session tokens that persist after logout. Session tokens must have high entropy, limited lifetime, and be invalidated on logout and password change. JWT tokens must validate signature algorithm and use strong secrets.
CRITICAL
A08
Software Integrity Failures
Applications that don't verify the integrity of updates, plugins, libraries, or CDN-delivered scripts. An attacker who compromises a CDN script or a CI/CD pipeline can inject malicious code into thousands of deployments simultaneously. Test by checking whether CDN scripts have Subresource Integrity (SRI) hashes, whether CI/CD pipelines validate package signatures, and whether auto-update mechanisms verify authenticity.
MEDIUM
A09
Logging & Monitoring Failures
Insufficient logging, monitoring, and alerting for security-relevant events — failed logins, privilege escalation, admin account access, bulk data operations. When breaches occur, the absence of logs prevents forensic analysis and extends mean-time-to-detect. Test by performing obviously malicious actions (failed login storms, directory traversal) and checking whether the security team detects and responds within a reasonable timeframe.
MEDIUM
A10
SSRF
Server-Side Request Forgery — the server can be manipulated to make requests to unintended targets: internal cloud metadata services (169.254.169.254), internal APIs not exposed externally, and internal network services. Particularly impactful in cloud environments where metadata service access yields IAM credentials. Test any URL-accepting parameter with internal addresses, cloud metadata URLs, and file:// protocol handlers.
CRITICAL

Phase V — Post-Exploitation & Impact Demonstration

Post-exploitation in web application testing demonstrates the full business impact achievable from successful vulnerability exploitation — moving beyond "SQLi confirmed" to "SQLi leads to full database extraction, credential dumping, and authenticated access to the admin panel." Chaining vulnerabilities dramatically increases the demonstrated risk rating and the business-impact narrative that makes executives prioritise remediation. All post-exploitation must stay within the defined scope and document the minimum evidence necessary to demonstrate impact without causing actual data loss.

Privilege Escalation via Web
Chain low-privilege access into administrative access — JWT role manipulation (changing role:user to role:admin in the payload), parameter pollution in profile update endpoints (adding admin:true to the POST body), IDOR from one user to an admin account, or cookie tampering where privilege levels are stored client-side. Document every step and screenshot the administrative access achieved.
Data Access Scope Assessment
From a confirmed exploitation point, systematically enumerate what data is accessible — map all database tables reachable via SQLi, document all API endpoints accessible with elevated tokens, and enumerate cloud storage accessible from SSRF. Provide the client with a precise scope of the data exposure: "37,412 user records including email, password hash, and payment information accessible."
SSRF to Cloud Metadata
From a confirmed SSRF vulnerability, test internal cloud metadata service access — AWS: 169.254.169.254/latest/meta-data/iam/security-credentials/, Azure: 169.254.169.254/metadata/instance, GCP: metadata.google.internal. Successful metadata service access yields temporary IAM credentials — the most critical cloud escalation path from a web application vulnerability.
Persistence & Backdoor Assessment
Demonstrate how an attacker maintains access after the initial compromise — web shell deployment via file upload vulnerability, persistent XSS in admin-visible stored content, OAuth token theft enabling long-lived access without credentials. All deployed persistence mechanisms must be removed and documented in the engagement log immediately after demonstration, with a cleanup log provided to the client.
Lateral Movement via Web
From web application compromise, demonstrate internal network access — SSRF to internal services unreachable from the internet, server-side code execution pivoting to internal network scanning, web application credentials reused on internal services. Documents the blast radius from a single web application compromise into the broader infrastructure.
Vulnerability Chaining
The most impactful finding is always a chain: reflected XSS + CSRF = account takeover without any user interaction; SQLi + misconfigured DB user = OS command execution; IDOR + PII access + no logging = undetected mass data exfiltration. Chain findings from different OWASP categories to demonstrate the highest-impact realistic attack path against the specific application architecture.

AI Integration in Web Application Security

Artificial Intelligence is fundamentally changing web application security from both the offensive and defensive sides. On the attack side, LLMs accelerate vulnerability research, generate targeted payloads for specific technology stacks, and assist in understanding complex application logic from source code or decompiled output — compressing the most time-intensive phases of WAPT. On the defensive side, AI-powered WAFs, runtime application self-protection (RASP), and code analysis tools are raising the bar for both exploitation difficulty and detection speed.

The most significant development is AI-assisted vulnerability discovery: LLMs can analyse JavaScript bundles, API response schemas, and application source code to identify vulnerability patterns that manual review would miss in the available engagement time — effectively acting as an always-available senior security reviewer that never gets tired of reading code.

[ AI ]

Research from leading security teams shows that LLMs provided with web application source code or decompiled JavaScript can identify exploitable IDOR patterns, authentication bypass opportunities, and injection sinks at a rate competitive with junior-to-mid-level human security reviewers — and do it in seconds rather than hours. For black-box assessments, AI can analyse API response schemas and infer likely backend query patterns, suggesting targeted injection probes before a single payload is sent.

AI-Powered Offensive WAPT Capabilities

🔍
AI Source Code Vulnerability Analysis
LLMs analyse JavaScript bundles, decompiled source, or provided source code to identify dangerous function calls, injection sinks, authentication logic flaws, and insecure data handling — producing a prioritised vulnerability list from thousands of lines of code in seconds. Particularly effective for finding business logic issues and second-order injection points that automated scanners miss entirely.
💉
Intelligent Payload Generation
AI generates targeted injection payloads customised to the specific technology stack, WAF signatures, and input context observed during testing — going beyond generic wordlists to create payloads that encode, fragment, and obfuscate specifically for the detected defensive environment. Adapts iteratively based on observed WAF responses, converging on effective bypass payloads faster than manual trial-and-error.
🧩
Business Logic Flaw Discovery
LLMs analyse application workflow documentation, API schemas, and observed request sequences to reason about business logic assumptions — identifying workflows that can be broken by reordering steps, manipulating state transitions, or submitting unexpected parameter combinations. Discovers the class of vulnerability that scanners are fundamentally unable to identify because it requires understanding the application's intended behaviour.
🔗
Vulnerability Chain Analysis
AI models process the full finding set from a WAPT engagement and automatically identify vulnerability chains — determining which combinations of lower-severity findings can be chained to produce Critical-level impact. Produces a risk-adjusted finding priority that accounts for chain exploitability rather than treating each finding in isolation, resulting in more accurate business risk communication.
🌐
API Schema Exploitation
LLMs analyse REST API documentation, OpenAPI/Swagger specs, and GraphQL schemas to automatically identify high-value endpoints, IDOR-susceptible parameters, missing authentication requirements, and mass assignment opportunities — generating a targeted API test plan from schema analysis alone before any active testing begins.
📝
Automated WAPT Report Generation
LLMs convert raw WAPT evidence — Burp request/response pairs, SQLmap output, XSS proof-of-concept scripts — into complete, OWASP-aligned finding entries with CVSSv3.1 scores, CWE references, business impact narratives, and developer-actionable remediation guidance in the specific framework and language of the vulnerable code. Eliminates the reporting bottleneck that follows intensive assessments.

AI-Powered Web Application Defence

🛡
AI-Powered WAF
Next-generation WAFs (Cloudflare, AWS WAF with ML, Imperva) use ML models trained on millions of attack patterns to classify requests as malicious or legitimate — detecting SQLi, XSS, and SSRF payloads including heavily obfuscated and encoded variants that bypass rule-based signature matching. Models retrain continuously on observed attack data, narrowing the window between novel technique emergence and detection.
🔬
Runtime Application Self-Protection (RASP)
RASP agents embedded in the application runtime use ML to monitor actual execution context — detecting SQL queries being constructed from user input, command strings being built from tainted data, or file paths being traversed based on external input. Detects exploitation attempts from inside the application rather than at the network perimeter, providing protection independent of WAF bypass techniques.
📊
AI SAST in Development
ML-enhanced SAST tools (GitHub Advanced Security, Semgrep, Checkmarx) reduce false positive rates by learning which flagged patterns result in real vulnerabilities versus acceptable code idioms — enabling developers to trust and act on scanner output rather than dismissing it as noise. CI/CD integrated AI SAST catches injection sinks, authentication issues, and cryptographic failures before code reaches production.
🔭
API Behaviour Analytics
AI models baseline normal API usage patterns per consumer — expected endpoints, parameter structures, request volumes, and data access patterns. Anomalous API behaviour (mass IDOR enumeration, unusual bulk export, access to endpoints inconsistent with the consumer's role) triggers real-time alerts that rule-based API gateways cannot generate without an enormous false positive rate.
🔮
Predictive Vulnerability Management
ML models analyse the application's dependency graph against the current CVE landscape to predict which third-party components are most likely to have exploitable vulnerabilities disclosed in the next quarter — enabling proactive patching before a public exploit is available. Integrates with package managers to surface risk-adjusted upgrade recommendations across the full dependency tree.
🤖
AI-Driven DAST Optimisation
ML models guide Dynamic Application Security Testing — learning which input locations, parameter types, and request sequences are most likely to trigger vulnerabilities based on observed application behaviour, and concentrating fuzzing effort accordingly. Produces higher-confidence findings in shorter scan windows compared to undirected full-surface scanning, reducing CI/CD pipeline delay from security testing.
