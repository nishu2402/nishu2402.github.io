Google Search Operators -- Core Syntax

Google Dorking (also called Google Hacking) uses advanced Google search operators to find information that is publicly indexed but not intended to be easily discoverable. First documented by Johnny Long in the Google Hacking Database (GHDB), dorking leverages the gap between what organisations inadvertently expose to search engine crawlers and what they believe is private. The technique requires no tools, no exploits, and no special access -- just precise knowledge of how Google's search syntax works.

Google indexes everything it can reach. Misconfigured servers, forgotten staging environments, publicly accessible admin panels, exposed database exports, and documents containing credentials are routinely indexed because the server never sent a Disallow directive in robots.txt, or because the file was temporarily public and Google cached it before it was restricted.

site:
site:domain.com [query]
Restrict results to a specific domain or subdomain. The most fundamental recon operator -- maps the entire indexed footprint of a target organisation.
filetype:
filetype:pdf site:target.com
Filter results by file extension. Works with pdf, doc, docx, xls, xlsx, ppt, sql, log, txt, xml, json, bak, env, and many others.
intitle:
intitle:"index of"
Match text in the page title. Used to find directory listings, admin panels, login pages, and application-specific interfaces.
inurl:
inurl:admin inurl:login
Match text in the URL path. Finds admin interfaces, login portals, configuration pages, and API endpoints by their URL patterns.
intext:
intext:"password" filetype:log
Match text in the page body. Used to find documents containing specific strings -- credentials, API keys, error messages, and sensitive content.
cache:
cache:target.com/page
View Google's cached version of a page. Useful when a target has removed sensitive content -- the cache may still contain the original indexed version.
related:
related:target.com
Find websites Google considers similar or related to the target -- reveals sister companies, subsidiaries, partner organisations, and shared infrastructure.
link: / info:
info:target.com
Retrieve information Google has about a specific URL -- cached version, similar pages, pages linking to it. Useful for initial recon of a specific page.
OR / AND / -
term1 OR term2 -exclude
Boolean operators: OR expands results, AND (default) requires both terms, minus (-) excludes. Chain with other operators for precise compound queries.
" " (exact)
"exact phrase here"
Exact phrase matching. Forces Google to return only results containing the complete phrase -- critical for finding specific strings like error messages or default banners.
* (wildcard)
"password * 2024"
Wildcard -- matches any word in that position within an exact phrase. Useful when part of a pattern is unknown or varies across targets.
before: / after:
site:target.com after:2023-01-01
Filter results by indexing date. Find recently exposed files, new pages on a target, or trace when sensitive content first appeared in search results.

Reconnaissance Dorks -- Target Mapping

Reconnaissance dorking maps the indexed surface of a target organisation without making a single request to the target's servers. All queries go to Google -- the target never sees your IP address, no IDS rule fires, no WAF logs the request. This passive recon phase identifies subdomains, exposed directories, login portals, technology stack, employee information, and infrastructure details that inform all subsequent active testing phases.

[ * ]

Passive by nature -- not illegal. Google dorking queries search a public index. No connection is made to the target. However, accessing exposed content you discover (downloading files, logging into found panels) may cross legal boundaries without authorisation. Use for asset discovery and exposure assessment in authorised engagements only.

site:target.com -site:www.target.com
Enumerate all subdomains indexed by Google for a target domain, excluding the main www site. Reveals development environments (dev.target.com), staging servers (staging.target.com), internal tools (jira.target.com, jenkins.target.com), and forgotten subdomains that may not appear in DNS brute force results because they are rarely queried.
Subdomain Enumeration
site:target.com intitle:"index of" "parent directory"
Finds open directory listings on the target -- Apache and Nginx servers with autoindex enabled that expose their file structure. Open directories commonly contain backup archives, log files, configuration files, source code dumps, and uploaded files that were never intended to be publicly accessible.
Open Directory Listing
site:target.com inurl:wp-admin OR inurl:administrator OR inurl:login OR inurl:signin
Discovers all login portals and admin interfaces on the target indexed by Google. Identifies the CMS (WordPress wp-admin, Joomla administrator, Drupal user/login), custom admin panels, and authentication pages -- the entry points for credential attacks and unauthorised access attempts in authorised engagements.
Login Panel Discovery
site:target.com ext:php inurl:?id= OR inurl:?page= OR inurl:?file=
Identifies PHP pages with URL parameters that accept user input -- the classic indicators of potential SQL injection, LFI (Local File Inclusion), and RFI (Remote File Inclusion) attack surfaces. Finding parameterised URLs via dorking is faster than crawling the entire site and surfaces the highest-risk input points first.
Input Parameter Discovery
site:target.com filetype:pdf OR filetype:docx OR filetype:xlsx
Retrieves all indexed documents from the target domain. Office and PDF documents frequently contain metadata (author names, internal server paths, software version strings, Active Directory usernames), internal project information, client data, financial figures, and sometimes embedded credentials in spreadsheets or configuration documents.
Document & Metadata Harvest
site:target.com intext:"Powered by" OR intext:"Running on" OR intext:"Apache" OR intext:"nginx"
Technology stack fingerprinting via indexed pages. Default server banners, CMS footer strings, framework acknowledgements, and error pages often reveal the exact versions of web servers, databases, frameworks, and CMS platforms -- enabling precise vulnerability lookup against known CVEs for those versions.
Technology Fingerprinting

Sensitive Data Exposure Dorks

The most critical class of Google dorks finds sensitive data that should never be publicly indexed -- credentials, API keys, private keys, database exports, configuration files, and internal documents. These exposures occur when developers accidentally commit secrets to public repositories, when configuration files are uploaded to web roots, when database exports are left in public directories, or when staging environments are indexed before access controls are applied. The GHDB (Google Hacking Database) at exploit-db.com/google-hacking-database catalogues thousands of these dork patterns.

intitle:"index of" ".env" OR "env" filetype:env
Finds exposed .env files -- the most dangerous single class of Google dork. Laravel, Django, Next.js, and virtually every modern web framework uses .env files to store database credentials, API keys, JWT secrets, AWS credentials, and mail server passwords. A single exposed .env file typically yields complete application compromise: database access, cloud account takeover, and third-party service access in a single file.
CRITICAL -- Environment Files
intitle:"index of" "id_rsa" OR "id_dsa" OR filetype:pem intext:"BEGIN RSA PRIVATE KEY"
Searches for exposed private SSH keys and SSL/TLS certificate private keys. SSH private keys in open directories or accidentally committed to accessible paths grant direct server access to any system configured to accept that key. SSL private keys in exposed PEM files enable decryption of historical HTTPS traffic captured via MITM and certificate impersonation attacks.
CRITICAL -- Private Keys
site:target.com intext:"password" filetype:log OR filetype:txt OR filetype:cfg
Finds log files, text files, and configuration files containing the word "password" on the target domain. Application logs frequently record authentication errors (including the attempted password), configuration files store database passwords in plaintext, and error reports include credential strings from failed connection attempts. These files are often left world-readable on misconfigured web servers.
Credentials in Files
site:target.com filetype:sql intext:"INSERT INTO" OR intext:"CREATE TABLE"
Discovers exposed SQL database dumps. Database administrators routinely export SQL dumps for backups, migrations, and development environments and occasionally leave them in web-accessible directories. These files contain the complete database structure and all data -- including user tables with hashed (or sometimes plaintext) passwords, personal information, financial records, and application configuration data.
Database Dumps
site:target.com filetype:xml inurl:config OR inurl:configuration OR inurl:settings
Finds XML configuration files -- web.config (ASP.NET), applicationContext.xml (Spring), struts.xml, and similar framework configuration files. These files commonly contain database connection strings with embedded credentials, SMTP server authentication, LDAP bind credentials, and encryption keys used by the application. ASP.NET web.config exposure is particularly impactful as it frequently contains machine keys used for session token forging.
Config Files
site:github.com OR site:gitlab.com "target.com" "api_key" OR "secret_key" OR "password"
Searches public code repositories for secrets associated with a target organisation. Developers frequently hardcode API keys, database credentials, and tokens directly in source code and push them to public repositories. This single dork pattern is responsible for a disproportionate share of all cloud account compromises -- AWS access keys discovered in GitHub repositories are found and abused within seconds of being pushed by automated scanners.
Code Repository Leaks

Vulnerability Discovery Dorks

Advanced dorks identify specific vulnerability indicators -- application error messages that reveal software versions and debug information, exposed management interfaces that should not be internet-accessible, unpatched CMS installations identifiable by their default pages, misconfigured cloud services, and web applications exhibiting behaviour characteristic of SQL injection or XSS vulnerabilities. These dorks form the passive vulnerability discovery phase before any active scanning begins in an authorised engagement.

intext:"SQL syntax" OR intext:"mysql_fetch" OR intext:"ORA-01756" OR intext:"ODBC Error"
Finds pages displaying raw database error messages -- the definitive indicator of potential SQL injection. Verbose SQL errors reveal the database type (MySQL, Oracle, MSSQL, PostgreSQL), table and column names from failed queries, the database version, and sometimes the complete query string. These errors occur when unsanitised user input breaks the SQL syntax and the error is reflected back in the HTTP response.
SQL Error / Injection Indicator
inurl:"wp-content/plugins" site:target.com filetype:txt intext:"Version"
Enumerates WordPress plugin versions via publicly accessible readme.txt files that plugins include by default. These files explicitly state the plugin name and version number, enabling precise lookup against the WPScan vulnerability database for known CVEs against that specific plugin version. Cross-reference discovered versions against CVEs in the WordPress vulnerability database (wpscan.com) and exploit-db.com.
WordPress Plugin Exposure
intitle:"phpinfo()" intext:"PHP Version" site:target.com
Finds exposed phpinfo() output pages -- one of the most information-rich discoveries possible. phpinfo() reveals: exact PHP version (cross-reference CVEs), all loaded PHP extensions and their versions, server path (DOCUMENT_ROOT, SERVER_SOFTWARE), environment variables (which may contain credentials), file upload settings, disable_functions list, and the contents of php.ini. This single page provides complete server fingerprinting.
phpinfo() Exposure
intitle:"Dashboard" inurl:dashboard site:target.com -login
Discovers exposed admin dashboards accessible without authentication. Jenkins, Grafana, Kibana, Kubernetes dashboards, and many other platforms have historically been deployed with no authentication by default or with authentication disabled for internal network use -- and are then inadvertently exposed to the internet. Finding an unauthenticated Jenkins dashboard typically means full RCE via the Groovy script console.
Unauthenticated Dashboards
site:target.com inurl:".git" OR inurl:"/.git/HEAD" OR intitle:"index of /.git"
Finds exposed .git directories on web servers. When a Git repository is deployed to a web root without blocking access to the .git folder, the entire repository history is downloadable -- including all commits, branches, and crucially all previously committed secrets that may have been removed in later commits. The tool gitdumper (or git-dumper) can reconstruct the entire repository from an exposed .git directory.
Git Repository Exposure

Dorking Tools & Automation

Manual Google dorking is effective but slow -- running 50+ dork queries against a target one at a time is tedious and rate-limited. Dedicated dorking tools automate query execution, parse results, remove duplicates, and categorise findings by risk level. The professional dorking toolkit combines Google's own API, GHDB query libraries, and custom automation scripts to reduce a full target dorking run from hours of manual work to minutes of automated scanning.

GHDB
Database
Google Hacking Database at exploit-db.com/google-hacking-database -- the canonical reference with 8,000+ categorised dork queries maintained by Offensive Security
GoogleDorker
Python Tool
github.com/six2dez/dorks_hunter -- automated target-specific dorking with GHDB integration and HTML report output
DorkScout
Go Tool
High-performance concurrent dorking tool with proxy rotation support and multiple search engine backends including Google, Bing, and DuckDuckGo
Pagodo
Python Tool
Automates running GHDB dorks against targets. Reads dorks from GHDB export, submits to Google, collects results. Supports proxy rotation to avoid rate limiting.
theHarvester
Multi-Source
Classic OSINT tool combining Google, Bing, LinkedIn, Hunter.io, and others for email, subdomain, and host discovery with dorking capabilities built in
Shodan
IoT Search
Internet-of-Things search engine -- the "Google for exposed services". Searches device banners, certificates, and open ports. Complements Google dorking for infrastructure discovery.

Professional Dorking Methodology

Effective Google dorking follows a structured methodology rather than random query execution. The professional workflow moves through five phases: scope mapping (what does Google know about this target?), infrastructure discovery (subdomains, technologies, entry points), sensitive data enumeration (credentials, keys, internal documents), vulnerability identification (error pages, exposed management interfaces, outdated software), and findings documentation (evidence capture, risk rating, remediation).

AI Integration in Google Dorking & OSINT

AI transforms Google dorking from a manual skill-dependent process into an intelligent automated intelligence-gathering pipeline. Frontier LLMs understand Google search operator syntax deeply enough to generate highly targeted dork queries from plain-English requirements, analyse discovered content for hidden intelligence value, correlate findings across dozens of sources into coherent attack surface maps, and produce professional OSINT reports at a speed and depth that manual analysis cannot match.

The most impactful shift: AI security tools are moving from query assistants to autonomous OSINT agents -- generating queries, executing them via the Google Custom Search API, interpreting results, generating follow-up queries based on discoveries, and producing final reports entirely automatically. What previously took a skilled analyst a full day of dorking now takes an AI pipeline under an hour with greater coverage.

[ AI ]

Security researchers using AI-assisted dorking workflows report generating 50+ targeted dork queries for a specific target in seconds, with AI adapting the query strategy based on initial discoveries. AI interprets ambiguous results, identifies high-value findings that human review might miss, and correlates information across multiple sources into a comprehensive exposure assessment -- all before a single active scan touches the target.

" "Return ONLY dorks, one per line, no numbering or explanation.\n\n" f"Organisation: {org}\nDomain: {domain}\nProfile: {profile}") r = openai.OpenAI().chat.completions.create( model=MODEL, messages=[{"role":"user","content":prompt}], temperature=0.3 ) return [d.strip() for d in r.choices[0].message.content.splitlines() if d.strip()] def google_search(query, n=5): r = requests.get("https://www.googleapis.com/customsearch/v1", params={"key":GOOGLE_API,"cx":GOOGLE_CX,"q":query,"num":n}, timeout=10) return [{"title":i["title"],"url":i["link"],"snippet":i.get("snippet","")} for i in r.json().get("items",[])] def ai_analyse(results, org): prompt = (f"You are a senior OSINT analyst. Analyse these Google dorking results for {org}. " "For each significant finding provide:\n" "1. SENSITIVITY: what sensitive data or access is exposed\n" "2. RISK: Critical / High / Medium / Low\n" "3. ATTACK VECTOR: how an attacker exploits this\n" "4. NEXT DORKS: follow-up queries to investigate further\n\n" "Results:\n" + json.dumps(results, indent=2)) r = openai.OpenAI().chat.completions.create( model=MODEL, messages=[{"role":"user","content":prompt}]) return r.choices[0].message.content org = "TargetCorp"; domain = "target.com"; profile = "UK financial services, Azure" dorks = generate_dorks(org, domain, profile) print(f"[*] Generated {len(dorks)} dorks") all_results = [] for dork in dorks[:10]: res = google_search(dork) if res: all_results.append({"dork":dork,"results":res}); print(f" [+] {dork[:55]}") print("\n[*] AI Analysis:\n") print(ai_analyse(all_results, org))
