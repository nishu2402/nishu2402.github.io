Nmap Fundamentals & Installation

Nmap (Network Mapper) was created by Gordon Lyon (Fyodor) in 1997 and is the most widely used network scanning tool in existence. It is open-source, cross-platform, and ships pre-installed in Kali Linux, Parrot OS, and BlackArch. Every major penetration testing methodology -- PTES, OWASP, OSSTMM, and the CPENT/OSCP curricula -- begins with Nmap. Its core capabilities include host discovery, port scanning, service and version detection, OS fingerprinting, and a powerful scripting engine (NSE) with over 600 built-in scripts covering vulnerability detection, authentication testing, and service enumeration.

Nmap operates by crafting and sending specially formed IP packets to target hosts and analysing the responses. The type of packets sent and how responses are interpreted is determined by the scan type chosen. Understanding the mechanics behind each scan type -- which packets are sent, what response indicates open/closed/filtered -- is essential for using Nmap effectively and interpreting its output correctly in professional engagements.

# Install Nmap sudo apt install nmap # Kali / Parrot / Ubuntu / Debian brew install nmap # macOS # Windows: https://nmap.org/download.html (includes Zenmap GUI) nmap --version # Verify and show version # Your very first scans nmap 192.168.1.10 # Default scan: top 1000 TCP ports nmap 192.168.1.0/24 # Scan entire /24 subnet nmap scanme.nmap.org # Legal public test target # Understanding Nmap port states # open -- application actively accepting connections # closed -- accessible but no application listening # filtered -- firewall/filter blocking -- Nmap cannot determine state # unfiltered -- accessible but open/closed undetermined (ACK scan only) # open|filtered -- open or filtered, cannot distinguish # closed|filtered -- closed or filtered (IP ID idle scan only) # Essential output flags -- always save your scans nmap 192.168.1.10 -oN scan.txt # Normal (human-readable) nmap 192.168.1.10 -oX scan.xml # XML (parse with tools/AI) nmap 192.168.1.10 -oG scan.gnmap # Grepable nmap 192.168.1.10 -oA scan # All three formats simultaneously nmap 192.168.1.10 -v # Verbose output nmap 192.168.1.10 -vv # Double verbose

Host Discovery Techniques

Host discovery (ping scanning) is the phase before port scanning -- identifying which hosts in the target range are alive before spending time scanning individual ports. Nmap provides numerous host discovery techniques because different network configurations block different probe types. ICMP is frequently blocked by firewalls; TCP SYN to common ports often bypasses them. Understanding which technique to use based on network conditions is a key skill that separates experienced Nmap users from beginners.

# DEFAULT DISCOVERY (combined probe -- most useful for local networks) nmap -sn 192.168.1.0/24 # Ping scan only, no port scan # ICMP probes sudo nmap -sn -PE 192.168.1.0/24 # ICMP echo request (classic ping) sudo nmap -sn -PP 192.168.1.0/24 # ICMP timestamp request sudo nmap -sn -PM 192.168.1.0/24 # ICMP address mask request # TCP probes (bypass ICMP blocks) sudo nmap -sn -PS80,443,22 192.168.1.0/24 # TCP SYN to specified ports sudo nmap -sn -PA80,443 192.168.1.0/24 # TCP ACK (requires root) # UDP probe sudo nmap -sn -PU53 192.168.1.0/24 # UDP to port 53 (DNS) # ARP scan (fastest, local network only -- bypasses all firewalls) sudo nmap -sn -PR 192.168.1.0/24 # Skip discovery entirely (treat all hosts as up) sudo nmap -Pn 192.168.1.10 # Assume host is up, go straight to port scan # Use -Pn when host blocks all probes but you know it exists # Read targets from file nmap -sn -iL targets.txt # One IP/range/hostname per line # Exclude specific hosts from scan nmap -sn 192.168.1.0/24 --exclude 192.168.1.1,192.168.1.254 nmap -sn 192.168.1.0/24 --excludefile exclude.txt # Parse results: extract live hosts from grepable output grep "Up" scan.gnmap | awk '{print $2}' > live_hosts.txt

Port Scanning Techniques

Nmap implements over a dozen distinct port scanning techniques, each with different detection profiles, accuracy levels, and requirements. The choice of scan type determines stealth, speed, and completeness. SYN scan (-sS) is the professional default -- it never completes the TCP handshake, making it faster and stealthier than connect scan. UDP scan (-sU) is essential but slow -- most network auditors miss critical UDP services (DNS, SNMP, TFTP) by skipping it entirely.

# TCP SCAN TYPES sudo nmap -sS 192.168.1.10 # SYN scan (stealth) -- DEFAULT, requires root # Sends SYN, gets SYN/ACK (open) or RST (closed) # Never completes handshake -- harder to log nmap -sT 192.168.1.10 # TCP Connect -- full 3-way handshake (no root needed) # Logged by target. Slower than SYN scan. sudo nmap -sA 192.168.1.10 # ACK scan -- map firewall rules, NOT open ports # RST=unfiltered No response=filtered sudo nmap -sW 192.168.1.10 # Window scan -- like ACK but examines TCP window sudo nmap -sN 192.168.1.10 # NULL scan -- no flags set sudo nmap -sF 192.168.1.10 # FIN scan -- only FIN flag sudo nmap -sX 192.168.1.10 # Xmas scan -- FIN+PSH+URG flags # NULL/FIN/Xmas: bypass some stateless firewalls; fail against Windows sudo nmap -sM 192.168.1.10 # Maimon scan -- FIN+ACK (bypasses some BSD filters) # UDP SCAN (critical -- do not skip) sudo nmap -sU 192.168.1.10 # UDP top 1000 ports (slow) sudo nmap -sU --top-ports 100 192.168.1.10 # UDP top 100 (faster) sudo nmap -sU -sS 192.168.1.10 # Combine UDP + SYN scan # PORT RANGE SPECIFICATION nmap -p 80 192.168.1.10 # Single port nmap -p 80,443,8080 192.168.1.10 # Port list nmap -p 1-1000 192.168.1.10 # Port range nmap -p- 192.168.1.10 # All 65535 ports (thorough) nmap --top-ports 100 192.168.1.10 # Top 100 most common ports nmap -F 192.168.1.10 # Fast scan: top 100 ports nmap -p http,ftp,ssh 192.168.1.10 # By service name # TIMING TEMPLATES (T0=slowest/stealthy, T5=fastest/aggressive) nmap -T0 192.168.1.10 # Paranoid IDS evasion nmap -T1 192.168.1.10 # Sneaky nmap -T2 192.168.1.10 # Polite -- throttled nmap -T3 192.168.1.10 # Normal -- default nmap -T4 192.168.1.10 # Aggressive -- pentest standard nmap -T5 192.168.1.10 # Insane -- may miss ports # THE PROFESSIONAL FULL SCAN (pentest gold standard) sudo nmap -sS -sV -sC -O -A -p- -T4 192.168.1.10 -oA fullscan

Service & Version Detection

Knowing a port is open is only half the information. Service and version detection (-sV) is the flag that transforms a list of open ports into actionable intelligence -- identifying the exact software, version number, and sometimes build date of every service listening on the target. This is what powers CVE lookup and exploit selection. An open port 80 tells you "there is a web server"; -sV tells you "Apache httpd 2.4.49 on Ubuntu Linux" -- which maps directly to CVE-2021-41773 (path traversal RCE).

# BASIC VERSION DETECTION nmap -sV 192.168.1.10 # Detect service versions # VERSION INTENSITY LEVELS (0=lightest, 9=try everything) nmap -sV --version-intensity 0 192.168.1.10 # Only most likely probes nmap -sV --version-intensity 5 192.168.1.10 # Default nmap -sV --version-intensity 9 192.168.1.10 # Comprehensive (slow) # SHORTCUTS nmap -sV --version-light 192.168.1.10 # Intensity 2 -- fast nmap -sV --version-all 192.168.1.10 # Intensity 9 -- thorough # BANNER GRABBING only (no probe -- read whatever service sends) nmap -sV --version-intensity 0 -p 21,22,25,80 192.168.1.10 # COMBINE with script scan for maximum intelligence nmap -sV -sC 192.168.1.10 # Version + default NSE scripts nmap -sV -A 192.168.1.10 # -A = -sV -sC -O --traceroute # RPC scan (for NFS, NIS, and RPC services) sudo nmap -sR 192.168.1.10 # RPC program version scan # Extract versions from XML for CVE lookup pipeline nmap -sV -oX versions.xml 192.168.1.10 grep "product=" versions.xml | grep "version=" # Then feed to AI pipeline for automatic CVE matching

OS Fingerprinting

OS detection (-O) works by sending a series of specifically crafted TCP, UDP, and ICMP packets to open and closed ports, then analysing the responses against a database of 2,600+ known OS fingerprints. Differences in TCP stack implementation between operating systems -- initial sequence numbers, IP TTL values, TCP window sizes, response to invalid flags -- create a unique fingerprint that Nmap matches against its database. OS detection requires at least one open and one closed port to work reliably.

# BASIC OS DETECTION sudo nmap -O 192.168.1.10 # OS DETECTION OPTIONS sudo nmap -O --osscan-limit 192.168.1.10 # Only scan hosts with open+closed port sudo nmap -O --osscan-guess 192.168.1.10 # Guess aggressively when not certain sudo nmap -O --max-os-tries 5 192.168.1.10 # Retry OS probe 5 times # OS + VERSION + SCRIPTS (most comprehensive) sudo nmap -A 192.168.1.10 # -A enables: OS detection, version detection, script scanning, traceroute # Understand OS detection output # "OS details: Microsoft Windows 10 1903-20H2" = high confidence match # "Aggressive OS guesses: Linux 3.11-4.1 (93%)" = confident guess # "No exact OS matches" = requires --osscan-guess or needs open+closed port # TTL hints for manual OS guessing (when Nmap OS fails) # TTL 64 = Linux / macOS / Android # TTL 128 = Windows # TTL 255 = Cisco / network devices ping -c 1 192.168.1.10 | grep ttl # Submit unknown fingerprint to Nmap.org database sudo nmap -O --send-ip 192.168.1.10 # Use raw IP for better accuracy

Nmap Scripting Engine (NSE)

The Nmap Scripting Engine (NSE) is the most powerful feature in Nmap and the capability that separates it from basic port scanners. NSE scripts are written in Lua and extend Nmap's functionality with service enumeration, vulnerability detection, authentication testing, backdoor detection, and malware discovery. There are currently over 600 official NSE scripts organised into 14 categories. Writing custom NSE scripts allows penetration testers to create bespoke detection capabilities for specific targets or custom vulnerability checks.

# NSE SCRIPT CATEGORIES # auth -- authentication bypass and credential testing # broadcast -- network broadcast discovery # brute -- brute-force credential testing # default -- safe, useful scripts run with -sC # discovery -- service enumeration and information gathering # dos -- denial of service testing (use carefully) # exploit -- vulnerability exploitation # external -- queries third-party databases # fuzzer -- sends unexpected/fuzzed inputs # intrusive -- not safe to run without authorisation # malware -- checks for malware/backdoors # safe -- unlikely to crash services # version -- version detection enhancement # vuln -- vulnerability detection # RUN SCRIPTS nmap -sC 192.168.1.10 # Default scripts only nmap --script vuln 192.168.1.10 # All vuln scripts nmap --script safe 192.168.1.10 # All safe scripts nmap --script discovery 192.168.1.10 # Discovery category nmap --script "http-*" 192.168.1.10 # All HTTP scripts nmap --script "smb-vuln-*" 192.168.1.10 # All SMB vuln scripts # HIGH-VALUE INDIVIDUAL SCRIPTS sudo nmap --script smb-vuln-ms17-010 192.168.1.10 # EternalBlue check sudo nmap --script smb-vuln-ms08-067 192.168.1.10 # Conficker era vuln sudo nmap --script http-shellshock 192.168.1.10 # ShellShock test sudo nmap --script ssl-heartbleed 192.168.1.10 # Heartbleed test sudo nmap --script ssl-poodle 192.168.1.10 # POODLE test sudo nmap --script ftp-anon 192.168.1.10 # Anonymous FTP check sudo nmap --script ftp-brute 192.168.1.10 # FTP brute force sudo nmap --script http-enum 192.168.1.10 # Web directory enum sudo nmap --script http-auth-finder 192.168.1.10 # Find auth methods sudo nmap --script dns-zone-transfer --script-args "dns-zone-transfer.domain=target.com" ns1.target.com # SCRIPT WITH ARGUMENTS nmap --script http-brute --script-args "userdb=users.txt,passdb=rockyou.txt" 192.168.1.10 nmap --script http-title --script-args "http.useragent=Mozilla/5.0" 192.168.1.0/24 # FIND AND INSPECT SCRIPTS ls /usr/share/nmap/scripts/ | grep smb nmap --script-help smb-vuln-ms17-010 # Firewall Evasion & IDS Bypass

Real-world penetration tests frequently encounter firewalls, IDS/IPS systems, and WAFs that block or detect standard Nmap scans. Nmap provides a comprehensive set of evasion techniques including packet fragmentation, decoy scanning, source port manipulation, timing adjustment, and IP spoofing. Understanding these techniques is essential both for evading detection during authorised engagements and for understanding what your own defences can and cannot detect.

# PACKET FRAGMENTATION (split packets confuse stateless firewalls) sudo nmap -f 192.168.1.10 # Fragment into 8-byte pieces sudo nmap -ff 192.168.1.10 # Fragment into 16-byte pieces sudo nmap --mtu 16 192.168.1.10 # Custom MTU (must be multiple of 8) # DECOY SCANNING (disguise real source among fake decoys) sudo nmap -D RND:10 192.168.1.10 # 10 random decoys sudo nmap -D 10.0.0.1,10.0.0.2,ME 192.168.1.10 # Specific decoys + real # Logs show multiple source IPs -- harder to isolate real attacker # SOURCE PORT MANIPULATION (bypass trust rules on specific ports) sudo nmap --source-port 53 192.168.1.10 # Appear as DNS traffic sudo nmap --source-port 80 192.168.1.10 # Appear as HTTP traffic # TIMING MANIPULATION (slow scan to avoid rate-based IDS) nmap -T0 192.168.1.10 # Paranoid: 5min between probes nmap --scan-delay 5s 192.168.1.10 # Custom delay between probes nmap --max-rate 10 192.168.1.10 # Max 10 packets/second # IP SPOOFING (requires same-subnet or specific routing) sudo nmap -S 10.0.0.5 -e eth0 192.168.1.10 # Spoof source IP # RANDOMISE TARGET ORDER (avoid sequential scan signatures) nmap --randomize-hosts 192.168.1.0/24 # APPEND RANDOM DATA (pad packets to avoid signature matching) nmap --data-length 25 192.168.1.10 # Add 25 random bytes to packets # IDLE/ZOMBIE SCAN (completely blind -- real source never touches target) sudo nmap -sI zombie_host 192.168.1.10 # Zombie must have predictable IP ID # SPOOF MAC ADDRESS sudo nmap --spoof-mac 0 192.168.1.10 # Random MAC sudo nmap --spoof-mac Apple 192.168.1.10 # Spoof Apple vendor MAC # COMBINED EVASION (professional engagement approach) sudo nmap -sS -f -T2 -D RND:8 --source-port 53 --data-length 16 192.168.1.10

AI Integration in Nmap Workflows

Nmap produces rich structured output -- especially in XML format -- that is ideal for AI analysis. LLMs can parse Nmap scan results and transform raw port/service/version data into prioritised attack plans, CVE hypothesis lists, Metasploit module recommendations, and report-ready findings in seconds. The combination of Nmap's scanning precision with GPT reasoning about discovered services is one of the most immediately useful AI workflows in offensive security -- bridging the gap between raw data collection and actionable intelligence.

Security professionals using AI-augmented Nmap workflows report 50-70% faster time-to-intelligence on large network scopes. Feeding Nmap XML to GPT eliminates the manual cross-referencing between open ports, service versions, and CVE databases that previously consumed hours of analyst time on large-scope engagements.

AI transforms Nmap output into professional penetration test findings -- converting open port and service version data into structured findings with CVSSv4.0 scores, business impact statements, and remediation recommendations. A list of Nmap-discovered vulnerable services becomes polished report sections ready for client delivery, reducing report writing time from hours to minutes.

Custom NSE Script Generation
