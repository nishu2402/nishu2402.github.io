Installation & Environment Setup

Splunk is the most widely deployed Security Information and Event Management (SIEM) platform, used by over 90 of the Fortune 100. It collects, indexes, and correlates machine data from any source -- network devices, endpoints, cloud infrastructure, applications, and security tools -- making it searchable and actionable in real time. The Splunk Enterprise Security (ES) premium app transforms Splunk into a full SIEM with pre-built detections, risk-based alerting, and incident management workflows.

Splunk operates on a distributed architecture: Forwarders collect and ship data from sources, Indexers store and index the data, and Search Heads provide the GUI and execute searches. For labs and smaller deployments, a single standalone instance performs all three roles. Splunk Cloud provides fully managed deployment.

01
Download and Install
Download from splunk.com/en_us/download/splunk-enterprise.html. Register for a free 60-day Enterprise trial or use the permanent Splunk Free licence (500 MB/day indexing). On Linux: install the .deb or .rpm package and start with sudo /opt/splunk/bin/splunk start --accept-license. On Windows: run the MSI installer -- Splunk starts automatically as a service.
Enable boot-start on Linux: sudo /opt/splunk/bin/splunk enable boot-start
02
Access the Splunk Web Interface
Navigate to http://localhost:8000 in your browser. Log in with the admin credentials you set during installation. The Splunk Home screen shows the Search & Reporting app, data ingestion status, and installed apps. The top navigation provides access to Search, Apps, Settings, and the Activity monitor.
03
Install Splunk Enterprise Security App
Navigate to Apps → Find More Apps and search for "Splunk Enterprise Security". Install the ES app -- this transforms Splunk into a full SIEM with pre-built MITRE ATT&CK dashboards, notable events, risk-based alerting, and incident review workflows. For lab use, install the free Splunk Security Essentials app which provides 250+ security use cases with guided SPL.
04
Configure Roles and Users
Go to Settings → Users and Authentication → Users. Create role-based accounts: SOC Analyst (search and alert access, no admin), Threat Hunter (full search, index access), SOC Manager (dashboards, reporting, no config). Splunk's RBAC controls which indexes, apps, and capabilities each user can access -- critical in production environments.
Use LDAP/SAML integration in production: Settings → Users and Authentication → Authentication Methods

Data Ingestion & Source Configuration

Data ingestion quality is the foundation of SIEM effectiveness -- Splunk can only detect what it can see. The ingestion architecture centres on Universal Forwarders (lightweight agents deployed on every endpoint and server), which send data to Splunk indexers. Splunk also ingests directly from APIs, syslog, S3 buckets, Kafka streams, and cloud service logs. Proper source typing and index assignment at ingestion time is critical -- these decisions determine how data is parsed, retained, and searched.

📜 Upload Files (Quick Start)
Navigate to Settings → Add Data → Upload. Drag and drop log files (CSV, JSON, syslog, Windows Event Logs). Splunk auto-detects the source type and shows a preview. Use for one-time ingestion of log samples, CTF challenge PCAPs, or evidence files during incident investigation. This is the fastest way to get data into Splunk for analysis.
🔗 Monitor Files and Directories
Navigate to Settings → Add Data → Monitor. Select a file or directory path. Splunk continuously monitors and ingests new log entries as they appear -- ideal for monitoring /var/log/ directories, application log folders, or Windows Event Log exports on the Splunk server itself.
📰 Universal Forwarder (Endpoints)
Install the lightweight Splunk Universal Forwarder on every endpoint, server, and network device you want to monitor. Download from splunk.com. Configure inputs.conf to specify what to monitor (event logs, file paths, network ports) and outputs.conf to point to your indexer. Forwarders use minimal CPU and memory.
🌐 Network / Syslog Input
Navigate to Settings → Add Data → Network. Configure Splunk to listen on a UDP or TCP port for syslog data. Point your firewalls, routers, switches, and Linux servers to send syslog to Splunk's IP on the configured port. Immediately starts receiving network device logs without installing forwarders.
☁️ Cloud Service Inputs
Install add-ons from Splunkbase: Splunk Add-on for AWS (CloudTrail, GuardDuty, VPC Flow Logs), Microsoft Azure Add-on (Azure AD, Sentinel alerts), Google Cloud Add-on. Each add-on provides pre-built source types, field extractions, and CIM-compliant data models for cloud telemetry without manual configuration.
📋 Source Types and Indexes
Every data source gets a sourcetype (tells Splunk how to parse it: WinEventLog:Security, syslog, access_combined) and an index (determines retention and RBAC). Organise indexes by data type: index=windows, index=network, index=endpoint. Proper index separation dramatically improves search performance and cost management.

Search Processing Language (SPL)

SPL (Search Processing Language) is Splunk's query language -- the single most important skill in the entire platform. Every Splunk capability -- dashboards, alerts, threat hunting, reports, and ML -- is powered by SPL searches. SPL uses a pipe-delimited command structure where data flows from left to right: search generates events, then each pipe command transforms them. Mastering SPL is what separates a Splunk user from a Splunk analyst.

Basic Search Syntax index=windows EventCode=4625 | stats count by src_ip, user | sort -count
Count failed Windows logon attempts (Event 4625) grouped by source IP and username, sorted by frequency descending. The pattern: index + filter selects events, | stats count by aggregates, | sort orders results. This single search surfaces brute force attempts across your entire Windows fleet.
Time Range and Field Filtering index=network earliest=-24h latest=now src_ip=192.168.1.0/24 dest_port=445 | table _time, src_ip, dest_ip, dest_port, action
Restrict search to the last 24 hours, filter to a subnet and a specific destination port (445 = SMB), then present results as a clean table. earliest and latest control time range. | table selects specific fields to display. Essential for investigating lateral movement via SMB.
Aggregation and Rare Commands index=windows EventCode=4688 | rare limit=20 CommandLine by host
Find the 20 rarest process command lines per host -- the rare command surfaces anomalies by showing what almost never happens. Rare processes on endpoints are high-signal indicators of malware execution, LOLBin abuse, or attacker tooling. Complement with top to understand normal vs abnormal baseline.
Transaction Command -- Session Reconstruction index=web_logs | transaction sessionid maxspan=30m | where eventcount > 50 | table sessionid, duration, eventcount, clientip
Group events by session ID into logical transactions with a 30-minute max span. Filter for sessions with more than 50 events. The transaction command reconstructs multi-event user sessions -- use to identify account takeover patterns, credential stuffing, and automated scraping behaviour from high-volume sessions.
Subsearch -- Chained Intelligence index=network [search index=threat_intel | fields src_ip | rename src_ip as dest_ip] | stats count by dest_ip, dest_port
Inner search (in brackets) queries a threat intelligence index for known-bad IPs, passes them as a filter to the outer search. This pattern cross-references live traffic against threat intel lookups -- the foundation of automated IOC-based detection in Splunk. The result shows connections to known-malicious IPs from your network.
SPL for DNS Tunnelling Detection index=dns | eval qname_len=len(query) | where qname_len > 50 | stats count, avg(qname_len) as avg_len by src_ip | where count > 100 | sort -count
Calculate DNS query name length, filter for unusually long names (DNS tunnelling encodes data in subdomains), then identify hosts making more than 100 such requests. DNS tunnelling produces queries far longer than normal (30+ characters vs typical 15-20). This search surfaces active DNS exfiltration attempts with statistical confidence.

Dashboards & Visualisations

Splunk dashboards transform SPL search results into persistent, real-time visual intelligence that SOC analysts can monitor without running manual searches. A well-designed security dashboard surfaces the most critical information at a glance -- alert counts by severity, authentication anomalies, top threat actors, network traffic baselines, and compliance status. Dashboards are built in the Dashboard Studio (modern, drag-and-drop) or Classic Dashboard Editor (XML-based, more customisable).

01
Create a New Dashboard
Navigate to Search & Reporting → Dashboards → Create New Dashboard. Give it a descriptive title (e.g. "SOC Overview -- Authentication Events"). Choose Dashboard Studio for modern layout with absolute and grid modes, or Classic for XML control. Set the dashboard as Shared in App so your whole SOC team can access it.
02
Add Search-Powered Panels
Click Add Panel and choose a visualisation type: Single Value (KPIs -- total alerts, critical event count), Column Chart (top 10 talkers, failed logins by host), Line Chart (event volume over time, traffic baseline), Map (geographic threat origin), or Table (detailed event list). Each panel runs its own SPL query on a configurable refresh interval.
03
Add Time Range Picker and Filters
In Edit → Add Input → Time, add a time range picker that controls all panels simultaneously. Add Dropdown or Text inputs as filter tokens (e.g. $host$, $index$) that analysts can use to drill into specific systems without editing SPL. Tokens make dashboards interactive investigation tools rather than static reports.
Set auto-refresh: Edit → Settings → Refresh → every 60 seconds for live SOC dashboards.
04
Essential SOC Dashboard Panels
Every SOC overview dashboard should include: Total Alerts (24h) single value KPI, Critical/High/Medium Alert Counts by severity, Authentication Failures Over Time line chart, Top Source IPs table, Network Traffic Volume by protocol, MITRE ATT&CK Technique Coverage heatmap (from ES app), and Open Incidents count. These seven panels give a full situational awareness picture at a glance.

Alerts & Correlation Rules

Splunk alerts automate detection -- they run SPL searches on a schedule and trigger actions when results meet defined conditions. A well-tuned alert library is the core of a functioning SIEM. Alert quality is defined by precision (low false positive rate) and recall (missing no true positives). Splunk Enterprise Security extends basic alerting with Correlation Searches and Risk-Based Alerting (RBA) -- accumulating risk scores across multiple weak signals before generating a single high-confidence notable event.

📢 Creating a Scheduled Alert
Run your detection SPL search. Click Save As → Alert. Set: Schedule (every 5 min, hourly, daily), Trigger Condition (Number of Results > 0, or custom condition), Throttling (suppress for N minutes after triggering to prevent alert storms). Throttling is critical -- a noisy alert with no throttle will flood your SOC and get ignored.
⚡ Alert Actions
Configure what happens when an alert fires: Send Email (to SOC distribution list with search results), Trigger a Webhook (call PagerDuty, Slack, Teams), Create a Notable Event (ES only -- appears in Incident Review), Run a Script (custom response automation), or Add to Risk (RBA -- contribute to a risk score without alerting yet).
🔎 Risk-Based Alerting (ES)
Instead of triggering on every weak signal, RBA accumulates risk scores on entities (user, host, IP). Low-confidence detections add 20-40 risk points; high-confidence add 80-100. When an entity's risk exceeds a threshold (e.g. 200 points), a single consolidated Risk Notable is created. This dramatically reduces alert fatigue while missing nothing -- the highest-value Splunk ES capability.
🔬 Incident Review Workflow (ES)
Navigate to Enterprise Security → Incident Review. All notable events appear here with status, owner, urgency, and MITRE ATT&CK tags. Analysts: click a notable, review supporting events, run investigation searches, update status (New → In Progress → Resolved), document findings, and close. This is the primary SOC analyst workflow -- the triage queue for all Splunk detections.
🎯 Correlation Search Examples
Key correlation searches every SOC should have: Brute Force (5+ failed logins from same IP in 5 min), Impossible Travel (same user authenticating from two geographically distant IPs within 1 hour), Process Injection (parent/child process anomalies), DNS Tunnelling (long query names, high volume), Data Staging (large file creation before network transfer).
📊 Alert Tuning and Suppression
Reduce false positives without losing detections: use | lookup whitelist.csv src_ip OUTPUT is_whitelisted | where NOT is_whitelisted=true to exclude known-good IPs. Use Alert Suppression to prevent duplicate alerts for the same entity within a window. Review all alerts weekly with false-positive rate tracking -- any alert firing more than 20 times per day without action needs tuning.

Threat Hunting with Splunk

Threat hunting is the proactive search for adversaries who have bypassed your automated detections. While SIEM alerts are reactive (trigger when known-bad patterns occur), threat hunting is hypothesis-driven -- starting with an assumption ("an attacker is staging data before exfiltration") and searching the data to prove or disprove it. Splunk's SPL, combined with the Splunk Security Essentials app's MITRE ATT&CK coverage heatmap, provides the ideal environment for structured threat hunting.

Hunting -- Lateral Movement via WMI (T1021.006) index=windows EventCode=4688 ParentImage="*wmiprvse.exe" | stats count by host, CommandLine | where count < 5 | sort -count
Find rare child processes spawned by the WMI provider host -- a common lateral movement technique. Attackers use WMI to execute commands remotely, which spawns processes under wmiprvse.exe. Rare command lines (count < 5) are highest priority for investigation. Maps to MITRE ATT&CK T1021.006.
Hunting -- Credential Dumping Indicators (T1003) index=windows EventCode=10 TargetImage="*lsass.exe" | stats count by SourceImage, host | where count > 1 | sort -count
Sysmon Event ID 10 logs process access to LSASS -- the memory location where Windows stores credentials. Any tool attempting to read LSASS memory (Mimikatz, ProcDump, Task Manager) generates these events. Any SourceImage other than known Windows binaries is a credential dumping indicator requiring immediate investigation.
Hunting -- Beaconing Detection (C2 Traffic) index=network dest_port!=80 dest_port!=443 | bucket _time span=1h | stats count by _time, src_ip, dest_ip | streamstats avg(count) as avg_count stdev(count) as std_count by src_ip, dest_ip | where abs(count - avg_count) < std_count * 0.5 | stats count as beacon_hours by src_ip, dest_ip | where beacon_hours > 6
Detect C2 beaconing -- connections that occur at highly regular intervals. This search finds src/dest pairs with consistent hourly connection counts (low standard deviation relative to mean) on non-standard ports over 6+ hours. Regular communication at fixed intervals is the hallmark of C2 beacon check-ins.

SOAR & Automation with Splunk SOAR

Splunk SOAR (Security Orchestration, Automation and Response) -- formerly Phantom -- connects Splunk detections to automated response playbooks that execute across your entire security tool stack. When Splunk ES creates a notable event, SOAR can automatically enrich it (VirusTotal lookup, passive DNS, GeoIP), contain the threat (isolate an endpoint via EDR, block an IP in the firewall), and notify the SOC -- all within seconds of detection, without human intervention. SOAR reduces mean time to respond (MTTR) from hours to seconds.

01
Understanding Playbooks
SOAR playbooks are visual, drag-and-drop automation workflows. Each playbook starts with a trigger (a notable event from Splunk ES, an alert, or a manual run), flows through a series of actions (API calls to security tools), includes decision blocks (if VT score > 50, proceed to containment), and ends with notifications or ticket creation. Navigate to SOAR → Playbooks → Create Playbook.
02
Phishing Response Playbook
Build a phishing triage playbook: Trigger: Email security alert with attachment. Step 1: Extract attachment hash → VirusTotal lookup. Decision: VT score > 5? Yes: Block sender domain in email gateway, quarantine similar emails, notify SOC. No: Add to watch list, close as benign. Always: Create JIRA ticket with all findings, update Splunk notable event status to resolved or escalated.
Pre-built playbook templates available in Splunk SOAR Community: splunkbase.splunk.com → search "SOAR playbook"
03
Endpoint Isolation Playbook
When Splunk detects ransomware indicators: Step 1: Get hostname from notable event. Step 2: CrowdStrike/Defender API → isolate endpoint from network. Step 3: Active Directory → disable compromised user account. Step 4: Create ServiceNow P1 incident ticket. Step 5: Send Slack alert to SOC channel with all context. Total automated response time: under 30 seconds from detection to containment.
04
App Connectors and Integrations
SOAR integrates with 350+ security products via app connectors. Essential connectors: CrowdStrike/SentinelOne (EDR -- isolate, get process list), Palo Alto/Cisco (firewall -- block IP, URL), VirusTotal/Hybrid Analysis (threat intel), Active Directory (disable account, reset password), JIRA/ServiceNow (ticketing), Slack/Teams (notification). Install from SOAR → Apps → Browse Apps.

AI Integration in Splunk Operations

AI transforms Splunk from a powerful data platform into an intelligent security operations brain. LLMs excel at the tasks that are most expensive in a SOC: translating security questions into SPL queries, explaining alert context in plain English, correlating disparate events into coherent attack narratives, and generating incident reports. Splunk itself is investing heavily in AI -- the Splunk AI Assistant provides native conversational SPL generation, while external AI pipelines via the Splunk API unlock custom intelligence workflows that no vendor product can replicate.

Junior analysts with AI assistance produce investigation quality comparable to senior analysts within weeks, dramatically scaling SOC capacity without proportional headcount increases.

✏️
SPL Query Generation
LLMs translate natural-language security questions into precise SPL -- "find all users who logged in after hours from a new device in the last 7 days" becomes a complete SPL query with the correct index, field names, time range, and statistical commands. Eliminates the SPL learning curve for new analysts, dramatically accelerates experienced analysts on complex hunts, and enables non-technical stakeholders to query Splunk directly.
🔎
Alert Triage Assistance
AI analyses notable event context -- the triggering SPL, the raw events, the affected entity's recent history -- and produces a natural-language triage assessment: what happened, why it triggered, what the likely explanation is (true positive vs false positive), and what investigation steps to take next. Reduces average triage time from 15 minutes to 2 minutes per alert and dramatically improves junior analyst decision quality.
📄
Incident Report Generation
AI converts Splunk incident timeline data, notable events, and investigation notes into professional incident reports -- executive summary, technical timeline, affected systems, attacker TTPs mapped to MITRE ATT&CK, indicators of compromise list, and remediation recommendations. Transforms the most time-consuming post-incident task from a 4-hour writing exercise into a 10-minute review and approval process.
👻
Threat Hunt Hypothesis Generation
AI generates structured threat hunt hypotheses from current threat intelligence reports and CVE disclosures -- "Based on this APT campaign report, what SPL searches would detect the described TTPs in a Splunk environment with Windows event logs and network flow data?" Produces a prioritised list of hunt hypotheses with the exact SPL to execute, mapped to MITRE ATT&CK, aligned to your data sources.
🔴
SOAR Playbook Generation
LLMs generate SOAR playbook logic and Python code for custom actions from plain-English requirement descriptions -- "write a Splunk SOAR playbook that triages phishing emails by extracting URLs, checking them against VirusTotal, and quarantining the email if any URL scores above 5." Produces fully functional Python action code ready to paste into the SOAR custom function editor.
📉
Detection Engineering
AI converts threat intelligence articles, CVE descriptions, and penetration test findings directly into Splunk correlation search SPL -- creating new detections without manual SIEM engineering. "This APT uses Cobalt Strike with these specific beacon parameters" becomes a working SPL detection search in minutes. Dramatically accelerates detection engineering velocity and reduces the time from threat disclosure to defensive coverage.
