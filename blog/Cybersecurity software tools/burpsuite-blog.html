Installation & Environment Setup

Burp Suite is the industry-standard web application security testing platform developed by PortSwigger. It operates as an intercepting HTTP/HTTPS proxy sitting between your browser and the target web application, allowing you to inspect, modify, and replay every HTTP request and response in real time. The Community Edition is free and covers most penetration testing needs; Burp Suite Professional adds the automated scanner, advanced Intruder attack types, Collaborator, and BApp store access.

Burp Suite ships pre-installed in Kali Linux and Parrot OS. For other platforms, download the installer from portswigger.net. Java 17+ is required. The most critical setup step is installing the Burp CA certificate into your browser -- without this, Burp cannot intercept HTTPS traffic, which comprises the majority of modern web application traffic.

01
Download & Install Burp Suite
Download from portswigger.net/burp/releases. Choose your platform: Windows (.exe installer), Linux (.sh or .deb), macOS (.dmg). Java 17+ is bundled -- no separate Java install needed. On Kali Linux, Burp Suite Community is pre-installed and can be updated via sudo apt install burpsuite.
02
Launch and Create a Project
Open Burp Suite. Community Edition opens a temporary project automatically. Professional Edition prompts to create a named project (saves all scan data, history, and settings persistently). For CE users, export important findings before closing -- temporary projects do not persist.
Pro: Always create named projects per engagement. Community: work is lost on close.
03
Configure Browser Proxy
Burp listens on 127.0.0.1:8080 by default. Configure your browser: Firefox → Settings → Network Settings → Manual Proxy → HTTP: 127.0.0.1, Port: 8080. Check "Also use for HTTPS". Install FoxyProxy extension for one-click toggle between Burp proxy and direct connection -- essential for efficient testing.
04
Install the Burp CA Certificate (CRITICAL)
With proxy active and Burp running, browse to http://burpsuite or http://burp in your browser. Click "CA Certificate" to download cacert.der. In Firefox: Settings → Privacy & Security → View Certificates → Authorities → Import → select cacert.der → check "Trust this CA to identify websites". Without this step, Burp cannot decrypt HTTPS traffic.
HTTPS is mandatory. Without the CA cert installed, modern web apps will show cert errors and Burp cannot intercept their traffic.
05
Verify Interception is Working
In Burp, go to Proxy → Intercept and turn Intercept ON. Browse to any HTTPS site in your browser. Burp should capture and display the request, pausing it until you click Forward. If this works, your proxy is correctly configured. Turn Intercept OFF for general browsing.

Proxy & Traffic Interception

The Burp Proxy is the core of all Burp Suite workflows. Every HTTP/HTTPS request from your browser passes through the Proxy, where Burp captures it for analysis, modification, or forwarding. Understanding the Proxy tab -- its intercept controls, HTTP history, WebSocket history, and options -- is the foundation that makes every other Burp tool useful. The Proxy HTTP history is your complete record of every request made during a testing session, searchable and filterable by host, method, status code, content type, and response size.

▶ Intercept Tab Controls
Intercept ON -- every request pauses for your inspection. You see the full request, can edit any header, parameter or body value, then Forward it. Intercept OFF -- traffic flows through silently, logged to HTTP History only. Use ON only when you have a specific request to inspect.
📋 HTTP History Tab
Complete log of every proxied request. Click any row to see the full request and response. Use the Filter bar (top of history) to show only in-scope items, filter by status code (2xx/3xx/4xx/5xx), MIME type (JSON/HTML/Script), or search by keyword. Right-click any request to send to Repeater, Intruder, or Scanner.
🔄 Match and Replace
Found in Proxy → Options → Match and Replace. Automatically modifies all requests or responses matching your rules -- add custom headers to every request, strip security headers from responses, replace User-Agent strings, or inject values into cookies. Rules run silently without needing to intercept each request individually.
👁 WebSocket History
Burp also intercepts WebSocket traffic -- increasingly common in modern web apps for real-time features. WebSocket history is separate from HTTP history. You can intercept, modify and replay WebSocket messages from this tab, which is essential for testing real-time auction sites, chat apps, trading platforms, and live dashboards.
🎯 Intercept Rules
Configure which requests Burp intercepts under Proxy → Options → Intercept Client Requests. Default rule intercepts everything. Add rules to intercept only in-scope items, only specific file extensions, or only requests matching URL patterns -- prevents wasting time on CDN, analytics and font requests.
⚙️ TLS Pass-through
Some hosts (banking apps, certificate pinning) reject Burp's CA. Add them to Proxy → Options → TLS Pass Through. Burp tunnels these connections without intercepting. Keeps the browser happy while you continue intercepting your actual target.

Only turn intercept ON when you have identified a specific request you want to inspect or modify. This prevents the constant click-to-forward interruption that slows down reconnaissance and application mapping phases.

Target Sitemap & Scope Control

The Target tab is Burp's intelligence centre -- it builds a visual sitemap of every URL, endpoint, parameter, and directory discovered during your session. As you browse the application with the proxy running, Burp automatically populates the sitemap with every host and path encountered. The Scope control is critical -- defining scope tells Burp which hosts and URLs are in-scope for testing, filtering all History, Scanner, and Intruder results to the authorised target. Running Burp without defined scope against anything other than your target is a risk no professional penetration tester should take.

01
Define Your Scope
Navigate to Target → Scope → Add. Enter your target: Protocol (HTTPS), Host (target.com or regex .*\.target\.com), Port (443), File (.*). Add exclusions for paths you should not test -- /logout, /admin/delete, /account/close. Scope definition is not optional -- it protects you legally and keeps results clean.
Always define scope before starting. Running Burp Scanner or Intruder outside authorised scope creates legal liability.
02
Filter History to Scope Only
In Proxy → HTTP History, click the Filter bar at the top and check "Show only in-scope items". This hides third-party CDN requests, Google Analytics, font CDNs, and ad trackers from your history -- leaving only traffic to/from your target. Dramatically reduces noise in large applications.
03
Browse and Build the Sitemap
Navigate to Target → Site map. With Intercept OFF, systematically browse the entire application: all navigation links, login/register/reset flows, all form submissions, file upload endpoints, API calls (visible as JSON in History), and admin panels if in scope. Burp builds the sitemap automatically from your browsing -- click any host or path to see what Burp knows about it.
04
Use Engagement Tools
Right-click any host or folder in the Sitemap to access Engagement tools: Find comments (HTML comments with developer notes), Find scripts (all JavaScript files for analysis), Search (keyword search across all captured content), and Analyze target (statistical summary of the site structure). These surface hidden intelligence about the application without additional requests.

Repeater -- Manual Request Testing

Repeater is the workhorse of manual web application penetration testing. It allows you to take any intercepted request, modify it, send it to the server, and immediately inspect the response -- repeating this cycle as many times as needed. Repeater is where you manually confirm vulnerabilities, test payload variations, exploit injection points, bypass authentication, and understand exactly how an application responds to edge cases. Every serious web application finding starts with Repeater exploration before being confirmed and documented.

01
Send a Request to Repeater
Right-click any request in Proxy History, Intercept, or Sitemap and select "Send to Repeater" (or press Ctrl+R). The Repeater tab gets a new sub-tab with the request loaded and ready to edit. Create separate named tabs for each vulnerability type you are testing -- right-click the tab to rename it (e.g. "SQLi /search", "IDOR /api/users").
02
SQL Injection Testing in Repeater
Find a parameter like GET /products?id=5. Send to Repeater. Modify the value: try id=5' -- look for SQL error in response. Then try id=5 OR 1=1-- -- should return all rows if injectable. Iterate quickly: click Send, read response, adjust payload, Send again. Repeater lets you iterate dozens of variations in minutes.
Tip: Use the response "Render" tab to see how the page looks -- helps spot reflected values and changes in page content visually.
03
Authentication Bypass Testing
Modify session cookies directly in the Request editor. Change role=user to role=admin. Add headers like X-Forwarded-For: 127.0.0.1 or X-Original-URL: /admin. Test parameter tampering: change price=99.99 to price=0.01. Each Send gives instant feedback -- no browser needed.
04
IDOR and Horizontal Privilege Escalation
Find user-specific endpoints like GET /api/users/1337/profile (logged in as user 1338). In Repeater, change 1337 to 1338, then to 1, then to 100. If you receive another user's data with a 200 OK, that is a confirmed IDOR. Repeater makes this trivially fast -- test dozens of IDs in minutes with the request history arrows to navigate back.
Repeater keeps a complete history of all requests sent in that tab -- use the left/right arrows to navigate previous requests and compare responses.

Intruder -- Automated Attack Payloads

Burp Intruder is the automated attack tool -- it allows you to mark positions in a request and fire a list of payloads at those positions, analysing responses for differences that indicate successful attacks. It is the primary tool for brute force, credential stuffing, fuzzing, enumeration, and parameter analysis. Community Edition rate-throttles Intruder to make it slow; Professional Edition runs at full speed. For most brute force tasks, ffuf or Hydra are faster alternatives if you are on Community.

[ ! ]

Authorised Engagements Only. Intruder generates high volumes of automated requests that will appear clearly in application logs and may trigger account lockouts or WAF blocks. Always confirm the authorised scope and lockout policy before running Intruder attacks. Use rate limiting in Options to avoid locking out legitimate users.

▶ Sniper  --  One Position, One Payload Set
Use when: Testing a single parameter with a full wordlist
The most common attack type. Mark one position (the value to fuzz), load a payload list. Intruder cycles through every payload in that position. Example: test a search field with a SQLi wordlist, or test a username field with a list of common usernames. In Burp: right-click the request value → "Add" to mark it, then configure payloads in the Payloads tab.
▶ Battering Ram  --  Same Payload in ALL Positions
Use when: Testing username = password (credential pairs like admin/admin)
Sends the same payload value to every marked position simultaneously. Mark both username and password fields. Load a wordlist. Intruder tries admin/admin, root/root, user/user. Useful for default credential testing where the username and password are often identical.
▶ Pitchfork  --  Parallel Payload Sets (one per position)
Use when: Credential spraying with username:password CSV pairs
Uses separate, synchronised payload lists for each position. Position 1 (username) gets payload list 1; Position 2 (password) gets payload list 2. Row 1 of list 1 pairs with row 1 of list 2. Perfect for credential stuffing from a breach database where you have matched username:password pairs.
▶ Cluster Bomb  --  All Combinations (Cartesian Product)
Use when: Unknown username + password brute force (small lists only)
Tries every combination of every payload across all positions. 100 usernames × 100 passwords = 10,000 requests. Warning: generates very large request volumes. Always confirm account lockout policy and rate limits before using this against a live system. Community Edition rate-throttles this significantly.
Intruder Setup Workflow in the GUI
01
Send to Intruder
Right-click any request in History or Intercept → Send to Intruder (Ctrl+I). Go to Intruder → Positions tab.
02
Mark Positions
Click [Clear] to remove Burp's auto-detected positions. Highlight the parameter value you want to fuzz, then click [Add]. Green markers appear: username=§admin§.
03
Configure Payloads
Go to Payloads tab. Payload type: Simple list (paste inline) or Runtime file (load wordlist from disk). Common lists: /usr/share/wordlists/rockyou.txt for passwords, SecLists for usernames.
04
Set Grep Match to Spot Success
In Options → Grep - Match, add the error string that appears on FAILURE (e.g. "Invalid credentials"). Sort results by the Grep column -- unchecked rows (where error string is absent) = SUCCESS.

Scanner & Vulnerability Detection

The Burp Scanner (Professional only) is one of the most capable automated web vulnerability scanners available. It combines passive analysis of traffic flowing through the proxy with active probing of endpoints for OWASP Top 10 vulnerabilities -- SQL injection, XSS, SSRF, XXE, command injection, path traversal, authentication issues, and many more. Unlike standalone scanners, Burp Scanner operates in the context of an authenticated session, finding vulnerabilities that unauthenticated scanners miss entirely.

01
Passive Scanning (Always On)
Burp passively analyses all traffic flowing through the proxy automatically -- no configuration needed. Check Dashboard → Issue activity for passive findings as they appear. Passive scanning detects: insecure cookie flags (missing HttpOnly/Secure), information disclosure in headers, mixed content, and outdated software versions -- all without sending a single additional request to the server.
02
Active Scan from HTTP History (Pro)
In Proxy → HTTP History, select the requests you want to scan (Ctrl+click to multi-select). Right-click → "Scan". The scan configuration dialog opens: choose Audit only (no additional crawling) or Crawl and audit. Set scan speed: Fast (fewer checks, less noise) / Balanced / Thorough (comprehensive, slower). Click OK -- results appear in the Dashboard as they are found.
For authenticated scanning, run scans from History captured while logged in. Burp reuses the session cookies from your intercepted requests.
03
New Scan from Dashboard (Pro)
Go to Dashboard → New Scan. Enter target URL, add login credentials (Burp will log in automatically before scanning), define scope, and configure scan type. Use this for comprehensive automated assessments where you want Burp to crawl and scan simultaneously. The Dashboard shows live progress, finding count by severity, and crawl coverage.
04
Reading and Verifying Findings
In Dashboard → Issue activity, click any finding to see the full detail panel: the specific request Burp sent, the response that confirmed the vulnerability, and exactly what evidence triggered the finding. Always verify findings in Repeater before reporting -- check that the vulnerable request can be manually reproduced. This eliminates false positives and gives you the evidence needed for the report.
Confidence levels: Certain (confirmed exploit) / Firm (strong indicators) / Tentative (possible, needs verification). Only report Certain and Firm without additional verification.

Extensions & BApp Store

The BApp Store (Burp App Store) provides hundreds of community and official extensions that extend Burp's capabilities. Extensions are written in Java, Python (via Jython), or Ruby (via JRuby) and integrate directly into the Burp UI and request processing pipeline. Essential extensions like Logger++, Autorize, JWT Editor, and Turbo Intruder are used on virtually every professional web application assessment.

Installing Extensions

Navigate to Extensions → BApp Store. Search for the extension by name. Click Install. Some Python-based extensions require Jython: go to Extensions → Options → Python environment and set the path to jython-standalone.jar (download from jython.org). Installed extensions appear in the Extensions tab and integrate directly into the Burp UI.

📊 Logger++
BApp Store → Logger++
Advanced HTTP logging with regex-based filtering, colour highlighting rules, live grep search, and CSV export. Far more powerful than the built-in HTTP History -- use it as your primary request log for complex assessments. Filter by any request/response field in real time.
🔒 Autorize
BApp Store → Autorize
Automatically tests every request with a low-privileged cookie and with no cookie. Marks responses in three colours: green (access denied correctly), orange (possible IDOR), red (access granted -- confirmed vulnerability). Essential for broken access control and horizontal privilege escalation testing at scale.
🔑 JWT Editor
BApp Store → JWT Editor
Adds a JWT tab to the request editor. Visually decodes JWT header, payload, and signature. Test algorithm confusion attacks (RS256 to HS256), "none" algorithm bypass, and weak secret cracking. Edit the payload and re-sign with a custom key. Indispensable for any app using JWTs for authentication.
🚀 Turbo Intruder
BApp Store → Turbo Intruder
Python-scripted high-speed request engine that bypasses the Community Edition rate limit. Critical for race condition testing -- sends requests simultaneously to exploit TOCTOU vulnerabilities (coupon reuse, balance manipulation, vote stuffing). Configure via Python script for complex multi-step attack logic.
🔧 Param Miner
BApp Store → Param Miner
Brute forces hidden GET and POST parameters that are not linked anywhere in the application. Right-click any request → Extensions → Param Miner → Guess params. Discovers unlinked parameters that affect application behaviour -- price manipulation, debug modes, feature flags, and internal APIs.
🔌 Hackvertor
BApp Store → Hackvertor
Encoding/decoding chains directly in the request editor using custom tags. Wrap payload values in tags like <@base64>payload<@/base64> and Burp automatically encodes the value on each Send. Chain multiple encodings. Invaluable for WAF bypass testing with layered encoding schemes.

AI Integration in Burp Suite Workflows

AI transforms Burp Suite from a manual testing tool into an intelligent vulnerability discovery platform. LLMs excel at analysing HTTP request/response pairs to identify subtle vulnerability indicators, generating payload lists tailored to specific application contexts, explaining complex response differences that indicate successful exploitation, and converting Burp findings into professional report sections. The combination of Burp's precise traffic capture with GPT-4 reasoning about application behaviour produces a testing capability that exceeds what either delivers independently.

[ AI ]

Web application penetration testers using AI-augmented Burp workflows report finding 30-40% more vulnerabilities on complex applications compared to manual testing alone. AI identifies subtle logic flaws, parameter pollution issues, and business logic vulnerabilities that traditional payload-based scanning misses -- and writes the report findings while you continue testing.

🔍
Request/Response Analysis
AI analyses HTTP request/response pairs from Burp History to identify vulnerability indicators invisible to automated scanners -- subtle differences in response timing, content length, error messages, and server behaviour that indicate injection points, logic flaws, or access control weaknesses. Paste a request/response pair and receive a structured vulnerability assessment with specific payload recommendations to confirm each finding in Repeater.
✏️
Payload Generation
LLMs generate context-specific Intruder payload lists from application analysis. "This login endpoint uses PHP and MySQL, what SQL injection payloads should I try in Intruder?" produces a complete, ordered payload list specific to the technology stack, WAF evasion requirements, and injection context. Dramatically better than generic wordlists for targeted manual testing against specific application types.
📄
Finding Documentation
AI converts Burp Scanner findings and Repeater-confirmed vulnerabilities into professional penetration test report sections -- complete with title, CVSSv4.0 score, technical description, business impact, HTTP request/response evidence, step-by-step reproduction instructions, and remediation guidance. Turns raw Burp output into client-ready findings in under 60 seconds per vulnerability.
🔐
JWT & Crypto Analysis
AI analyses JWT tokens, session cookies, and cryptographic implementations found in Burp traffic. Decode a JWT and ask "what attacks should I try on this token?" -- receive analysis of the algorithm, recommended attacks (none algorithm, RS256-to-HS256 confusion, weak secret brute force), and the exact JWT Editor workflow to test each. Also analyses custom crypto implementations for common flaws.
🛠️
Custom Extension Generation
LLMs generate custom Burp extensions in Python (Jython) or Java for specific testing needs -- "write a Burp extension that automatically adds a UUID to every request header and logs the response status to a file." Enables security professionals to create bespoke automation for specific application quirks or recurring testing patterns without deep Java/Burp API expertise.
🔴
Business Logic Analysis
AI identifies business logic vulnerabilities from Burp sitemap data and traffic patterns -- price manipulation opportunities, workflow bypass conditions, race condition candidates, and trust boundary violations that automated scanners cannot detect. Analyses the application's intended workflow and identifies deviations an attacker could exploit -- the vulnerability class most commonly missed by tool-only testing methodologies.
