The Bash Environment

Bash (Bourne Again Shell) is the default command-line interpreter on virtually every Linux distribution, macOS (pre-Catalina), and the Windows Subsystem for Linux. It is not just a terminal — it is a fully featured programming environment. For cybersecurity professionals, Bash is the primary automation layer: scripts that would take a human analyst hours to execute manually run in seconds, consistently, without error. Understanding the Bash environment — how variables work, how processes execute, how the shell resolves commands — is prerequisite knowledge for both writing effective scripts and understanding how attackers exploit the shell layer.

Bash Scripting Fundamentals

A Bash script is a plain text file containing a sequence of shell commands. The first line — the shebang (#!/bin/bash) — tells the OS which interpreter to use. Everything after that executes top-to-bottom, with variables, conditionals, loops, and functions providing the control flow needed to automate complex tasks. For security work, mastering control flow is what separates a script that runs one command from one that scans an entire network, analyses thousands of log lines, and sends an alert — all without human intervention.

#!/bin/bash # ───────────────────────────────────────────────────── # VARIABLES # ───────────────────────────────────────────────────── TARGET="192.168.1.1" # No spaces around = PORT_RANGE="1-1024" TIMESTAMP=$(date +%Y%m%d_%H%M%S) # Command substitution LOG_FILE="/tmp/scan_${TIMESTAMP}.log" echo "Target: $TARGET | Log: $LOG_FILE" # ───────────────────────────────────────────────────── # CONDITIONALS — if / elif / else # ───────────────────────────────────────────────────── if [ $EUID -ne 0 ]; then echo "[!] This script requires root. Run with sudo." exit 1 fi # String comparison: = for equal, != for not equal if [ -z "$1" ]; then echo "Usage: $0 <target-ip>"; exit 1 fi # ───────────────────────────────────────────────────── # LOOPS — for / while # ───────────────────────────────────────────────────── # Loop over a list of IPs for ip in 192.168.1.{1..10}; do if ping -c 1 -W 1 $ip &>/dev/null; then echo "[+] HOST UP: $ip" fi done # Read file line by line while IFS= read -r line; do echo "Processing: $line" done < hosts.txt # ───────────────────────────────────────────────────── # FUNCTIONS — reusable blocks # ───────────────────────────────────────────────────── log() { local level="$1"; shift echo "[$(date +%H:%M:%S)] [$level] $*" | tee -a $LOG_FILE } check_tool() { if ! command -v "$1" &>/dev/null; then log "ERROR" "Required tool '$1' not found. Install it first." exit 1 fi } # Call functions check_tool nmap log "INFO" "Script started — target: $TARGET"

Reconnaissance & Network Scanning Scripts

Reconnaissance is where Bash scripting delivers the most immediate value in penetration testing. Wrapping tools like Nmap, Gobuster, theHarvester, and Nikto in Bash scripts automates the systematic coverage of a target's attack surface — running multiple tools in sequence, saving structured output, and flagging high-priority findings for manual investigation. The scripts below represent the core recon automation workflow used in professional assessments.

#!/bin/bash # AUTOMATED RECONNAISSANCE SCRIPT # Usage: ./recon.sh <target-ip-or-domain> # Authorised use only TARGET="${1:?Usage: $0 <target>}" OUT_DIR="./recon_${TARGET//\//_}_$(date +%Y%m%d)" mkdir -p "$OUT_DIR"/{nmap,web,osint} banner() { echo ""; echo "═══════════════════════════════"; echo " $1"; echo "═══════════════════════════════"; } banner "PHASE 1 — NMAP DISCOVERY" # Fast ping sweep first nmap -sn -T4 $TARGET -oN "$OUT_DIR/nmap/ping_sweep.txt" # Full port scan + service/version detection nmap -sV -sC -T4 --open -oA "$OUT_DIR/nmap/full_scan" $TARGET echo "[+] Nmap complete → $OUT_DIR/nmap/" banner "PHASE 2 — WEB FINGERPRINTING" if grep -q "80/open\|443/open\|8080/open" "$OUT_DIR/nmap/full_scan.nmap"; then echo "[+] Web service detected — running Nikto & Gobuster" nikto -h https://$TARGET -nointeractive -o "$OUT_DIR/web/nikto.txt" 2>/dev/null & gobuster dir -u https://$TARGET \ -w /usr/share/wordlists/dirb/common.txt \ -x php,html,txt,js -q -o "$OUT_DIR/web/gobuster.txt" 2>/dev/null & wait echo "[+] Web scan complete → $OUT_DIR/web/" else echo "[-] No web services detected — skipping web scan" fi banner "PHASE 3 — SUBDOMAIN ENUMERATION" if [[ "$TARGET" =~ [a-zA-Z] ]]; then amass enum -passive -d $TARGET -o "$OUT_DIR/osint/subdomains.txt" 2>/dev/null echo "[+] Found $(wc -l < "$OUT_DIR/osint/subdomains.txt") subdomains" fi echo "" echo "[✓] Recon complete. All results in: $OUT_DIR"
#!/bin/bash # INTERNAL NETWORK SWEEP # Pings /24 subnet, then checks common ports on live hosts SUBNET="${1:-192.168.1}" PORTS="22 80 443 445 3389 8080 8443" LIVE_HOSTS=() echo "[*] Sweeping subnet: ${SUBNET}.0/24" for i in {1..254}; do ip="${SUBNET}.${i}" if ping -c 1 -W 1 "$ip" &>/dev/null; then LIVE_HOSTS+=("$ip") echo "[+] $ip is UP" fi done echo "" echo "[*] Checking common ports on live hosts..." for host in "${LIVE_HOSTS[@]}"; do open=() for port in $PORTS; do if timeout 1 bash -c "echo >/dev/tcp/$host/$port" 2>/dev/null; then open+=("$port") fi done if [ ${#open[@]} -gt 0 ]; then echo " $host → Open: ${open[*]}" fi done

Log Analysis & Threat Detection Scripts

Log analysis is where Bash's text-processing power — grep, awk, sed, sort, uniq — combines into a forensics toolkit. Linux systems log authentication events, system calls, cron activity, and kernel events to structured log files under /var/log/. Bash scripts that parse, correlate, and alert on these logs provide a lightweight SIEM capability that runs on any Linux system without additional tooling.

#!/bin/bash # AUTHENTICATION LOG ANALYSER # Detects brute force attacks, successful logins from new IPs, sudo abuse AUTH_LOG="/var/log/auth.log" SYSLOG="/var/log/syslog" THRESHOLD=10 # Alert if IP has >10 failures banner() { printf '\n\e[1;32m[%s]\e[0m %s\n' "$(date +%H:%M:%S)" "$1"; } # ─── SSH BRUTE FORCE DETECTION ─── banner "Failed SSH login attempts (top offenders)" grep "Failed password" $AUTH_LOG 2>/dev/null | \ awk '{print $NF}' | # Extract IP address (last field) sort | uniq -c | sort -rn | \ awk -v thresh="$THRESHOLD" '$1 >= thresh {printf " [!] %s failed attempts from IP: %s\n", $1, $2}' # ─── SUCCESSFUL LOGINS (all) ─── banner "Successful SSH logins (last 24 hours)" grep "Accepted password\|Accepted publickey" $AUTH_LOG 2>/dev/null | \ awk '{print $1, $2, $3, "User:", $9, "From:", $11}' | tail -20 # ─── SUDO USAGE LOG ─── banner "Recent sudo commands" grep "sudo:" $AUTH_LOG 2>/dev/null | grep "COMMAND" | \ awk '{print $1, $2, $3, $NF}' | tail -15 # ─── NEW USER ACCOUNTS CREATED ─── banner "User account changes" grep "useradd\|usermod\|passwd" $AUTH_LOG 2>/dev/null | tail -10 # ─── CRONTAB CHANGES (persistence indicator) ─── banner "Recent crontab modifications" find /var/spool/cron /etc/cron* -newer /tmp/.ref_time 2>/dev/null | \ xargs -I{} ls -la {} 2>/dev/null # ─── ACTIVE NETWORK CONNECTIONS (suspicious outbound) ─── banner "Established network connections" ss -tulnp | grep LISTEN ss -tp | grep ESTAB | grep -v "127.0.0.1\|::1"
#!/bin/bash # REAL-TIME THREAT MONITOR # Watches auth.log and alerts on new brute-force attempts # Run with: nohup ./monitor.sh & to run in background LOG="/var/log/auth.log" ALERT_THRESHOLD=5 WINDOW=60 # seconds declare -A ATTEMPTS echo "[*] Monitoring $LOG for brute-force attacks (Ctrl+C to stop)" tail -f $LOG | while read line; do if echo "$line" | grep -q "Failed password"; then IP=$(echo "$line" | grep -oP 'from \K[0-9.]+') USER=$(echo "$line" | grep -oP 'for \K\S+') ATTEMPTS[$IP]=$(( ${ATTEMPTS[$IP]:-0} + 1 )) COUNT=${ATTEMPTS[$IP]} if [ "$COUNT" -ge "$ALERT_THRESHOLD" ]; then printf '\e[1;31m[ALERT %s] BRUTE FORCE: %s → %d attempts, targeting user: %s\e[0m\n' \ "$(date +%H:%M:%S)" "$IP" "$COUNT" "$USER" # Optional: auto-block with UFW # sudo ufw insert 1 deny from $IP to any comment "auto-blocked brute force" fi fi done

Privilege Escalation Audit Scripts

Privilege escalation audit scripts identify misconfigurations on a Linux system that could allow a low-privileged user to gain root access. This is used offensively during penetration tests (to find escalation paths) and defensively during security hardening audits (to find and remediate them before an attacker does). The checks below represent the most commonly exploited Linux privilege escalation vectors — every sysadmin should run these against their own systems periodically.

[ ⚠ ]

Authorised Use Only. Run privilege escalation audit scripts exclusively on systems you own or have explicit written authorisation to test. These scripts identify security weaknesses — in the wrong hands on an unauthorised system, they constitute computer misuse. Always test in your own lab environment first.

#!/bin/bash # LINUX PRIVILEGE ESCALATION AUDIT SCRIPT # Checks for common misconfiguration vectors — authorised use only check() { printf '\n\e[1;36m[CHECK]\e[0m %s\n' "$1"; } found() { printf '\e[1;31m [VULN]\e[0m %s\n' "$1"; } ok() { printf '\e[1;32m [OK]\e[0m %s\n' "$1"; } check "Current user and groups" id; groups check "SUID binaries (can execute as owner/root)" find / -perm -4000 -type f 2>/dev/null | while read f; do owner=$(stat -c '%U' "$f") if [ "$owner" = "root" ]; then found "SUID root binary: $f"; fi done check "SGID binaries" find / -perm -2000 -type f 2>/dev/null | head -10 check "World-writable directories (potential script injection paths)" find / -type d -perm -0002 -not -path "/proc/*" 2>/dev/null | head -10 check "Files writable by current user (excluding home)" find /etc /usr /opt -writable -not -path "$HOME/*" 2>/dev/null | head -10 check "Sudo privileges (check for NOPASSWD)" sudo -l 2>/dev/null | grep -i "nopasswd\|ALL" && found "Privileged sudo rules found" check "Cron jobs running as root" cat /etc/cron* /etc/crontab /var/spool/cron/crontabs/root 2>/dev/null find /etc/cron.* -type f 2>/dev/null | xargs ls -la 2>/dev/null check "Processes running as root (non-kernel)" ps aux | awk '$1=="root" && $11 !~ /^\[/ {print $1, $11}' | sort -u | head -15 check "Sensitive files readable by current user" for f in /etc/shadow /etc/sudoers /root/.bash_history /root/.ssh/id_rsa; do if [ -r "$f" ]; then found "READABLE: $f"; else ok "Protected: $f"; fi done check "PATH variable — check for dangerous entries" echo $PATH | tr ':' '\n' | while read p; do if [ -w "$p" ] 2>/dev/null; then found "Writable PATH dir: $p — PATH hijacking possible" fi done

System Hardening Scripts

System hardening scripts automate the application of security baselines — configuring firewalls, disabling unnecessary services, enforcing SSH security settings, setting audit rules, and establishing file permission policies. Running a hardening script on a fresh system deployment is orders of magnitude faster and more consistent than manual configuration. Combined with a configuration management tool, hardening scripts ensure every deployed system starts from an identical, documented security baseline.

#!/bin/bash # LINUX SYSTEM HARDENING SCRIPT # Tests and applies security controls — run on fresh deployments # Requires root [ $EUID -ne 0 ] && { echo "Run as root"; exit 1; } apply() { printf '\e[1;32m[APPLY]\e[0m %s\n' "$1"; } # ─── SSH HARDENING ─── apply "Hardening SSH configuration" SSH_CONF="/etc/ssh/sshd_config" cp $SSH_CONF "${SSH_CONF}.backup.$(date +%Y%m%d)" sed -i 's/#PermitRootLogin yes/PermitRootLogin no/' $SSH_CONF sed -i 's/PermitRootLogin yes/PermitRootLogin no/' $SSH_CONF sed -i 's/#PasswordAuthentication yes/PasswordAuthentication no/' $SSH_CONF sed -i 's/X11Forwarding yes/X11Forwarding no/' $SSH_CONF # Append secure settings if not already present for setting in \ "MaxAuthTries 3" \ "LoginGraceTime 20" \ "AllowTcpForwarding no" \ "ClientAliveInterval 300" \ "ClientAliveCountMax 2"; do key=$(echo "$setting" | cut -d' ' -f1) if ! grep -q "^$key" $SSH_CONF; then echo "$setting" >> $SSH_CONF fi done systemctl restart sshd # ─── FIREWALL (UFW) ─── apply "Configuring UFW firewall" ufw default deny incoming ufw default allow outgoing ufw allow 22/tcp comment "SSH" ufw deny 23/tcp comment "Block Telnet" ufw deny 21/tcp comment "Block FTP cleartext" ufw --force enable ufw status verbose # ─── DISABLE UNNECESSARY SERVICES ─── apply "Disabling unnecessary services" for svc in bluetooth cups avahi-daemon rpcbind; do if systemctl is-active --quiet "$svc" 2>/dev/null; then systemctl stop "$svc" && systemctl disable "$svc" echo " Disabled: $svc" fi done # ─── FILE PERMISSION HARDENING ─── apply "Hardening sensitive file permissions" chmod 640 /etc/shadow 2>/dev/null chmod 644 /etc/passwd chmod 640 /etc/gshadow 2>/dev/null chmod 600 /etc/crontab chmod 700 /root echo "" echo "[✓] Hardening complete. Review /etc/ssh/sshd_config and ufw status."

AI Integration in Bash & Security Automation

Artificial Intelligence is transforming Bash-based security automation — from AI-assisted script generation and debugging to intelligent log analysis that goes far beyond grep patterns. For security practitioners, the most immediate impact is in script development velocity: LLMs can generate complete, production-quality Bash scripts from natural-language descriptions in seconds, debug complex sed/awk pipelines that would take an engineer an hour to troubleshoot, and explain legacy scripts that lack documentation. On the log analysis side, AI converts raw shell output into structured intelligence with context that pattern-matching alone can never provide.

[ AI ]

A security engineer using AI-assisted Bash script generation can produce a complete, functional, error-handled log analysis script in under 3 minutes by describing the requirement in plain English. The same script written from scratch would take 20–40 minutes and contain more bugs. AI does not replace Bash knowledge — it multiplies the productivity of someone who already understands the shell environment.

✍️
AI Script Generation
LLMs generate complete, error-handled Bash scripts from plain-English descriptions — "write a script that monitors /var/log/auth.log, counts failed SSH logins per IP, and sends a Slack alert when any IP exceeds 5 failures in 60 seconds." Produces working scripts including edge case handling, input validation, and logging that manual scripting frequently omits.
🐛
AI Script Debugging
LLMs explain cryptic Bash errors, debug complex sed/awk/grep pipelines, and identify logical errors in control flow — the class of bugs that are hardest to catch because the script runs without error but produces wrong output. Paste the script and the actual vs expected output; AI identifies the issue and explains why it occurs and how to fix it.
📊
Intelligent Log Analysis
Feed raw log snippets to LLMs and receive structured intelligence: "explain what this auth.log sequence shows about the attacker's methodology," "identify which of these processes are suspicious and why," or "correlate these three log events into an attack timeline." Converts raw shell output into narrative threat intelligence that grep alone cannot produce.
🔍
Script Security Review
LLMs review Bash scripts for security vulnerabilities — command injection risks from unquoted variables, insecure temporary file creation, race conditions in file permission checks, PATH hijacking exposure, and hardcoded credentials. Acts as an automated code review that catches the security anti-patterns that even experienced scripters miss under time pressure.
📖
Legacy Script Explanation
LLMs parse and explain undocumented legacy Bash scripts — the 200-line shell script that runs as root via cron and nobody remembers who wrote it. Provides line-by-line explanations, identifies what the script does, flags potential issues, and documents it in a format that can be safely inherited by the next engineer.
🤖

Secure Bash Scripting Best Practices

Security scripts that are themselves insecure are worse than no script at all — they run with elevated privileges, handle sensitive data, and operate on critical systems. Bash scripts in security contexts are frequent targets for privilege escalation: a world-writable cron script running as root, an insecurely created temp file, or an unquoted variable used in a command — each is a potential root escalation for any local attacker. Every security-relevant Bash script must be written with the same care as the systems it protects.

Script Safety Headers
  • Always start with set -euo pipefail — exit on errors, undefined variables, and pipe failures
  • Use trap cleanup EXIT to ensure temporary files and resources are always cleaned up
  • Declare readonly for constants — prevents accidental modification of critical values
  • Use local for all function variables — prevents unintended global variable pollution
Input & Variable Safety
  • Always double-quote variables: use "$var" not $var — prevents word splitting and globbing
  • Validate all user-supplied input — reject input containing shell metacharacters when not expected
  • Use absolute paths for all commands in security scripts — prevents PATH hijacking
  • Never eval user input — command injection via eval is one of the most dangerous Bash mistakes
File & Permission Security
  • Create temp files with mktemp — never use predictable names like /tmp/script_$$
  • Set chmod 700 on security scripts — only the owner should be able to read or execute them
  • Store scripts in root-owned, non-world-writable directories — especially if run via cron as root
  • Never store credentials in scripts — use environment variables or encrypted vaults (pass, gpg, HashiCorp Vault)
#!/usr/bin/env bash # ───────────────────────────────────────────────────────────────────── # SCRIPT: security_task.sh # PURPOSE: [description] # USAGE: ./security_task.sh [options] # AUTHOR: Nisarg Chasmawala (Shroff) // HEAVEN # ───────────────────────────────────────────────────────────────────── # SAFETY HEADERS — always include these set -euo pipefail # Exit on error, undefined vars, pipe failures IFS=$'\n\t' # Safer IFS — no accidental word splitting on spaces # CONSTANTS — readonly prevents accidental overwrite readonly SCRIPT_NAME=$(basename "$0") readonly LOG_DIR="/var/log/security" readonly LOG_FILE="${LOG_DIR}/${SCRIPT_NAME%.sh}.log" readonly LOCK_FILE="/var/run/${SCRIPT_NAME%.sh}.lock" # CLEANUP — always runs on exit (normal or error) cleanup() { rm -f "$LOCK_FILE" # Add: remove temp files, close file descriptors, etc. } trap cleanup EXIT # PREVENT CONCURRENT EXECUTION if [ -e "$LOCK_FILE" ]; then echo "[!] Another instance running (lock: $LOCK_FILE)"; exit 1 fi touch "$LOCK_FILE" # LOGGING FUNCTION log() { local level="${1^^}"; shift printf '[%s] [%s] %s\n' "$(date -Iseconds)" "$level" "$*" | \ tee -a "$LOG_FILE" } # ROOT CHECK if [ $EUID -ne 0 ]; then log error "Root required. Run: sudo $0 $*"; exit 1 fi # ─── YOUR SCRIPT LOGIC BELOW ───────────────────────────────────────── log info "Script started" # Example: safe temp file creation TMPFILE=$(mktemp /tmp/$SCRIPT_NAME.XXXXXX) trap 'rm -f "$TMPFILE"' EXIT # Ensure temp file cleaned up log info "Script completed successfully"
